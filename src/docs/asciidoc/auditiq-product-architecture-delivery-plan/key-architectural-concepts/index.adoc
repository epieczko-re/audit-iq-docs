== Key Architectural Concepts

=== Audit Packs: The “Brain” Plugin Model

An audit pack encapsulates everything needed to audit a specific regulatory domain. It is a versioned, declarative configuration artifact — not executable code. The platform interprets the pack to drive ingestion validation, schema mapping, comparison rules, scoring, and report generation. This makes AuditIQ extensible: adding a new regulation means creating a new pack, not modifying platform code.

*Phase 1:* One pack — HMDA Origination QC. Pre-built by RiskExec. *Phase 2:* CRA, SBL (Section 1071). Pre-built by RiskExec. *Future:* Client-customizable packs, potentially a third-party pack marketplace.

=== Three-Layer Confidence Scoring

AuditIQ applies confidence scoring at three independent layers:

[arabic]
. *Mapping confidence:* How confident is the system that target field X corresponds to source field Y? (Based on field name similarity, data type compatibility, value distribution, historical acceptance.)
. *Extraction confidence:* How confident is the OCR/extraction pipeline that the extracted value from a source document is correct? (Based on OCR quality, layout complexity, document tier.)
. *Rule application confidence:* How confident is the system that a validation rule correctly applies to this data context? (Based on precondition satisfaction, data quality, rule specificity.)

These three scores are orthogonal. A finding can have high mapping confidence but low extraction confidence (the field match is clear, but the OCR struggled with the document). The combined confidence score — computed as the minimum of the three applicable scores — determines the human review requirement.

*Combined confidence formula:*

....
combined_confidence = min(mapping_confidence, extraction_confidence, rule_confidence)
....

When a layer is not applicable (e.g., no OCR involved in a record-to-record audit), that layer is set to 1.0 (full confidence) and does not reduce the combined score.

=== Findings Severity vs. Confidence (Orthogonal Dimensions)

Severity measures the regulatory importance of a finding (how bad is it?). Confidence measures how sure the system is that the finding is real (how certain is it?). These are independent:

[cols="25%,25%,30%,20%",options="header"]
|===
|Scenario |Severity |Confidence |Action
|Action Taken code clearly wrong |Critical |High |Immediate remediation; no additional verification needed
|Loan Amount might not match (OCR uncertain) |High |Low |Route to manual review; do not auto-generate finding
|Whitespace formatting difference |Low |High |Log as informational; no action required
|Property address potentially wrong but document blurry |Medium |Low |Flag for manual verification with original document
|===

=== Immutable Audit Trail with Regulator-Safe Semantics

Every state change in the system is recorded as an immutable event. Events are append-only and hash-chained for tamper evidence. The system differentiates between truly immutable artifacts (evidence, events) and correctible artifacts (findings, mappings) where corrections are modeled as new events that supersede prior state. See <<sect-immutability>> for the full immutability/correctability specification.

=== Architecture Diagrams

==== Diagram 1: Audit Run State Machine

[mermaid]
----
---
title: Audit Run State Machine
---
stateDiagram-v2
    [*] --> Created : Admin creates audit run
    Created --> Ingesting : Upload target dataset
    Ingesting --> Validating : Dataset received
    Validating --> Ready : Validation passed
    Validating --> ValidationFailed : Validation errors
    ValidationFailed --> Ingesting : Re-upload corrected data
    Ready --> InProgress : Start audit
    InProgress --> PreScanRunning : Launch automated pre-scan
    PreScanRunning --> ReviewReady : Pre-scan complete
    InProgress --> ReviewReady : Skip pre-scan (manual mode)
    ReviewReady --> UnderReview : Reviewers assigned
    UnderReview --> SecondReview : Primary review complete + sampling
    UnderReview --> ReportGeneration : Primary review complete (no second review)
    SecondReview --> ReportGeneration : Second review complete
    ReportGeneration --> PendingApproval : Reports generated
    PendingApproval --> Finalized : Manager approves
    PendingApproval --> UnderReview : Manager returns for rework
    Finalized --> [*]
----
