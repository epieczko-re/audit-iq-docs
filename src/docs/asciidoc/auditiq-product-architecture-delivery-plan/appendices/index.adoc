=== Appendix A: Glossary (Expanded from v0.1)

*Audit (noun):* A review of lending compliance for one or more loan records against a pack of rules and policies.

*Audit Event:* Immutable timestamped record of a change to audit state (e.g., “finding created”, “manager approval”, “report finalized”). See Appendix C.

*Audit Pack (or Pack):* A declarative JSON specification defining rules, scoring weights, evidence requirements, and narratives for a compliance audit category (e.g., HMDA, CRA, ECOA). See <<sect-audit-pack-spec>>.

*Calibration:* Process of tuning confidence score thresholds to match historical reviewer decisions. See <<sect-confidence-calibration>>.

*Confidence Score:* Numeric value (0.0–1.0) representing the AI system’s certainty in a data extraction, schema mapping, or finding. Composed of mapping confidence + extraction confidence + rule confidence.

*Docling:* Open-source document parsing library that extracts text, tables, images from PDFs. Used for HMDA document extraction in Phase 1a.

*Evidence:* A document, table cell, or data field that is linked to a finding as proof of compliance or violation.

*Finding:* A structured record of a single compliance issue (or absence thereof) identified during an audit. Contains finding_id, target_record_id, rule_id, severity, confidence, narrative, approver.

*Human-in-Loop:* Requirement for human (reviewer or manager) to explicitly approve or correct an AI-generated result before finalization.

*Immutability:* Property that audit events, evidence files, and finalized findings cannot be deleted or retroactively modified. Corrections are recorded as new compensating events (<<sect-immutability>>).

*LogMap:* Semantic web ontology matching library; AuditIQ evaluates it as a potential schema matcher replacement in Phase 2.

*Manager:* Reviewer role with authority to escalate findings, adjust severity, and approve audit finalization.

*Pack Manifest:* JSON file defining pack metadata, rule list, evidence requirements, and versioning. See Appendix B.

*Reviewer:* Primary auditor role; responsible for extracting evidence, creating findings, and requesting overrides.

*RLS (Row-Level Security):* PostgreSQL feature that restricts data access based on user tenant. AuditIQ relies on RLS for tenant isolation.

*Sampling:* Random or stratified selection of records from a population to audit. Pack can define sampling methodology.

*Schema Matcher:* Component that maps extracted HMDA field values to audit pack schema fields with confidence scores. See <<sect-schema-mapping-service>>.

*Snapshot:* Immutable record of audit state at a point in time (used for reproducibility & rollback to prior version without re-running extraction). See <<sect-immutability>>.

*SOC 2:* Service Organization Control audit; certifies security, availability, and confidentiality controls. AuditIQ target compliance by Jun 30.

*Tenant:* A customer organization in the multi-tenant AuditIQ system; data isolated by row-level security + encryption.

*Audit Trail / Audit Log:* Complete record of all events (extractions, approvals, corrections) for an audit. Append-only; forms basis of regulatory compliance proof.

=== Appendix D: Report Template Outlines

*Audit Summary Report (PDF template, Phase 1a MVP minimum):*

....
Cover Page:
  - Organization name, audit period, auditor name, date

Executive Summary:
  - Audit scope (pack, record count, sampling methodology)
  - Key findings count by severity (critical, major, minor)
  - Compliance assessment (Pass / Fail / Pass w/ Observations)
  - Recommended corrective actions (summary)

Detailed Findings:
  - Table: Finding ID | Rule | Severity | Confidence | Evidence | Narrative | Approver

Audit Trail Summary:
  - Table: Event ID | Event Type | Timestamp | Actor | Details

Attachments:
  - Evidence index (source documents, extraction confidence)
  - Pack version & configuration
  - Sampling methodology documentation
....

*Extended Reports (Phase 2+):*

* Trend analysis (findings over time by category)
* Drill-down reports (filter by severity, rule, business line)
* Comparative audits (current vs. prior audit, same pack version)
* Regulatory submission format (FDIC, OCC specific)
