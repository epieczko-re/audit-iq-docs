=== Appendix A: Canonical Glossary

____
*Purpose.* This glossary is the *single source of truth* for every term, confidence metric, routing threshold, and severity level used in AuditIQ.
All other sections reference these definitions; if a value in another section conflicts with this glossary, this glossary governs.
When thresholds are tuned, update *this file first*, then propagate.
____

==== General Terms

*Audit (noun):* A review of lending compliance for one or more loan records against a pack of rules and policies.

*Audit Event:* Immutable timestamped record of a change to audit state (e.g., "finding created", "manager approval", "report finalized").
See Appendix C.

*Audit Pack (or Pack):* A declarative JSON specification defining rules, scoring weights, evidence requirements, and narratives for a compliance audit category (e.g., HMDA, CRA, ECOA).
See <<sect-audit-pack-spec>>.

*Audit Trail / Audit Log:* Complete record of all events (extractions, approvals, corrections) for an audit.
Append-only; forms basis of regulatory compliance proof.

*Calibration:* Process of tuning confidence score thresholds to match historical reviewer decisions.
See <<sect-confidence-calibration>>.

*Confidence Score:* Numeric value (0.0–1.0) representing the AI system's certainty in a data extraction, schema mapping, or finding.
Composed of mapping confidence + extraction confidence + rule confidence.

*Docling:* Open-source document parsing library that extracts text, tables, images from PDFs.
Used for HMDA document extraction in Phase 1a.

*Evidence:* A document, table cell, or data field that is linked to a finding as proof of compliance or violation.

*Finding:* A structured record of a single compliance issue (or absence thereof) identified during an audit.
Contains finding_id, target_record_id, rule_id, severity, confidence, narrative, approver.

*Human-in-Loop:* Requirement for human (reviewer or manager) to explicitly approve or correct an AI-generated result before finalization.

*Immutability:* Property that audit events, evidence files, and finalized findings cannot be deleted or retroactively modified.
Corrections are recorded as new compensating events (<<sect-immutability>>).

*LogMap:* Semantic web ontology matching library; AuditIQ evaluates it as a potential schema matcher replacement in Phase 2.

*Manager:* Reviewer role with authority to escalate findings, adjust severity, and approve audit finalization.

*Pack Manifest:* JSON file defining pack metadata, rule list, evidence requirements, and versioning.
See Appendix B.

*Reviewer:* Primary auditor role; responsible for extracting evidence, creating findings, and requesting overrides.

*RLS (Row-Level Security):* PostgreSQL feature that restricts data access based on user tenant.
AuditIQ relies on RLS for tenant isolation.

*Sampling:* Random or stratified selection of records from a population to audit.
Pack can define sampling methodology.

*Schema Matcher:* Component that maps extracted HMDA field values to audit pack schema fields with confidence scores.
See <<sect-schema-mapping-service>>.

*Snapshot:* Immutable record of audit state at a point in time (used for reproducibility & rollback to prior version without re-running extraction).
See <<sect-immutability>>.

*Source Documents (Source Data):* The evidence documents against which target records are compared — original loan file artifacts (PDFs, images, or a second structured dataset in record-to-record audits).
Extraction from source documents produces confidence scores (AFC) because OCR and layout parsing introduce uncertainty.
See <<sect-target-source-schemas>>.

*SOC 2:* Service Organization Control audit; certifies security, availability, and confidentiality controls.
AuditIQ target compliance by Jun 30.

*Target Dataset (Target Data):* The structured loan records under audit — the institution's own data as filed, originated, or reported.
Validated deterministically during ingestion against the audit pack schema.
See <<sect-target-source-schemas>>.

*Tenant:* A customer organization in the multi-tenant AuditIQ system; data isolated by row-level security + encryption.

==== Confidence Metrics

[cols="18%,12%,40%,30%",options="header"]
|===
|Metric |Range |Definition |Canonical Source

|*AFC* (AuditIQ Field Confidence)
|0.0–1.0
|Per-field composite score measuring extraction reliability. *AFC v2* (with Docling-Graph): computed from 6 weighted signals: `docling_page_grade` (0.12), `ocr_segment_confidence` (0.23), `pattern_validity` (0.20), `layout_anchor_consistency` (0.18), `cross_doc_redundancy` (0.12), `graph_corrob_score` (0.15). AFC v1 (5-signal, without `graph_corrob_score`) remains valid during Phase 0 POC.
|<<sect-doc-processing>>

|*extraction_confidence*
|0.0–1.0
|Alias for the AFC score when used as input to combined confidence. Represents how confident the system is in the extracted value from a source document.
|<<sect-scoring-framework>> (Layer 2)

|*mapping_confidence* (`conf_mapping`)
|0.0–1.0
|Measures how confident the system is that target field X corresponds to source field Y. Layer 1 raw score from the hybrid heuristic matcher: `name_similarity` (0.30), `synonym_match` (0.25), `type_compatibility` (0.20), `document_section` (0.15), `historical_acceptance` (0.10).
|<<sect-schema-mapping-service>>, <<sect-scoring-framework>> (Layer 1)

|*rule_confidence* (`conf_rule`)
|0.0–1.0
|Measures how confidently a validation rule applies to a specific data context. Base confidence from pack rule definition, reduced by 0.7× if precondition not satisfied, 0.5× if normalization failed.
|<<sect-scoring-framework>> (Layer 3)

|*combined_confidence*
|0.0–1.0
|`min(mapping_confidence, extraction_confidence, rule_confidence)`. The floor-based composite that drives finding-level routing. If extraction is not applicable, `extraction_confidence` defaults to 1.0.
|<<sect-scoring-framework>>

|*graph_corrob_score*
|0.0–1.0
|Cross-document graph corroboration signal introduced in AFC v2. Computed from the number and strength of corroborating edges in the Docling-Graph knowledge graph (e.g., loan_amount matching across Closing Disclosure and Loan Estimate nodes). Weighted at 0.15 in the AFC v2 formula. Absent (defaults to 0.0) when Docling-Graph is not enabled (AFC v1 mode).
|<<sect-doc-processing>>, <<sect-docling-graph>>

|*calibrated_score* (Layer 2 mapping)
|0.0–1.0
|Calibrated decision score combining `conf_mapping`, `conf_extraction`, and `conf_rule` via temperature scaling (Phase 1) or isotonic regression (Phase 2). Used for schema mapping routing only. ECE must be ≤ 0.05.
|<<sect-schema-mapping-service>>

|*alignment_confidence*
|0.0–1.0
|Composite score measuring how confidently a policy sentence maps to a regulatory concept during policy-to-regulation alignment. Computed from four weighted signals: `graph_anchor_strength` (0.35), `llm_reranker_confirmation` (0.25), `structural_distance` (0.20), `semantic_similarity` (0.20). Alignments scoring < 0.70 are flagged for mandatory human review.
|<<sect-alignment-confidence>>

|*graph_anchor_strength*
|0.0–1.0
|Cosine similarity between a policy sentence embedding (Layer 2) and an ontology node embedding (Layer 1) at anchoring time. Higher values indicate the sentence is semantically close to the formal compliance concept. Weighted at 0.35 in the alignment confidence formula.
|<<sect-alignment-confidence>>

==== Routing Thresholds

NOTE: AuditIQ uses *three distinct routing contexts* with different threshold bands, each calibrated for its specific decision context.
AFC routing (extraction) uses lower thresholds because extraction errors are caught downstream.
Schema mapping routing uses higher thresholds because incorrect mappings propagate through the entire audit.
Combined confidence routing reflects the floor of all three layers.

===== AFC Routing (Document Extraction — <<sect-doc-processing>>)

[cols="25%,25%,50%",options="header"]
|===
|Band |Threshold |Action

|*High (Auto-confirm)*
|AFC ≥ 0.85
|No human review required. Field value accepted automatically.

|*Medium (Human review)*
|0.50 ≤ AFC < 0.85
|Field highlighted in UI. Reviewer validates or overrides extracted value.

|*Low (Manual entry)*
|AFC < 0.50
|Extraction failure. Routed to manual entry queue; reviewer must type value.
|===

===== Schema Mapping Routing (Calibrated Score — <<sect-schema-mapping-service>>)

[cols="25%,25%,50%",options="header"]
|===
|Band |Threshold |Action

|*Auto-accept*
|calibrated ≥ 0.95
|Mapping locked without review (~10s visual confirmation). NOTE: Conservative initial threshold per <<sect-decision-d2, Decision D2>>; may be lowered to ≥ 0.90 after calibration on ≥ 500 mappings.

|*Human review*
|0.70 ≤ calibrated < 0.95
|Reviewer evaluates suggestion + alternatives (~45s per field).

|*Manual mapping*
|calibrated < 0.70
|No suggestion shown. Reviewer selects from field list.
|===

===== Combined Confidence Routing (Findings — <<sect-scoring-framework>>)

[cols="25%,25%,50%",options="header"]
|===
|Band |Threshold |Action

|*High (Auto-confirm)*
|combined_confidence ≥ 0.85
|Does not require human review. Auto-confirm if desired.

|*Medium (Human review)*
|0.50 ≤ combined_confidence < 0.85
|Recommend human verification; display in UI for secondary review.

|*Low (Reject)*
|combined_confidence < 0.50
|Reject finding; flag for manual re-review if critical issue.
|===

==== Severity Definitions

*Severity measures regulatory importance.
It is orthogonal to confidence.*

[cols="13%,17%,16%,27%,27%",options="header"]
|===
|Level |Definition |Examples |Action Required |Review Timeline

|*Critical*
|Regulatory violation with potential enforcement action; data misrepresentation; prohibited conduct
|Action Taken code wrong, pricing discrimination pattern, missing disclosure
|Immediate remediation, escalation to Compliance Officer
|Within 1 day

|*High*
|Potentially reportable discrepancy; regulatory attention likely if unresolved
|Loan Amount mismatch > 2%, co-applicant name inconsistent
|Prompt investigation; remediation within 3 days
|Within 3 days

|*Medium*
|Compliance or data quality issue; non-trivial to fix; not immediately reportable
|Property address formatting inconsistent, fee description unclear
|Standard remediation workflow
|Within 7 days

|*Low*
|Minor data quality issue; cosmetic; no regulatory impact
|Whitespace extra, trailing zeros
|Log and monitor; no urgent action required
|Within 14 days

|*Informational*
|Data point of interest; no action required
|Historical note, data type conversion artifact
|Log only; no action
|Informational only
|===

Severity is assigned by: (1) pack rule definition (default), (2) optionally overridden by Reviewer with justification, (3) may be escalated by Manager if pattern detected.
Severity is immutable once finding is triaged unless explicitly re-scored by Manager with audit trail event.

==== Calibration & Go/Kill Criteria

[cols="30%,35%,35%",options="header"]
|===
|Metric |Threshold |Context

|*Calibration error alert*
|> 5% of high-confidence findings where human disagreed
|Triggers threshold adjustment (<<sect-confidence-calibration>>)

|*Expected Calibration Error (ECE)*
|≤ 0.05
|Required for Phase 2 deployment (<<sect-schema-mapping-service>>)

|*Phase 0 AFC distribution target*
|≥ 60% of fields above 0.85 on 50-document sample
|POC acceptance gate (<<sect-doc-processing>>)

|*Phase 1a Go/Kill (AFC)*
|Go: ≥ 85% precision on key HMDA fields at AFC ≥ 0.85. Kill: < 70% precision or fallback rate > 40%.
|<<sect-doc-processing>>

|*Phase 1b Go/Kill (Mapping)*
|Go: ≥ 95% precision at conf_mapping ≥ 0.90 on CD-supported fields; override rate ≤ 25%. Kill: > 40% override rate or scores uncalibratable.

==== Regulatory Terminology

[cols="25%,75%",options="header"]
|===
|Term |Definition

|*HMDA (Home Mortgage Disclosure Act)*
|Federal law requiring financial institutions to collect and report data about mortgage lending activity.
Regulation C implements HMDA and defines ~110 reportable data fields per loan application.

|*Regulation C*
|The implementing regulation for HMDA, issued by the CFPB.
Specifies data collection, reporting, and disclosure requirements for mortgage lenders.

|*CRA (Community Reinvestment Act)*
|Federal law requiring banks to meet the credit needs of the communities they serve, including low- and moderate-income neighborhoods.

|*ECOA (Equal Credit Opportunity Act)*
|Federal law prohibiting credit discrimination on the basis of race, color, religion, national origin, sex, marital status, age, or receipt of public assistance.

|*TRID (TILA-RESPA Integrated Disclosure)*
|Combined disclosure rules under the Truth in Lending Act (TILA) and Real Estate Settlement Procedures Act (RESPA).
Requires Loan Estimate and Closing Disclosure forms for most residential mortgage loans.

|*FCRA (Fair Credit Reporting Act)*
|Federal law governing the collection, dissemination, and use of consumer credit information.

|*MISMO (Mortgage Industry Standards Maintenance Organization)*
|Industry body that maintains the standard data model for mortgage transactions.
AuditIQ uses a flattened subset of the MISMO Reference Model v3.x as its canonical schema (<<sect-decision-d6>>).

|*SR 11-7 / OCC 2011-12*
|Interagency supervisory guidance on Model Risk Management.
Requires model inventory, validation, effective challenge, governance, and documentation for any model used in a decision process.

|*NIST AI RMF (AI Risk Management Framework)*
|Voluntary framework from the National Institute of Standards and Technology for managing AI risk.
Organized around four functions: Govern, Map, Measure, Manage.

|*CFPB (Consumer Financial Protection Bureau)*
|Federal agency responsible for consumer protection in the financial sector.
Issues regulations (Regulation C, Regulation B, Regulation Z) and enforcement actions relevant to AuditIQ's audit domains.

|*FFIEC (Federal Financial Institutions Examination Council)*
|Interagency body that prescribes uniform principles, standards, and report forms for examining financial institutions.
FFIEC IT Examination Handbook governs audit trail and information security controls.

|*GLBA (Gramm-Leach-Bliley Act)*
|Federal law requiring financial institutions to explain their information-sharing practices and safeguard sensitive data.
Regulation P implements GLBA privacy requirements.

|*SOC 2 (Service Organization Control 2)*
|AICPA audit framework that certifies a service organization's controls for security, availability, processing integrity, confidentiality, and privacy.
AuditIQ targets SOC 2 Type II readiness by June 30, 2026.

==== AI & Embedding Terminology

[cols="25%,75%",options="header"]
|===
|Term |Definition

|*Sentence Transformer (SBERT)*
|A variant of the BERT architecture optimized for producing semantically meaningful sentence-level embeddings.
AuditIQ uses a single domain-fine-tuned SBERT model shared across the Regulatory Citation Engine and the Policy Alignment Pipeline, fine-tuned on financial regulatory language for negation sensitivity, numeric threshold context, and deontic language precision.
See <<sect-embedding-domain-finetuning>>.

|*OWL2Vec**
|An ontology embedding method that generates vector representations of OWL ontology nodes by combining random walks over the graph structure with textual features from labels and descriptions.
Used to produce Layer 1 (structural) embeddings for the unified compliance knowledge graph.
See <<sect-two-layer-embeddings>>.

|*on2vec*
|An extension of ontology embedding that integrates graph neural network (GNN) structural features with sentence transformer text features into a single embedding.
An alternative or complement to OWL2Vec* for generating ontology node embeddings.
See <<sect-two-layer-embeddings>>.

|*Graph Anchoring*
|The process of linking an unstructured policy or regulation sentence to a precise concept node in the compliance knowledge graph via cross-space cosine similarity between sentence embeddings (Layer 2) and ontology node embeddings (Layer 1), confirmed by an LLM reranker.
Produces `implements` edges in the graph.
See <<sect-embedding-architecture>>.

|*Cross-Space Cosine Similarity*
|A similarity metric computed between vectors from two different embedding spaces — specifically, between a document sentence embedding (produced by the sentence transformer) and an ontology node embedding (produced by OWL2Vec* or on2vec).
Used during graph anchoring to identify candidate concept nodes for a given policy sentence.

|*`implements` Edge*
|A graph edge of type `PolicySentence ──implements──→ OntologyNode` created during graph anchoring.
Links an institution's policy or procedure sentence to the regulatory concept it addresses.
Distinct from `corroborates` edges, which link extracted field values across loan documents.
See <<sect-graph-edge-taxonomy>>.

|*`corroborates` Edge*
|A graph edge linking extracted field values across documents for the same loan (e.g., loan_amount matching across Closing Disclosure and Loan Estimate).
Created by the Docling-Graph extraction pipeline.
Drives `graph_corrob_score` in AFC v2.
See <<sect-graph-edge-taxonomy>>.

==== RFC 8785 Canonical JSON & Hash Chain Walkthrough

AuditIQ uses RFC 8785 (JSON Canonicalization Scheme) to ensure that logically identical JSON always produces the same SHA-256 hash.
This is critical because non-canonical JSON (e.g., different key ordering, whitespace, or number formatting) produces different hashes for the same data, breaking chain verification.

*RFC 8785 Canonicalization Rules (summary):*

[arabic]
. *Key ordering:* Object keys sorted lexicographically by Unicode code point
. *No whitespace:* No spaces, tabs, or newlines between tokens
. *Number formatting:* No leading zeros; no trailing zeros after decimal; no positive sign; integers have no decimal point
. *String escaping:* Minimal escaping; only required characters escaped
. *No duplicate keys:* Each key appears once per object

*Worked Example — Event `evt_20260215_002`:*

*Step 1: Raw event (as constructed by the application):*

[source,json]
----
{
  "event_type": "finding_created",
  "event_id": "evt_20260215_002",
  "timestamp": "2026-02-15T10:45:00Z",
  "actor_id": "system_scoring_engine",
  "audit_id": "audit_2026_02_15_loanXYZ",
  "sequence_id": 1002,
  "previous_hash": "a1b2c3d4e5f6...sha256_of_evt_001",
  "event_details": {
    "severity": "critical",
    "finding_id": "finding_hmda_001",
    "rule_id": "hmda_field_completeness",
    "combined_confidence": 0.87,
    "narrative": "Loan amount of $250,000 verified in loan estimate."
  }
}
----

*Step 2: RFC 8785 canonicalized form (keys sorted, no whitespace):*

[source,json]
----
{"actor_id":"system_scoring_engine","audit_id":"audit_2026_02_15_loanXYZ","event_details":{"combined_confidence":0.87,"finding_id":"finding_hmda_001","narrative":"Loan amount of $250,000 verified in loan estimate.","rule_id":"hmda_field_completeness","severity":"critical"},"event_id":"evt_20260215_002","event_type":"finding_created","previous_hash":"a1b2c3d4e5f6...sha256_of_evt_001","sequence_id":1002,"timestamp":"2026-02-15T10:45:00Z"}
----

Note: `event_details` keys are also sorted (`combined_confidence` before `finding_id` before `narrative` before `rule_id` before `severity`).

*Step 3: Compute SHA-256 hash:*

....
event_hash = SHA-256(canonicalized_json_bytes)
           = "b7f3a9e1c2d4..."  (64 hex characters)
....

*Step 4: Chain linkage:*

The next event (`evt_20260215_003`) includes `"previous_hash": "b7f3a9e1c2d4..."` — creating an append-only hash chain.
Any modification to event 002 changes its hash, which breaks the chain at event 003.

*Verification algorithm (round-trip test):*

[arabic]
. Read event from database
. Strip `event_hash` field from the record
. Canonicalize remaining fields per RFC 8785
. Compute SHA-256 of canonical bytes
. Compare to stored `event_hash` — must match exactly
. Verify `previous_hash` equals the `event_hash` of the preceding event (by `sequence_id`)
. If either check fails, emit `audit_trail_integrity_violation` alert

*WORM Anchor Points:*

Every 1,000 events (configurable), the system writes a cryptographic anchor to WORM (Write Once Read Many) storage (Azure Immutable Blob).
The anchor contains: `tenant_id`, `anchor_sequence_range` (e.g., 1001–2000), `final_event_hash`, `timestamp`, and `anchor_hash` (SHA-256 of the anchor record itself).
This prevents an attacker with database write access from silently rewriting the entire chain — the WORM-stored anchor hashes cannot be modified.
