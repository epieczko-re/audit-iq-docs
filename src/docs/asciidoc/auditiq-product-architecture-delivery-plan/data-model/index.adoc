[[sect-mismo-alignment]]
=== MISMO Alignment Strategy

*Design Principle:* AuditIQ’s canonical data model adopts MISMO (Mortgage Industry Standards Maintenance Organization) Reference Model naming conventions and field definitions for all loan-level attributes.
This alignment reduces translation friction when ingesting data from LOS, servicing, and origination systems — most of which already export MISMO-formatted data — and increases credibility with examiners who recognize MISMO field names.

*Scope of MISMO Adoption:*

* *Canonical loan data fields:* All loan-level attributes stored in `staging_datasets` and referenced in `findings` use MISMO Reference Model v3.x naming where a corresponding MISMO data point exists.
Examples: `LoanIdentifier`, `BorrowerMailingAddress`, `PropertyValuationAmount`, `ActionTaken` (HMDA), `RateSpreadPercent`.
* *Document classification:* Evidence documents are classified using the MISMO Document Custodian taxonomy (e.g., `BorrowerAuthorization`, `CreditReport`, `AppraisalReport`) for consistent naming across packs and evidence uploads.
* *HMDA Reporting Dataset:* Phase 1 HMDA packs map directly to the MISMO HMDA Reporting Dataset, which provides a well-defined field set for LAR data.
This covers all fields required for HMDA compliance auditing.
* *Audit Pack field references:* Pack `field_mappings` definitions reference MISMO canonical names as the target schema.
Source fields (from customer uploads) are mapped _to_ MISMO names by the schema mapping engine.

*What is NOT MISMO-aligned:*

* *Audit operational tables:* `audit_runs`, `findings`, `field_mappings`, `evidence`, and `audit_trail.events` use AuditIQ-native schemas.
These are operational/workflow constructs with no MISMO equivalent.
* *Pack configuration schema:* Audit Packs remain proprietary JSON (<<sect-audit-pack-spec>>).
Packs _reference_ MISMO field names in their rules, but the pack format itself is not a MISMO construct.
* *AI/ML metadata:* Confidence scores, provenance metadata, and model evaluation data are AuditIQ-specific.

*Implementation Approach — Flattened Subset:*

MISMO’s full XML schema is deeply nested and verbose (thousands of data points).
AuditIQ adopts a *flattened subset* relevant to each audit domain:

* *Phase 1 (HMDA):* ~80 MISMO data points from the HMDA Reporting Dataset, flattened into a single-level JSON structure stored in `staging_datasets.payload`.
Field names match MISMO exactly (e.g., `ActionTakenType`, `LienStatusType`, `LoanPurposeType`).
* *Phase 2+ (additional regulations):* Each new Audit Pack defines its MISMO subset in the pack manifest under `canonical_schema.mismo_fields[]`.
The schema mapping engine validates that all referenced MISMO fields exist in the MISMO Reference Model dictionary.
* *Unmapped fields:* Fields that have no MISMO equivalent (custom institution fields, derived metrics) use a `custom:` namespace prefix (e.g., `custom:InternalRiskScore`) to clearly distinguish them from MISMO-standard fields.

*MISMO Reference Model Dictionary:*

AuditIQ maintains a lookup table of adopted MISMO fields:

[source,sql]
----
CREATE TABLE reference.mismo_dictionary (
  mismo_field_id VARCHAR(255) PRIMARY KEY, -- e.g., 'ActionTakenType'
  mismo_xpath VARCHAR(1024), -- Full MISMO XML path for traceability
  data_type VARCHAR(50), -- 'string', 'decimal', 'date', 'integer', 'enum'
  enum_values TEXT[], -- For enumerated types, valid values
  regulation_domain VARCHAR(100)[], -- e.g., '{hmda}', '{trid, hmda}'
  description TEXT,
  mismo_version VARCHAR(20) DEFAULT '3.6', -- MISMO Reference Model version
  added_in_auditiq_version VARCHAR(20),
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_mismo_domain ON reference.mismo_dictionary USING GIN (regulation_domain);
----

This dictionary is populated during pack deployment and used by the schema mapping engine to validate field references and generate mapping suggestions.

=== Core Tables

==== audit_runs

[source,sql]
----
CREATE TABLE audits.audit_runs (
  audit_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL, -- Multi-tenancy
  pack_id VARCHAR(255) NOT NULL, -- e.g., 'hmda-origination'
  pack_version VARCHAR(20) NOT NULL, -- e.g., '1.0.0'
  audit_name VARCHAR(512),
  audit_scope VARCHAR(100), -- 'origination', 'servicing', etc.
  created_by UUID NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  status VARCHAR(50) DEFAULT 'created', -- Enum: created, validating, ready, in_progress, review_complete, reporting, finalized

  -- Input data
  target_dataset_id UUID NOT NULL, -- Foreign key to staging_datasets
  target_record_count INT,

  -- Execution tracking
  started_at TIMESTAMP,
  completed_at TIMESTAMP,
  execution_mode VARCHAR(50), -- 'manual', 'ocr_assisted', 'automated'

  -- Configuration
  second_review_percentage DECIMAL(5,2) DEFAULT 0.0,
  confidence_threshold DECIMAL(3,2) DEFAULT 0.5,

  -- Audit trail link
  audit_context_id VARCHAR(255), -- Foreign key to audit_trail.contexts

  FOREIGN KEY (tenant_id) REFERENCES tenants(tenant_id),
  FOREIGN KEY (created_by) REFERENCES users(user_id),
  FOREIGN KEY (target_dataset_id) REFERENCES audits.staging_datasets(dataset_id)
);
CREATE INDEX idx_audit_tenant ON audits.audit_runs(tenant_id);
CREATE INDEX idx_audit_pack ON audits.audit_runs(pack_id, pack_version);
----

==== findings

[source,sql]
----
CREATE TABLE audits.findings (
  finding_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  audit_id UUID NOT NULL,
  record_id VARCHAR(255) NOT NULL, -- Target record ID from uploaded dataset
  field_id VARCHAR(255) NOT NULL, -- Field in target schema

  -- Comparison details
  rule_id VARCHAR(255), -- Pack rule that triggered this finding
  target_value TEXT,
  source_value TEXT,
  match_result VARCHAR(50), -- 'match', 'mismatch', 'unable_to_determine'

  -- Confidence & severity
  mapping_confidence DECIMAL(3,2), -- 0.0–1.0
  extraction_confidence DECIMAL(3,2), -- 0.0–1.0
  rule_confidence DECIMAL(3,2), -- 0.0–1.0
  combined_confidence DECIMAL(3,2), -- min(mapping, extraction, rule)
  severity VARCHAR(50), -- 'critical', 'high', 'medium', 'low', 'informational'

  -- Lifecycle
  status VARCHAR(50) DEFAULT 'identified', -- State machine
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  created_by UUID,

  -- Triage
  triaged_at TIMESTAMP,
  triaged_by UUID,
  root_cause VARCHAR(255),

  -- Remediation
  remediation_owner UUID,
  due_date DATE,
  corrective_action TEXT,
  corrected_at TIMESTAMP,

  -- Retest
  retested_at TIMESTAMP,
  retested_by UUID,
  retest_result VARCHAR(50), -- 'passed', 'failed'

  -- Closure
  closed_at TIMESTAMP,
  closed_by UUID,
  resolution VARCHAR(255),

  -- Regulation
  regulation_citation VARCHAR(255), -- e.g., '12 CFR 1003.2'

  -- Audit trail link
  audit_event_id UUID, -- Foreign key to audit_trail.events

  FOREIGN KEY (audit_id) REFERENCES audits.audit_runs(audit_id),
  FOREIGN KEY (created_by) REFERENCES users(user_id),
  FOREIGN KEY (triaged_by) REFERENCES users(user_id),
  FOREIGN KEY (remediation_owner) REFERENCES users(user_id),
  FOREIGN KEY (retested_by) REFERENCES users(user_id),
  FOREIGN KEY (closed_by) REFERENCES users(user_id)
);
CREATE INDEX idx_findings_audit ON audits.findings(audit_id);
CREATE INDEX idx_findings_status ON audits.findings(status);
CREATE INDEX idx_findings_severity ON audits.findings(severity);
CREATE INDEX idx_findings_confidence ON audits.findings(combined_confidence);
----

[[sect-policy-documents-table]]
==== policy_documents (First-Class Policy Library)

*Strategic rationale:* The Institution Policy Profile (<<sect-institution-policy-profile>>) currently embeds a `policy_document_registry` array inside the profile JSON.
This table promotes policy documents to a first-class entity, decoupled from any specific audit or policy profile.
This decoupling enables three future capabilities at near-zero marginal cost:

. *Standalone Policy Library:* Institutions can manage their policy document inventory, review cycles, and version chains without creating an audit or a policy profile — delivering immediate value during onboarding, before the first audit runs.
. *Document Control Workflows:* The `status` lifecycle and `next_review_date` field enable a policy review cycle engine using the same state-machine and event-publishing infrastructure as audit runs and findings.
. *Cross-Audit Policy Reuse:* Multiple audit packs and policy profiles can reference the same policy document without duplication, maintaining a single source of truth for policy version chains.

[source,sql]
----
CREATE TABLE compliance.policy_documents (
  document_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL,
  institution_id UUID NOT NULL,

  -- Document identity
  title VARCHAR(512) NOT NULL,                 -- e.g., 'ABC Bank HMDA Data Integrity Policy v3.2'
  document_type VARCHAR(50) NOT NULL,          -- 'board_policy', 'compliance_policy', 'operating_procedure', 'job_aid', 'board_resolution', 'risk_appetite_statement'
  regulations_addressed TEXT[] NOT NULL,        -- e.g., '{12 CFR 1003, Reg C}'

  -- Version chain
  version VARCHAR(50),
  effective_date DATE NOT NULL,
  supersedes_document_id UUID,                 -- Previous version in the chain
  superseded_by_document_id UUID,              -- Next version (populated when superseded)

  -- Ownership & review cycle
  owner VARCHAR(255),                          -- Department or individual responsible
  department VARCHAR(255),
  review_cycle_months INT DEFAULT 12,          -- Board-mandated review cadence
  next_review_date DATE,                       -- Computed: effective_date + review_cycle_months; triggers review workflow when approaching
  last_reviewed_date DATE,
  last_reviewed_by UUID,

  -- Lifecycle
  status VARCHAR(50) DEFAULT 'draft',          -- 'draft', 'under_review', 'approved', 'published', 'expired', 'retired'

  -- Storage & extraction
  storage_reference VARCHAR(512),              -- URI to document in Azure Blob / institution's document store
  sha256_hash VARCHAR(64),                     -- Integrity verification
  ai_extraction_confidence DECIMAL(3,2),       -- Overall confidence from AI extraction pipeline

  -- Audit trail link
  audit_event_id UUID,                         -- Link to event store for full lifecycle history

  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

  FOREIGN KEY (tenant_id) REFERENCES tenants(tenant_id),
  FOREIGN KEY (supersedes_document_id) REFERENCES compliance.policy_documents(document_id),
  FOREIGN KEY (superseded_by_document_id) REFERENCES compliance.policy_documents(document_id)
);
CREATE INDEX idx_policy_docs_tenant ON compliance.policy_documents(tenant_id, institution_id);
CREATE INDEX idx_policy_docs_status ON compliance.policy_documents(status);
CREATE INDEX idx_policy_docs_review ON compliance.policy_documents(next_review_date) WHERE status = 'published';
CREATE INDEX idx_policy_docs_regulation ON compliance.policy_documents USING GIN (regulations_addressed);
----

*Relationship to Institution Policy Profile:* The Policy Profile's `policy_document_registry` array references `document_id` values from this table.
This table is the source of truth for document metadata; the profile captures the point-in-time snapshot used for a specific audit overlay.

*Review cycle engine (Phase 2+):* When `next_review_date` approaches (configurable alert window, default 30 days), the Workflow Engine creates a review workflow event.
The review follows the same dual-approval pattern as Policy Profile review (<<sect-ai-policy-pipeline>>, Phase 5).
Every review event is recorded in `audit_trail.events` with `entity_type = 'policy_document'`.

==== audit_trail.events (Immutable Log)

NOTE: This schema must match the authoritative definition in <<sect-audit-trail-event-store>>.

*Platform extensibility note:* The `entity_type` and `audit_id` columns are deliberately generic.
`entity_type` is a VARCHAR discriminator — not an enum — so that future platform modules (complaint management, standalone policy management, document control workflows) can store events in the same immutable log without schema migration.
`audit_id` is nullable for non-audit entities (e.g., standalone policy lifecycle events, complaint intake events).
This design ensures that the audit trail infrastructure built for MVP serves as the enterprise-grade event backbone for all future modules, avoiding costly event store duplication.

[source,sql]
----
CREATE TABLE audit_trail.events (
  event_id UUID PRIMARY KEY,                   -- UUID v7 (time-sortable)
  tenant_id UUID NOT NULL,                     -- Multi-tenant isolation
  sequence_id BIGINT NOT NULL,                 -- Global monotonic per tenant
  audit_id UUID,                               -- Nullable: non-audit entities (complaints, policy lifecycle) have no audit_id
  timestamp TIMESTAMPTZ NOT NULL,              -- Server-assigned only; no client timestamps
  entity_type VARCHAR(50) NOT NULL,            -- Extensible discriminator: 'finding', 'mapping', 'evidence', 'audit_run', 'policy_document', 'complaint', etc.
  entity_id VARCHAR(255) NOT NULL,
  event_type VARCHAR(100) NOT NULL,

  actor_id VARCHAR(255) NOT NULL,
  actor_role VARCHAR(50) NOT NULL,

  payload JSONB NOT NULL,                      -- Must be RFC 8785-serializable

  -- Hash chain for tamper detection (RFC 8785 canonical JSON + SHA-256)
  previous_event_hash VARCHAR(64) NOT NULL,
  event_hash VARCHAR(64) NOT NULL UNIQUE,

  -- Cryptographic anchoring (WORM-stored every N events)
  is_anchor BOOLEAN DEFAULT FALSE,
  anchor_hash VARCHAR(64),                     -- Populated only when is_anchor = true

  source_system VARCHAR(100) NOT NULL,
  api_version VARCHAR(20) NOT NULL,
  schema_version VARCHAR(20) NOT NULL DEFAULT '1.0',

  UNIQUE (tenant_id, sequence_id),             -- Enforces monotonic per tenant
  -- audit_id FK is NOT enforced: allows non-audit entities (policy_document, complaint) to use the event store
  -- Application-level validation ensures audit_id references a valid audit_run when entity_type is audit-scoped
);

-- Performance indexes
CREATE INDEX idx_events_audit ON audit_trail.events(audit_id, timestamp) WHERE audit_id IS NOT NULL;
CREATE INDEX idx_events_entity ON audit_trail.events(entity_id);
CREATE INDEX idx_events_entity_type ON audit_trail.events(entity_type, timestamp);
CREATE INDEX idx_events_actor ON audit_trail.events(actor_id);
CREATE INDEX idx_tenant_sequence ON audit_trail.events(tenant_id, sequence_id);
CREATE INDEX idx_anchor_events ON audit_trail.events(tenant_id, is_anchor) WHERE is_anchor = TRUE;

=== Data Types and Constraints

* *UUIDs:* All primary keys are UUIDs for distributed systems compatibility
* *Timestamps:* Stored in UTC, always timestamptz in PostgreSQL
* *Confidence scores:* DECIMAL(3,2) — allows 0.00 to 1.00 with 2 decimal precision
* *Text fields:* Use TEXT for findings/evidence; VARCHAR(n) for enums and short strings
* *JSON payloads:* JSONB in PostgreSQL for efficient querying and indexing
* *Hashes:* VARCHAR(64) for SHA-256 (hex-encoded)
* *Tenant isolation:* Every table includes tenant_id; row-level security policies enforce isolation at DB level (TODO: Implement RLS policies)

[[sect-versioning-schema-evolution]]
=== Versioning & Schema Evolution

*Approach:* Schema versioning via semantic versioning on tables; migrations tracked in schema_migrations table.

[source,sql]
----
CREATE TABLE schema_migrations (
  migration_id UUID PRIMARY KEY,
  migration_version VARCHAR(20), -- e.g., '1.1.0'
  description VARCHAR(512),
  executed_at TIMESTAMP,
  execution_time_ms INT,
  status VARCHAR(50), -- 'success', 'failed', 'rolled_back'
  sql_script TEXT,
  UNIQUE(migration_version)
);
----

*Policy:*

* New audit runs always use latest pack version and schema
* Active audits remain pinned to their pack version; no auto-upgrade
* Schema changes are backward-compatible within MINOR versions
* MAJOR schema changes require audit run completion or explicit migration

=== Analytics Storage (Separate Read-Only Replica)

*Rationale:* Aggregation queries (dashboards, trend reports) impact operational DB.
A read-only replica is cheaper and doesn’t contend with transactional workload.

*Approach:* Stream findings, status transitions, and metrics to a separate analytics database (Aurora or Redshift) via Change Data Capture or nightly batch export.
