== 25. Model Deployment: Mistral Self-Hosted (Production Decision)

____
*Executive decision (2026-03-01): YES.* AuditIQ should self-host selected open-weight Mistral models for regulated production extraction, with a deliberate hybrid fallback to cloud for difficult edge cases and burst elasticity. This section records the rationale, target architecture, and implementation plan.
____

=== 25.1 Decision Summary

AuditIQ's extraction workload is a strong fit for controlled local inference:

* High-throughput, batch-oriented extraction where queueing and continuous batching improve utilization.
* Strong data residency and privacy requirements.
* Audit-grade provenance requirements (model/version pinning, request traces, reproducibility).
* Existing Docling-Graph architecture already supports local runtimes (vLLM/Ollama) and remote providers through LiteLLM.

*Production posture:*

* *Default path (self-host):* `Mistral Small 3.2 (24B)` as primary production extraction model.
* *Cost/latency tier (self-host):* `Ministral 3 (8B/14B)` for lower-complexity or high-volume jobs.
* *Exception path:* Route low-confidence OCR, schema-failure retries, and complex edge documents to `Mistral Large 3` (if provisioned) or approved cloud models through LiteLLM policy routing.

=== 25.2 Why This Is the Right Choice for AuditIQ

[width="100%",cols="30%,70%",options="header"]
|===
|Driver |AuditIQ Implication

|*Residency + compliance*
|Self-hosting keeps documents/prompts in AuditIQ-controlled infrastructure and simplifies evidence generation for audits.

|*Provenance control*
|AuditIQ can pin model artifacts, capture digests, control rollout cadence, and retain deterministic execution metadata.

|*Framework fit*
|Docling-Graph already supports LiteLLM-based routing to local or cloud endpoints with structured JSON enforcement.

|*Throughput control*
|vLLM provides OpenAI-compatible serving, continuous batching, and queue-friendly operations for batch extraction.

|*Cost model realism*
|Self-hosting is not automatically cheapest per token; it wins when utilization is high, fallback is constrained, and compliance requirements dominate.
|===

=== 25.4 Production Architecture Pattern

*Target serving pattern:*

[arabic]
. Docling conversion and chunking run in standard AuditIQ pipeline.
. LiteLLM router enforces model policy (allowlist + routing rules).
. Default traffic goes to private vLLM OpenAI-compatible service.
. Fallback traffic can route to approved cloud providers for overflow/edge complexity.
. Response handling enforces structured output and JSON repair before downstream graph materialization.
. Provenance logs capture model digests, config hashes, runtime metadata, and timing.

=== 25.5 Security, Compliance, and Provenance Requirements

* *Private inference boundary:* No public inference endpoints. Run within private subnets/VPC.
* *Transport security:* mTLS/TLS between pipeline services, router, and inference runtime.
* *PII controls remain mandatory:* Apply deterministic redaction before LLM calls for both self-hosted and fallback paths.
* *Model registry (deny-by-default):* Permit only approved model IDs and artifact digests.
* *Provenance fields (minimum):* `doc_sha256`, `prompt_sha256`, `schema_sha256`, `response_sha256`, model ID, model artifact digest, container digest, runtime version, decoding params, queue/latency metrics, fallback reason.
* *Legal gate:* Legal/compliance sign-off required before production rollout of each model family and checkpoint.

=== 25.6 Hardware and Scaling Guidance

____
*Planning note:* Capacity must be validated with AuditIQ document distributions and Docling-Graph contract modes. Published throughput claims are directional only.
____

[width="100%",cols="25%,25%,50%",options="header"]
|===
|Infra tier |Typical GPU profile |Practical guidance

|*Entry tier*
|A10G 24GB
|Good for 8B and some 14B quantized workloads; use for early production canary lanes.

|*Core production tier*
|A100 40/80GB
|Better headroom for 24B models, larger contexts, and higher concurrency.

|*High-end tier*
|H100 80GB
|Preferred for high-throughput clusters and heavy long-context workloads.

|*Frontier escalation tier*
|Multi-GPU node (A100/H100)
|Required for practical Mistral Large 3 operation; treat as specialized exception capacity.
|===

Operational guardrails:

* Cap max context length and batched tokens to avoid KV-cache-driven OOM.
* Enforce admission control and queue backpressure.
* Prefer data-parallel replicas for smaller models; use tensor parallelism only when memory constraints force it.

=== 25.7 Rollout Plan and Go/No-Go Gates

*Stage A — Baseline (cloud):*

* Establish extraction quality, latency, and cost baselines with current cloud setup.

*Stage B — Self-host canary:*

* Route 5–10% traffic to vLLM self-hosted tier through LiteLLM.
* Compare F1, schema-valid JSON rate, retry/repair rate, P95 latency, and fallback rate.

*Stage C — Hybrid production default:*

* 80–95% self-host default path.
* Keep cloud fallback for burst and exception handling.

*Go/No-Go acceptance gates:*

* Quality: agreed F1 target met; schema-valid JSON >=99% post-repair.
* Reliability: fallback rate within threshold and queue drains within SLA windows.
* Compliance: model license and provenance controls approved.
* Security: private networking, secrets management, and logging controls validated.
* Cost: per-extraction cost meets approved budget threshold.
