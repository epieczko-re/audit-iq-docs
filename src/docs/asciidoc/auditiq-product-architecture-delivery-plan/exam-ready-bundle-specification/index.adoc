[[sect-exam-ready-bundle]]
== Exam-Ready Bundle Specification

=== Purpose & Design Philosophy

The exam-ready bundle is a *first-class product object* — not a one-time export, but a continuously updated dossier that the platform maintains throughout the audit lifecycle.
It represents the complete, examination-ready state of an audit at any point in time: coverage, findings, evidence, approvals, exceptions, and change history.

*What buyers actually want:* A single artifact they can hand to an examiner that proves the audit was thorough, well-governed, and defensible — without the examiner needing to navigate the platform.
The exam pack must answer these examiner questions on its own:

* What was tested, and how comprehensively? (Coverage & sampling)
* What was found, and how certain are we? (Findings & confidence)
* What evidence supports each finding? (Evidence links & provenance)
* Who reviewed and approved what? (Approval manifests & Three Lines Model certification)
* What changed since the last version? (Change log)
* Were there any overrides or exceptions? (Exceptions log)

=== Bundle Versioning & Lifecycle

The exam pack is created when the audit enters `InProgress` state and updated on every material event.
Each update produces a new bundle version linked to its predecessor, forming an auditable version chain.

*Version triggers (automatic):*

* Finding created, closed, reopened, or escalated
* Evidence superseded or new evidence attached
* Approval recorded by any line of defense
* Corrective action completed or verified
* Sampling methodology updated

*Version semantics:*

* *Major version (v1 → v2):* Structural changes — new findings added, findings reopened, approval status changed
* *Minor version (v1.0 → v1.1):* Metadata updates — corrective action progress, evidence additions, narrative edits

*Bundle state machine:*

* *Draft:* Bundle created at audit start; updated continuously as findings and evidence accumulate
* *Review:* Manager initiates bundle review; integrity checks run automatically
* *Approved:* All required approvals collected; bundle locked for export
* *Exported:* Bundle downloaded or transmitted to examiner; hash recorded
* *Superseded:* New version created after post-export changes; previous version preserved in version chain

=== Detailed File Specifications

==== A. Audit Summary Report (PDF)

*Template structure:*

....
[Cover Page]
- Organization name, audit period, pack name, date
- Auditor name & signature block
- Bundle version and generation timestamp

[Executive Summary]
- Scope (record count, sampling methodology)
- Overall compliance assessment (Pass / Fail / Pass w/ Observations)
- Key findings summary by severity
  * Critical: N findings
  * Major: N findings
  * Minor: N findings
- Coverage summary (% of population tested, confidence distribution)
- Recommended corrective actions (summary)

[Detailed Findings]
- Table: Finding ID | Rule | Severity | Confidence | Evidence Link | Narrative | Approver

[Coverage & Sampling Summary]
- Population tested, sample design, stratification
- Confidence distribution histogram
- Rule coverage matrix (which rules applied, which passed)

[Approval History]
- Three Lines Model sign-off summary
- Key events (creation, approvals, re-reviews)
- Timestamp, actor, action

[Attestation]
- "This audit was performed using AuditIQ Audit Pack [version]."
- "Audit trail, evidence, and scoring methodology documented in bundle."
- Digital signature or manager approval stamp
....

==== B. Findings Register (CSV + PDF)

*CSV columns:*

....
finding_id,rule_id,rule_name,target_record_id,severity,confidence_extraction,confidence_mapping,confidence_rule,combined_confidence,evidence_ids,narrative,approver_id,approval_date,status,defense_line,override_applied,override_reason
....

*PDF:* Same data, formatted as professional table with severity color-coding and confidence tier indicators.

==== C. Corrective Action Log (CSV)

*Columns:*

....
finding_id,corrective_action_description,responsible_party,target_completion_date,actual_completion_date,evidence_reference_id,status,sla_met,retest_result,retested_by,retested_at
....

*Example:*

....
finding_001,"Implement missing disclosure language per Reg Z",mgmt_team_lead,2026-03-31,2026-03-28,evidence_reg_z_template,closed,true,passed,reviewer_002,2026-03-29T10:00:00Z
....

=== Bundle API Endpoints

The Exam Pack Lifecycle Manager exposes the following API surface (see <<sect-exam-pack-lifecycle-manager>> for full contracts):

* `GET /api/v1/audits/{audit_id}/bundle` — Returns the current (latest) bundle metadata
* `GET /api/v1/audits/{audit_id}/bundle/versions` — Returns the version chain
* `GET /api/v1/audits/{audit_id}/bundle/versions/{version}` — Returns a specific version
* `GET /api/v1/audits/{audit_id}/bundle/diff?from={v1}&to={v2}` — Returns the delta between two versions
* `POST /api/v1/audits/{audit_id}/bundle/approve` — Records a line-of-defense approval
* `POST /api/v1/audits/{audit_id}/bundle/export` — Generates and returns the ZIP bundle for a specific version
* `GET /api/v1/audits/{audit_id}/bundle/coverage` — Returns current coverage and sampling statistics
* `GET /api/v1/audits/{audit_id}/bundle/exceptions` — Returns current exceptions log
