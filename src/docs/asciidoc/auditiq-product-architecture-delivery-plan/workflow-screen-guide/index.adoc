== Workflow & Screen Guide by Lifecycle Stage

____
This section groups every MVP screen and workflow by audit lifecycle stage.
For each stage, you will find: the screens involved, the FR IDs and acceptance criteria that apply, diagram references, and a Developer Context note previewing which services and components are involved.
Detailed wireframes are embedded in their respective specification sections (Part IV and Part VI) for in-context reference; the Screen Inventory table and end-to-end workflow diagrams follow this guide in the next section.

*Read this section first when estimating sprint work.*
Each lifecycle stage maps directly to one or more sprints in the Workflow Delivery Matrix (<<sect-workflow-delivery-matrix>>).
____

=== Stage 1: Upload & Ingestion

This stage covers audit creation, configuration, dataset upload, document upload, and initial validation.
The user creates an audit run, selects an audit pack, uploads the HMDA LAR target dataset and source documents, and monitors ingestion progress.

==== Screens

|===

|WF-01
|<<wf-audit-dashboard, Audit Dashboard>>
|FR-10.1, FR-10.4
|Dashboard shows all audits with RBAC-filtered visibility; status badges, progress metrics, quick actions.

|WF-02
|<<wf-audit-config, Audit Configuration>>
|FR-10.2, FR-10.5
|Create audit run with pack selection, sample size, confidence threshold, dataset upload; validates schema against pack.

|WF-04
|<<wf-doc-upload, Document Upload>>
|FR-1.3, FR-1.4, FR-1.5
|Drag-drop or batch upload of PDF/TIFF/ZIP; auto-classification with confidence; loan-record matching.

|WF-03
|<<wf-audit-progress, Audit Progress Monitor>>
|FR-10.3, FR-10.4
|Real-time phase progress (upload, extraction, mapping, rules, review, finalization); AFC routing distribution.
|===

==== Happy Path

[arabic]
. Manager creates a new audit → selects HMDA origination pack → defines sample (50 loans) and date range.
. Manager uploads HMDA LAR CSV (target dataset) → system validates 37 Key Data Fields against pack schema → confirms record count.
. Reviewer uploads source documents (CDs, 1003s) → system auto-classifies and matches to LAR records by loan number.
. Audit enters `DocumentsUploaded` state → extraction begins automatically.

==== Key Edge Cases

* *Schema validation failure:* Missing required columns → system rejects upload with specific error listing; user re-uploads corrected file.
* *Unmatched documents:* Document classification confidence < 0.85 → routed to manual classification queue.
* *Oversized file:* > 100 MB single file → client-side rejection before upload begins (FR-1.5).

==== Developer Context

*Services:* Ingestion Service, Azure Blob Storage (encryption + quarantine), Document Classifier.
*Data flow:* CSV → Ingestion Service (validation + SHA-256 checksums) → PostgreSQL staging tables.
PDFs → Blob Storage → Document Classifier → link to LAR records.
*API contracts:* `IngestionRequest`, `ValidationResult` interfaces (see <<sect-product-requirements>>).
*Architecture detail:* <<sect-architecture-by-workflow, Upload & Ingestion stage>>.

=== Stage 2: Extraction & Mapping

This stage covers document processing through Docling-Graph, AFC scoring, field extraction review, schema mapping, and synonym dictionary management.
The system extracts structured data from source documents and maps extracted fields to the HMDA/MISMO target schema.

==== Screens

|===
|WF # |Screen |FR IDs |Acceptance Criteria Summary

|WF-05
|<<wf-processing-status, Processing Status>>
|FR-2.1, FR-2.5
|Per-document extraction progress, page grades, fallback triggers, field counts.

|WF-06
|<<wf-extraction-review, Field Extraction Review>>
|FR-2.1, FR-2.2, FR-2.3
|Split-pane: document viewer (left) + extracted fields with AFC badges (right); accept/edit/enter actions.

|WF-07
|<<wf-doc-viewer, Document Viewer>>
|FR-2.4
|Full-page viewer with zoom, search, field bounding-box overlays; loads within 3s per page.

|WF-08
|<<wf-mapping-review, Mapping Review>>
|FR-3.1, FR-3.2, FR-3.3
|Two-layer confidence routing; auto-accept (>=0.95), human review (0.70-0.94), manual (<0.70); alternatives + override.

|WF-09
|<<wf-synonym-dict, Synonym Dictionary>>
|FR-3.4
|CRUD for curated synonym pairs; overrides from mapping review feed back here for approval.
|===

==== Happy Path

[arabic]
. System processes uploaded documents through Docling-Graph pipeline → extracts text, tables, key-value pairs.
. AFC v2 scoring (6 signals including `graph_corrob_score`) routes each field: >=0.85 auto-confirm, 0.50-0.84 human review, <0.50 manual entry.
. Reviewer validates flagged fields in split-pane extraction review → accepts or overrides extracted values.
. Schema mapping service suggests field mappings with two-layer confidence → reviewer approves mapping set.
. Approved mappings feed into rules engine.

==== Key Edge Cases

* *Low-quality scan:* Page grade POOR/FAIR → 3-tier fallback: VLM → LLM (Mistral via LiteLLM) → cloud OCR (Azure).
Each fallback logged with `extraction_method` provenance.
* *CD-only coverage gap:* With CD-only documents, expect ~10-20% of HMDA fields at >=0.85 confidence.
Full coverage requires 1003, appraisal, and title documents (Phase 2).
* *High override rate:* If override rate >25%, triggers confidence recalibration alert (<<sect-glossary>>).

==== Developer Context

*Services:* Document Processing Service (Docling-Graph + AFC), Schema Mapping Service (hybrid heuristic matcher), Synonym Dictionary.
*Data flow:* Blob Storage → Docling-Graph pipeline → Pydantic template extraction → AFC scoring → field routing → Schema Mapping Service → approved mappings → Rules Engine.
*API contracts:* 3-tier fallback chain pseudocode, `FindingOutput` interface (see <<sect-product-requirements>>).
*Architecture detail:* <<sect-architecture-by-workflow, Extraction and Mapping stage>>.

|WF-13
|<<wf-escalation, Manager Escalation>>
|FR-4.1

|===

==== Happy Path

[arabic]
. Rules engine applies pack rules against mapped data for all 37 Key Data Fields → generates preliminary findings with combined confidence scores.
. LLM drafts finding narratives with citations → AI provenance logged (21-field schema).
. Findings enter review queue sorted by severity and confidence.
. Reviewer opens each finding → exercises effective challenge (SR 11-7) → approves, edits, or rejects.
. Critical/High findings auto-escalate to manager → manager approves or requests re-review.
. Approved findings enter remediation lifecycle.

==== Key Edge Cases

* *False positive:* Reviewer rejects AI-generated finding → must provide rationale → finding marked as false positive with audit trail event.
* *Stage 1 error rate breach:* Error rate approaches pack-defined threshold (e.g., 5% for HMDA) → system displays threshold proximity alert → proactive sample expansion.
* *Automation bias:* Reviewer approves without effective challenge → system enforces mandatory challenge documentation; randomized challenge prompts (R26 mitigation).

==== Developer Context

*Services:* Normalization Engine, Comparison & Rules Engine, LLM Integration (LiteLLM/Mistral), Findings Management Service, Provenance Logger.
*Data flow:* Approved mappings + pack rules → Normalization → Comparison Engine → findings → LLM narrative → review queue → effective challenge → approved findings.
*AI governance:* 3 required disclaimers per AI output; 21-field provenance schema per AI call; PII redaction before all LLM calls.
*Architecture detail:* <<sect-architecture-by-workflow, Rule Execution stage>>.

|WF-17
|<<wf-bundle-export, Exam-Ready Bundle Export>>
|FR-8.4, FR-8.5, FR-8.6, FR-8.7

|WF-18
|<<wf-findings-register, Findings Register>>
|FR-8.1
|Complete findings list with export, filtering, and status summary.
|===

==== Happy Path

[arabic]
. Finding enters `Identified` state → reviewer triages (confirms severity, assigns remediation owner).
. Owner receives notification → submits corrective action with evidence within SLA.
. Independent reviewer verifies correction → retest passed → manager approves closure (required for Critical/High).
. All findings closed → manager generates audit summary report.
. Exam-ready bundle exported as versioned ZIP (14 artifacts) → three-lines-of-defense certification.

==== Key Edge Cases

* *SLA breach:* Remediation owner misses SLA → aging dashboard alerts manager → automatic escalation.
* *Retest failed:* Reviewer determines correction insufficient → finding returns to `Remediation Assigned` with notes.
* *Post-export update:* Finding reopened after bundle export → Exam Pack Lifecycle Manager auto-creates new bundle version; previous version marked `superseded`.

==== Developer Context

*Services:* Findings Management Service (state machine), Remediation Workflow Engine, Reporting Service, Exam Pack Lifecycle Manager.
*Data flow:* Approved findings → remediation state machine → SLA tracking → report generation → bundle packaging.
*State machine:* 9 states (Identified → Triaged → Remediation Assigned → Corrected → Retest Passed/Failed → Closed → Escalated → Reopened).
See <<sect-finding-remediation>> for full state diagram.
