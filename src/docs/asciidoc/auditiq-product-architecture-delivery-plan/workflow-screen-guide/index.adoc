== UI Workflows, Screens & User Journeys

____
This section is the single reference for every MVP screen, user workflow, and lifecycle stage in AuditIQ.
It merges the previous Workflow & Screen Guide and UI Workflows sections into one narrative that flows from audit creation through exam-ready export.

*How to read this section:*

* *Screen Inventory* — quick-reference table of all 26 MVP screens with priority and component notes.
* *End-to-End Workflow Diagrams* — seven flowcharts showing the primary user journeys through the system.
* *Lifecycle Stages 1–6* — per-stage detail: screens, wireframes, happy paths, edge cases, and developer context.
* *Functional requirements* live in <<sect-product-requirements>>.
Each screen entry here cites the applicable FR IDs and links back to Part IV for full acceptance criteria and contracts.

*Read this section first when estimating sprint work.*
Each lifecycle stage maps directly to one or more sprints in the Delivery Plan (<<sect-delivery-plan>>).
____

=== UI Development Strategy

____
*Dual-track development:* AuditIQ follows both UI-driven and API-driven development practices.
The workflows below define the user journeys; the API contracts in <<sect-component-design>> define the backend interfaces.
____

*Design Principles (Flowbite / Tailwind CSS):*

* *Sidebar navigation* — persistent left sidebar with grouped nav items; collapsible on mobile (Flowbite Sidebar component)
* *Card-based layout* — each logical section wrapped in a bordered card with header (Flowbite Card component)
* *Data tables* — sortable, filterable tables with pagination for all list views (Flowbite Table component)
* *Badge components* — color-coded pills for confidence scores, severity levels, and status (Flowbite Badge component)
* *Modal dialogs* — confirmation dialogs for destructive actions and approval workflows (Flowbite Modal component)
* *Form inputs* — labeled inputs with validation feedback for all data entry (Flowbite Forms)
* *Breadcrumb navigation* — contextual breadcrumbs in top bar for deep navigation (Flowbite Breadcrumb)
* *Toast notifications* — non-blocking status messages for async operations (Flowbite Toast)
* *Confidence is visible everywhere* — color-coded badges per <<sect-glossary, UI Confidence Badge Colors>>
* *AI provenance is always one click away* — every AI-generated element links to its provenance record
* *Effective challenge is enforced by design* — reviewer cannot approve without documenting their independent verification
* *Audit trail is ambient* — every action generates an event; users see event count and last-event timestamp per artifact

*Color Legend (workflow diagrams):*

* *Blue* (#4A6FA5) = User actions (auditor/reviewer interactions)
* *Purple* (#7B68EE) = System actions (deterministic processing)
* *Teal* (#20B2AA) = AI actions (LLM or ML-assisted)
* *Orange* (#FF7E2D) = Decision points (routing logic)
* *Dark* (#2B3E50) = Data / configuration / annotation boxes
* *Green* (#2E8B57) = Success / confirmed states
* *Red* (#C0392B) = Error / manual entry / rejection states
* *Maroon* (#8B0000) = Terminal completion

==== Happy Path

[arabic]
. System processes uploaded documents through Docling-Graph pipeline → extracts text, tables, key-value pairs.
. AFC v2 scoring (6 signals including `graph_corrob_score`) routes each field: >=0.85 auto-confirm, 0.50-0.84 human review, <0.50 manual entry.
. Reviewer validates flagged fields in split-pane extraction review → accepts or overrides extracted values.
. Schema mapping service suggests field mappings with two-layer confidence → reviewer approves mapping set.
. Approved mappings feed into rules engine.

==== Key Edge Cases

* *Low-quality scan:* Page grade POOR/FAIR → 3-tier fallback: VLM → LLM (Mistral via LiteLLM) → cloud OCR (Azure).

*Data flow:* Blob Storage → Docling-Graph pipeline → Pydantic template extraction → AFC scoring → field routing → Schema Mapping Service → approved mappings → Rules Engine.
