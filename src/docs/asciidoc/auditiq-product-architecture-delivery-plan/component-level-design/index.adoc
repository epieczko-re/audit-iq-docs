[[sect-ingestion-service]]
=== Ingestion Service

The Ingestion Service is the entry point for all structured data entering AuditIQ.
It receives *target datasets* — the tabular loan-level records (e.g., HMDA LAR files) that define the population of loans under audit — and validates, normalizes, and stages them for downstream processing.

*Why a separate service from Document Processing:* Target datasets and source documents follow fundamentally different pipelines.
Target datasets are structured data (CSV rows, API payloads) that require schema validation, encoding normalization, and relational staging — all deterministic operations with no AI involvement.
Source documents (PDFs, scanned images) require OCR, layout analysis, field extraction, and confidence scoring — the domain of the <<sect-doc-processing, Document Processing Service>>.
Separating these concerns keeps the ingestion path fast, deterministic, and independently testable, while the document processing path can scale separately with its heavier compute requirements (Docling, OCR fallback, AFC scoring).

*Relationship to Document Processing:* The two services operate on different data types within the same audit but converge downstream at the <<sect-schema-mapping-service, Schema Mapping Service>>.
The Ingestion Service produces validated target records in the staging area; the Document Processing Service produces extracted fields from source documents.
The Schema Mapping Service then maps extracted document fields onto the target schema so the <<sect-comparison-engine, Comparison Engine>> can compare source evidence against target values.

*When the Document Processing Service is not used:* In *Manual* execution mode, the Ingestion Service stages the target dataset and source documents are uploaded to Blob Storage, but no automated extraction occurs.
Reviewers compare documents to target records manually in a side-by-side viewer.
In *Automated Pre-Scan* mode, both target and source data are structured (no PDFs), so the Ingestion Service handles both inputs and the Document Processing Service is bypassed entirely.
Only *OCR-Assisted* mode engages both services in sequence.

*Responsibilities:*

* Build AuditIQ Field Confidence (AFC) scores per extracted field (see below)
* Extract text, tables, key-value pairs via deterministic field extractor layer on top of Docling
* Fall back to cloud OCR for pages graded POOR or FAIR by Docling
* Store extracted text, AFC metadata, and Docling page-level confidence grades

[IMPORTANT]
====
*Critical Architecture Clarification (per Docling Technical Validation Report v1):* Docling is a *document conversion + layout understanding engine*, not a per-field extraction system.
Its confidence scores (`layout_score`, `ocr_score`, `parse_score`) are *conversion-quality indicators* (POOR/FAIR/GOOD/EXCELLENT) that grade how well the page was converted — they are NOT per-field extraction confidence.
AuditIQ must build its own *deterministic field extractor* that consumes Docling’s `DoclingDocument` output and produces per-field AFC scores.
Do not use Docling’s numeric scores directly for field-level gating.
====

==== Docling-Graph Integration

*Overview.* AuditIQ adopts *Docling-Graph* as the Phase 1 baseline for document extraction and structured output.
Docling-Graph combines Docling’s document conversion and VLM extraction capabilities with a Pydantic-based schema enforcement layer and a directed knowledge graph.
This transforms free-text and OCR outputs into validated, strongly-typed Pydantic objects and a directed graph of entities and relationships that the Schema Mapping Service and Comparison Engine consume.

*Primary pipeline (Phase 1):*

....
Docling (document conversion & VLM)
  → Docling-Graph (Pydantic templates & structured extraction)
    → Knowledge Graph (directed graph with stable node IDs & edge metadata)
      → Schema Mapping Service (graph-aware mapping)
        → Comparison & Scoring Engine
....

*Why Docling-Graph:*

* Deterministic, schema-validated outputs (Pydantic objects) — extraction results are typed and validated at extraction time, not downstream
* Cross-document corroboration through explicit graph edges — improves `cross_doc_redundancy` and adds a new `graph_corrob_score` AFC signal
* Exportable graph artifacts for exam-ready bundles (CSV, Cypher, Neo4j) — supports <<sect-exam-ready-bundle>> evidence requirements
* Supports both VLM and LLM extraction backends — flexible inference routing
* Trace/debug exports for provenance and troubleshooting
* *MIT-licensed* — lower licensing risk than GPL-linked alternatives

_See: Docling-Graph README (https://github.com/docling-project/docling-graph)_

*LLM Integration (LiteLLM / Mistral):* Docling-Graph integrates with *LiteLLM* — a single interface to local and remote LLM runtimes (vLLM, Ollama) and cloud providers (Mistral, OpenAI).
For Phase 1 the recommended LLM inference path is *self-hosted Mistral* using vLLM or Ollama orchestrated via LiteLLM, with `provider_override` set to `mistral` in Docling-Graph configs.
This provides high throughput, lower per-call cost at scale, and improved data residency controls while preserving the option to use cloud LLMs during early validation.
See <<sect-model-deployment, Model Deployment>> for hardware and operational guidance.

*Integration guidance:*

* *Templates:* Implement Pydantic templates for each key document type (Closing Disclosure, Loan Estimate, 1003/URLA) and version them.
Templates must include `model_config` fields for stable graph IDs.
* *Fallback strategy:* Prefer Docling-Graph LLM extraction (LiteLLM → self-hosted Mistral) when Docling page grade or initial AFC is borderline.
Use cloud OCR fallback only for documents where Docling conversion fails at the page level (e.g., encrypted PDFs, severely degraded scans).
See updated fallback logic in <<sect-fallback-logic>>.
* *Provenance:* Every extracted Pydantic object records `pydantic_template_id`, `template_version`, `graph_node_id`, and `graph_export_version` in the extraction metadata (see <<sect-ai-provenance>> for provenance fields).
* *Legal:* Docling-Graph is MIT-licensed.
If using self-hosted Mistral model weights, obtain legal sign-off confirming the chosen Mistral model license permits self-hosting under your deployment terms.

==== AuditIQ Field Confidence (AFC)

AFC is a deterministic, per-field composite score built from six signals (v2 — updated to include Docling-Graph corroboration):

[cols="20%,10%,30%,40%",options="header"]
|===
|Signal |Weight |Source |Description
|`docling_page_grade` |0.12 |Docling conversion output |Page-level conversion quality (POOR=0.2, FAIR=0.5, GOOD=0.8, EXCELLENT=1.0)
|`ocr_segment_confidence` |0.23 |Docling OCR or cloud OCR fallback |Character-level OCR confidence for the text segment containing this field
|`pattern_validity` |0.20 |AuditIQ field extractor |Regex/parse validation (e.g., date format, currency pattern, HMDA enum membership)
|`layout_anchor_consistency` |0.18 |AuditIQ field extractor |Does the field appear in the expected document region with expected label proximity?
|`cross_doc_redundancy` |0.12 |AuditIQ field extractor |Does the same value appear in a corroborating document (e.g., loan amount on both CD and LE)?
|`graph_corrob_score` |0.15 |Docling-Graph knowledge graph |Graph-based corroboration: does the extracted value appear as a graph node with matching normalized values and correct linking edges across documents? Exploits Docling-Graph's directed knowledge graph for explicit entity-relationship validation.
|===

NOTE: AFC v1 (5-signal) remains valid during Phase 0 POC and early Phase 1a until Docling-Graph pipeline is integrated. Systems must support both v1 and v2 formulas and log which version was used.

*AFC formula (v2 — with Docling-Graph):*

....
afc = (docling_page_grade * 0.12) +
      (ocr_segment_confidence * 0.23) +
      (pattern_validity * 0.20) +
      (layout_anchor_consistency * 0.18) +
      (cross_doc_redundancy * 0.12) +
      (graph_corrob_score * 0.15)
....

NOTE: Weights are calibration targets — tune during Phase 1a using labeled extraction data. See Canonical Glossary (Appendix) for routing thresholds (unchanged: ≥0.85 auto-confirm, 0.50–0.85 human review, <0.50 manual entry).

*AFC routing thresholds:*

* `afc >= 0.85`: Auto-confirm (no human review required)
* `0.50 <= afc < 0.85`: Human review required (field highlighted in UI)
* `afc < 0.50`: Flag as extraction failure; route to manual entry queue

[[sect-fallback-logic]]
==== Fallback Logic (Refined — Docling-Graph-First)

*Phase 1 fallback chain:* Docling-Graph (VLM) → Docling-Graph (LLM via LiteLLM/Mistral) → Cloud OCR → Manual entry

....
for each document:
  // Step 1: Attempt Docling-Graph structured extraction (VLM backend)
  context = docling.convertDocument(document)
  graph_result = docling_graph.run_pipeline(
    context, template=pydantic_template, backend="vlm"
  )
  compute afc including graph_corrob_score

  if graph_result.success and afc >= 0.50:
    persist extraction (Pydantic objects + graph)
    recordExtractionEvent(auditId, documentId, 'docling_graph_vlm')
  else:
    // Step 2: Attempt LLM extraction via LiteLLM (self-hosted Mistral/vLLM)
    llm_result = docling_graph.run_pipeline(
      context, template=pydantic_template, backend="llm",
      provider="litellm", provider_override="mistral"
    )
    recompute afc

    if llm_result.success and afc >= 0.50:
      persist extraction (Pydantic objects + graph)
      recordExtractionEvent(auditId, documentId, 'docling_graph_llm_mistral')
    else:
      // Step 3: Fall back to cloud OCR or manual entry
      triggerCloudOCRFallback(document)
      recordFallbackEvent(auditId, documentId, 'cloud_ocr_fallback')
    end
  end
end
....

*Legacy page-level fallback (still applies within each step):*

....
for each page in document:

[[sect-matching-algorithm]]
==== Matching Algorithm (Phase 1 — Hybrid Heuristic)

The matcher uses a COMA-style combination of multiple independent signals, weighted to produce a raw mapping confidence:

* *Name similarity (0.30):* Levenshtein distance + Jaccard token overlap on field names, boosted by MISMO field description matching (uses `mismo_dictionary.description`)
* *HMDA/CD synonym dictionary (0.25):* Curated domain-specific equivalence table mapping known HMDA field names to CD label variants, abbreviations, and industry synonyms (e.g., `loan_type` ↔ `mortgage_product`, `prop_addr` ↔ `property_address`)
* *Type compatibility (0.20):* Target (MISMO) and source types must be compatible; MISMO enum fields validated against `mismo_dictionary.enum_values`
* *Document-section context (0.15):* Boost confidence when source field appears in expected document section (e.g., loan amount in CD page 1, Section J)
* *Historical acceptance (0.10):* If this mapping has been approved before by any reviewer in this tenant, confidence += up to 0.10 (decayed by recency)

*Mapping Pattern Taxonomy (5 types):*

[cols="34%,33%,33%",options="header"]
|===
|Pattern |Example |Signal Used
|Exact name match |`loan_amount` → `loan_amount` |Name similarity = 1.0
|Abbreviation mismatch |`prop_addr` → `property_address` |Synonym dictionary
|Synonym |`loan_type` → `mortgage_product` |Synonym dictionary
|Derived field |`ltv_ratio` (computed) |Rule engine; flagged as “derived” not “mapped”
|Multi-source field |`total_income` (W-2 + 1003 + pay stubs) |Flagged as “Phase 2: requires additional documents”
|===

[[sect-two-layer-confidence]]
==== Two-Layer Confidence Design

*Layer 1 — Raw signals (produced per mapping suggestion):*

....
conf_mapping = (
  (name_sim * 0.30) +
  (synonym_match * 0.25) +
  (type_compat * 0.20) +
  (doc_section_context * 0.15) +
  (historical_acceptance * 0.10)
)
....

where each component is normalized to [0.0, 1.0].

Additional raw signals passed to Layer 2: - `conf_extraction`: AFC score from <<sect-doc-processing>> (how confident we are in the extracted value) - `conf_rule`: Business rule pass/fail from the Comparison &amp; Rules Engine (does the mapped value pass validation?)

*Layer 2 — Calibrated decision score:*

* Combines `conf_mapping`, `conf_extraction`, and `conf_rule` into a single calibrated probability
* Phase 1: Temperature scaling on logistic regression of Layer 1 signals (requires ≥200 labeled mapping decisions for initial calibration)
* Phase 2: Isotonic regression for non-parametric calibration (requires ≥500 labeled decisions)
* Calibration validation: Reliability diagrams (predicted confidence vs. actual accuracy in 10 bins); Expected Calibration Error (ECE) must be ≤0.05

*Routing thresholds (calibrated score):*

* ≥0.95: Auto-accept (reviewer sees green checkmark; ~10s visual confirmation).
NOTE: Initial conservative threshold per <<sect-decision-d2, Decision D2>>; may be lowered to ≥0.90 after calibration validation on ≥500 mappings.
* 0.70–0.95: Human review required (reviewer evaluates suggestion; ~45s per field)
* <0.70: Manual mapping required (no suggestion shown; reviewer selects from field list)

==== Bootstrap Calibration Procedure

Layer 2 calibration requires labeled mapping decisions that do not exist at launch.
This section defines the bootstrap procedure that transitions the system from uncalibrated heuristic scoring to a calibrated decision model.

*Phase 0 — Pre-launch seed data (before first customer):*

[arabic]
. Generate synthetic mapping datasets from 5 sample HMDA LAR files (public HMDA data + redacted CDs).
. Internal team completes ~100 mapping decisions across 3 pack variants (HMDA origination, HMDA servicing, CRA).
. Train initial logistic regression on these 100 decisions using leave-one-out cross-validation.
. Expected initial ECE: 0.08–0.12 (above the ≤0.05 target — acceptable for soft launch).

*Phase 1a — Early customer calibration (first 3–5 customers):*

[arabic]
. Auto-accept threshold set to ≥0.98 (ultra-conservative — virtually no auto-accepts during bootstrap).
. Every reviewer accept/override/manual decision is captured with `calibratedScoreAtDecision` and `disposition` in `ApprovedMapping`.
. At 200 cumulative labeled decisions (across all tenants, anonymized), trigger first production calibration:
.. Split data: 80% train / 20% validation (stratified by disposition type).
.. Fit temperature-scaled logistic regression on train set.
.. Compute ECE on validation set.
.. If ECE ≤ 0.05: deploy calibrated model; lower auto-accept threshold to ≥0.95.
.. If ECE > 0.05 but ≤ 0.08: deploy but keep threshold at ≥0.98; schedule recalibration at 350 decisions.
.. If ECE > 0.08: keep heuristic scoring; flag for engineering review.

*Phase 1b — Continuous recalibration:*

[arabic]
. Recalibration runs automatically every 200 new decisions (batch) or weekly (whichever comes first).
. Reliability diagrams and ECE are computed and stored for each calibration run.
. If ECE degrades by >0.02 from the previous calibration, an alert is sent to the ML team and the auto-accept threshold reverts to ≥0.98 until the next calibration passes.

*Go/kill criteria for calibration quality:*

[cols="30%,35%,35%",options="header"]
|===
|Metric |Go (deploy calibrated model) |Kill (revert to heuristic)

|ECE
|≤ 0.05
|> 0.10 after 500+ decisions

|Auto-accept precision
|≥ 0.98 (false-accept rate < 2%)
|< 0.95

|Override rate on auto-accepted fields
|< 5% of auto-accepted fields subsequently overridden
|> 10%

|Reviewer trust signal
|> 80% of reviewers skip review of auto-accepted fields
|< 60% (indicates reviewers do not trust auto-accepts)
|===

*Data imbalance mitigation:* Early labeled data skews toward easy mappings (exact name matches).
To ensure calibration covers hard cases, the system tracks disposition rates by matching pattern (exact, abbreviation, synonym, derived, multi-source) and flags under-represented patterns for targeted reviewer sampling.

[[sect-mismo-alignment]]
==== MISMO Alignment Strategy

*Design Principle:* AuditIQ's canonical data model adopts MISMO (Mortgage Industry Standards Maintenance Organization) Reference Model naming conventions and field definitions for all loan-level attributes.
This alignment reduces translation friction when ingesting data from LOS, servicing, and origination systems — most of which already export MISMO-formatted data — and increases credibility with examiners who recognize MISMO field names.

*Scope of MISMO Adoption:*

* *Canonical loan data fields:* All loan-level attributes stored in `staging_datasets` and referenced in `findings` use MISMO Reference Model v3.x naming where a corresponding MISMO data point exists.
Examples: `LoanIdentifier`, `BorrowerMailingAddress`, `PropertyValuationAmount`, `ActionTaken` (HMDA), `RateSpreadPercent`.
* *Document classification:* Evidence documents are classified using the MISMO Document Custodian taxonomy (e.g., `BorrowerAuthorization`, `CreditReport`, `AppraisalReport`) for consistent naming across packs and evidence uploads.
* *HMDA Reporting Dataset:* Phase 1 HMDA packs map directly to the MISMO HMDA Reporting Dataset, which provides a well-defined field set for LAR data.
This covers all fields required for HMDA compliance auditing.
* *Audit Pack field references:* Pack `field_mappings` definitions reference MISMO canonical names as the target schema.
Source fields (from customer uploads) are mapped _to_ MISMO names by the schema mapping engine.

*What is NOT MISMO-aligned:*

* *Audit operational tables:* `audit_runs`, `findings`, `field_mappings`, `evidence`, and `audit_trail.events` use AuditIQ-native schemas.
These are operational/workflow constructs with no MISMO equivalent.
* *Pack configuration schema:* Audit Packs remain proprietary JSON (<<sect-audit-pack-spec>>).
Packs _reference_ MISMO field names in their rules, but the pack format itself is not a MISMO construct.
* *AI/ML metadata:* Confidence scores, provenance metadata, and model evaluation data are AuditIQ-specific.

*Implementation Approach — Flattened Subset:*

MISMO's full XML schema is deeply nested and verbose (thousands of data points).
AuditIQ adopts a *flattened subset* relevant to each audit domain:

* *Phase 1 (HMDA):* ~80 MISMO data points from the HMDA Reporting Dataset, flattened into a single-level JSON structure stored in `staging_datasets.payload`.
Field names match MISMO exactly (e.g., `ActionTakenType`, `LienStatusType`, `LoanPurposeType`).
* *Phase 2+ (additional regulations):* Each new Audit Pack defines its MISMO subset in the pack manifest under `canonical_schema.mismo_fields[]`.
The schema mapping engine validates that all referenced MISMO fields exist in the MISMO Reference Model dictionary.
* *Unmapped fields:* Fields that have no MISMO equivalent (custom institution fields, derived metrics) use a `custom:` namespace prefix (e.g., `custom:InternalRiskScore`) to clearly distinguish them from MISMO-standard fields.

*MISMO Reference Model Dictionary:*

AuditIQ maintains a lookup table of adopted MISMO fields:

[[sect-normalization-engine]]
=== Normalization Engine

*Responsibilities:*

* Standardize values before comparison (remove formatting, case, whitespace)
* Coerce types (string to decimal, date string to ISO 8601)
* Apply format rules defined in pack

*Normalization rules (pack-defined):*

[source,json]
----
{
  "field_normalizations": [
    {
      "field_id": "loan_amount",
      "type": "decimal",
      "rules": [
        "remove_currency_symbols",
        "remove_thousands_separators",
        "round_to_2_decimals"
      ]
    },
    {
      "field_id": "property_address",
      "type": "string",
      "rules": [
        "uppercase",
        "remove_extra_whitespace",
        "normalize_state_abbrev"
      ]
    },
    {
      "field_id": "loan_origination_date",
      "type": "date",
      "rules": ["parse_to_iso8601", "normalize_timezone_utc"]
    }
  ]
}
----

*Unicode normalization:* All string values are converted to Unicode NFC (Canonical Decomposition followed by Canonical Composition) before any other normalization rule applies.
This prevents silent mismatches when source documents contain decomposed characters (NFD) — for example, "José" composed vs. decomposed would fail equality checks without NFC normalization.

*Error handling — type coercion failures:*

[cols="25%,25%,25%,25%",options="header"]
|===
|Error Type |Example |Fallback Behavior |Finding Impact

|Date parse failure
|"02/31/2025" (invalid day)
|Set value to `null`; emit `normalization_failed` event; flag for manual review
|`conf_rule` reduced to 0.0; finding generated with `match_result = 'unable_to_determine'`

|Numeric parse failure
|"$N/A" in loan_amount field
|Set value to `null`; emit `normalization_failed` event
|`conf_rule` reduced to 0.0; finding generated with `match_result = 'unable_to_determine'`

|Currency symbol ambiguity
|"£250,000" when USD expected
|Preserve raw value; add `currency_mismatch` warning; do not convert
|Finding generated with `severity = 'medium'` and narrative noting currency mismatch

|Whitespace anomaly
|"123\u00A0Main\u2003St" (non-breaking space + em space)
|Collapse all Unicode whitespace categories (Zs) to single ASCII space; trim leading/trailing
|None — transparent normalization

|State abbreviation unknown
|"Calif." (non-standard)
|Attempt lookup in extended abbreviation table (150+ variants); if no match, preserve raw value and flag
|`conf_rule` unaffected if lookup succeeds; finding if target expects "CA" and normalization fails
|===

*Decimal precision rules by field type:*

[cols="30%,20%,50%",options="header"]
|===
|Field Category |Precision |Rationale

|Loan amounts, dollar values
|2 decimal places (round half-even / banker's rounding)
|HMDA LAR reporting standard; avoids ±$0.01 false positives

|Interest rates, APR
|3 decimal places
|Rate spread tolerance per FFIEC requires 3-decimal precision (e.g., 6.250% vs. 6.251%)

|LTV ratio, DTI ratio
|2 decimal places
|Industry convention; pack rules define tolerance (typically ±0.5%)

|Percentages (general)
|2 decimal places
|Standard reporting precision
|===

*Complex transformation examples:*

[cols="20%,40%,40%",options="header"]
|===
|Field |Normalization Rules |Example Input → Output

|`property_address`
|NFC → uppercase → collapse whitespace → normalize state → strip unit designator abbreviations → USPS standardization
|"123 main st, apt 4b, san francisco, calif. 94102" → "123 MAIN ST APT 4B SAN FRANCISCO CA 94102"

|`borrower_name`
|NFC → trim → collapse whitespace → remove honorifics (Mr/Mrs/Dr) → preserve case
|" Mr. José  García-López " → "José García-López"

|`loan_origination_date`
|Parse flexible formats (MM/DD/YYYY, YYYY-MM-DD, DD-Mon-YY, "January 15, 2025") → ISO 8601 UTC
|"01/15/25" → "2025-01-15T00:00:00Z" (with 2000-pivot for 2-digit years)

|`action_taken_type`
|Strip whitespace → map to HMDA enum code if narrative provided
|"Loan Originated" → "1" (per HMDA Action Taken codes)
|===

*Determinism:* Deterministic — same input + same rules = same output.
Unicode NFC normalization and banker's rounding are both deterministic operations.

[[sect-confidence-scoring]]
==== Confidence Scoring (Three Layers)

Every finding carries three independent confidence measures, each produced by a different upstream component:

*Layer 1: Mapping Confidence*

* Measures: How confident is the system that target field X corresponds to source field Y?
* Inputs: Field name similarity, data type compatibility, historical acceptance, value distribution
* Range: 0.0–1.0
* Formula (v1): `0.30 * name_similarity + 0.25 * synonym_match + 0.20 * type_compat + 0.15 * doc_section_context + 0.10 * historical_acceptance` (5-signal hybrid heuristic; see <<sect-schema-mapping-service>> for full specification and <<sect-two-layer-confidence, Two-Layer Confidence Design>> for calibrated scoring)
* Computed by: Schema Mapping Service during mapping phase
* Immutable: No, can be corrected via new mapping approval

*Layer 2: Extraction Confidence*

* Measures: How confident is the system that the extracted value from a source document is correct?
* Inputs: OCR engine quality, layout complexity, document page tier, model version
* Range: 0.0–1.0
* Determinant: AuditIQ Field Confidence (AFC) composite score — see <<sect-doc-processing, Document Processing>> for the six-signal formula
* Computed by: Document Processing Service
* Immutable: No (extraction confidence is metadata on extracted values, not a state change)

*Layer 3: Rule Application Confidence*

* Measures: How confident is the system that a validation rule correctly applies to this specific data context?
* Inputs: Precondition satisfaction, data quality, rule specificity, normalization success
* Range: 0.0–1.0
* Formula (v1): `base_confidence = pack_rule_definition.confidence   if precondition_not_satisfied:     rule_confidence *= 0.7 (reduced due to uncertain context)   if normalization_failed:     rule_confidence *= 0.5 (values may not be comparable)   return max(0.0, min(1.0, rule_confidence))`
* Computed by: Comparison & Rules Engine
* Immutable: No (metadata on finding, not a state change)

*Combined Confidence Formula:*

....
combined_confidence = min(
  mapping_confidence,
  extraction_confidence (if applicable, else 1.0),
  rule_confidence
)
....

NOTE: The `min()` formula is a conservative gate used at the *finding* level.
The Schema Mapping Service uses a separate <<sect-two-layer-confidence, Two-Layer Confidence Design>> with calibrated logistic regression for *mapping-specific* routing decisions.
Both feed into the finding's overall confidence score.

*Interpretation and routing thresholds:* See <<sect-glossary, Canonical Glossary — Routing Thresholds>> for the authoritative threshold bands (≥0.85 auto-confirm, 0.50–0.84 human review, <0.50 reject) and <<sect-glossary, UI Confidence Badge Colors>>.

==== Confidence Calibration & Monitoring

*Calibration rationale:* Confidence scores from Docling, schema matcher, and rules engine are learned/heuristic-based.
They must be calibrated against ground truth (human decisions) to remain accurate.

*Calibration approach (Phase 2):*

[arabic]
. *Collection:* After human review, compare AI-generated confidence scores to actual outcomes:
* Did AI mark it high-confidence? Did human agree?
* Did AI mark it low-confidence? Did human override?
. *Metrics:*
* *Calibration error:* % of high-confidence findings where human disagreed
* *Coverage:* % of findings where AI confidence threshold correctly predicted human review need
. *Adjustment:* If calibration error >5% for a given component (e.g., Docling), adjust thresholds or retrain model.

* Alert: If calibration error exceeds 5% threshold
* Trends: How confidence scores evolve as packs mature

[[sect-audit-trail-event-store]]
=== Audit Trail & Event Store (Immutable Log)

*Responsibilities:*

* Record every state change as an immutable, hash-chained event with RFC 8785 canonical JSON serialization
* Maintain global monotonic sequence IDs (per-tenant) for gap and reorder detection
* Produce cryptographic anchor points at configurable intervals for tamper resistance
* Support querying events by audit, finding, user, date range
* Enable audit replay for investigations and regulator exam response
* Support litigation defensibility via FRE Rule 902(13)/(14) self-authentication

[IMPORTANT]
====
*Critical design requirement (per Audit Trail Defensibility Report):* Hash chaining alone is necessary but NOT sufficient.
An attacker with database write access can rewrite the entire chain.
This design adds three mandatory layers: (1) RFC 8785 canonical JSON serialization for repeatable hash verification, (2) global monotonic sequence IDs for gap/reorder detection, (3) cryptographic anchor points that cannot be silently rewritten.
See <<sect-decision-d3, Decision D3>> for design tier rationale.
====

==== Event Schema

[source,typescript]
----
interface AuditEvent {
  event_id: string;                    // UUID v7 (time-ordered)
  tenant_id: string;                   // Tenant isolation boundary
  sequence_id: number;                 // Global monotonic per tenant (NOT per audit)
  timestamp: string;                   // Server-assigned ISO 8601; client timestamps rejected
  audit_id: string;
  entity_type: 'finding' | 'mapping' | 'evidence' | 'audit_run' | 'policy_document' | 'complaint' | 'user_action' | 'ai_action' | 'system';  // Extensible: new entity types added without schema migration
  entity_id: string;

  event_type: string;                  // e.g., 'FindingCreated', 'FindingTriaged', 'FindingClosed'
  actor_id: string;                    // userId or system principal
  actor_role: string;

  // Event payload — MUST be serializable via RFC 8785
  payload: object;

  // Tamper evidence (hash chain)
  previous_event_hash: string;         // SHA-256 of RFC 8785-canonicalized predecessor event
  event_hash: string;                  // SHA-256 of RFC 8785-canonicalized THIS event

  // Cryptographic anchor (populated on anchor events only)
  is_anchor: boolean;                  // True every N events (configurable; default N=100)
  anchor_hash?: string;                // If is_anchor: hash written to WORM storage

  // Metadata
  source_system: string;               // e.g., 'comparison_engine', 'audit_ui', 'remediation_workflow'
  api_version: string;
  schema_version: string;              // Event schema version for forward compatibility
}
----

==== RFC 8785 Canonical JSON Serialization

*MANDATORY:* All event payloads MUST be serialized using RFC 8785 (JSON Canonicalization Scheme) before hashing.
This ensures: - Deterministic key ordering (lexicographic) - Consistent number formatting (no trailing zeros, no positive sign) - Consistent Unicode normalization - No optional whitespace

....
canonical_json = RFC8785.canonicalize(event_without_hash_fields)
event_hash = SHA-256(canonical_json)
....

*Implementation requirement:* Use a well-tested RFC 8785 library (e.g., `canonicalize` npm package or equivalent).
Do NOT hand-roll canonical JSON — edge cases (Unicode escaping, number precision, key ordering with special characters) are subtle and produce silent hash mismatches.

*Write-time validation:* Every event MUST pass round-trip verification before persistence:

....
assert(RFC8785.canonicalize(JSON.parse(RFC8785.canonicalize(event))) === RFC8785.canonicalize(event))
....

==== Hash Chain Computation

....
event_hash = SHA-256(
  RFC8785.canonicalize({
    event_id, tenant_id, sequence_id, timestamp, audit_id,
    entity_type, entity_id, event_type, actor_id, actor_role,
    payload, previous_event_hash, source_system, api_version, schema_version
  })
)
....

*Chain rules:*

* First event in tenant chain: `previous_event_hash = SHA-256("GENESIS:" + tenant_id)`
* Each subsequent event: `previous_event_hash = event_hash` of the immediately preceding event (by sequence_id)
* Gap in sequence_id → chain is broken → alert + investigation required

==== Cryptographic Anchor Points

Every N events (default N=100, configurable per tenant), the system writes an *anchor event*: - `is_anchor = true` - `anchor_hash` = SHA-256 of the event_hash at this point - Anchor hash is written to *WORM storage* (e.g., Azure Immutable Blob Storage) — cannot be overwritten or deleted during retention period - Anchor verification: compare stored anchor hash against recomputed hash from event chain; mismatch = tampering detected

*Phase 2 upgrade:* Replace WORM anchors with KMS-signed checkpoint events (Azure Key Vault) for non-repudiation and litigation support under FRE 902(13)/(14).

==== Operational Integrity Controls (NIST AU-9 / FFIEC)

Per NIST SP 800-53 Rev 5 AU-9 and FFIEC IT Examination Handbook:

* *No admin delete:* Database roles for audit_trail schema must NOT include DELETE permission; enforced at PostgreSQL role level
* *Separate storage:* Audit trail stored in dedicated schema/database; not co-located with application data that admins routinely modify
* *Read-only access for audit:* Regulator/auditor access via read-only database replica or export; no write access to production audit trail
* *Retention:* Minimum 7 years (configurable per tenant; aligned with NIST AU-11); events never purged during retention window
* *Backup verification:* Weekly automated restore-and-verify of audit trail backup; hash chain re-verified on restored data

==== Tamper Detection & Verification

*Chain verification tool (MVP — Phase 1a):*

* CLI tool that reads entire tenant event chain and verifies: 1.
Every `event_hash` matches recomputed SHA-256 of RFC 8785-canonicalized event 2.
Every `previous_event_hash` matches `event_hash` of preceding event (by sequence_id) 3.
No gaps in sequence_id 4.
All anchor hashes match WORM-stored values
* Output: “OK” or list of corrupted/missing events with sequence_id range

*Periodic integrity check (automated):*

* Daily: Verify last 24 hours of events (chain continuity + hash correctness)
* Weekly: Full chain verification for each tenant
* On anchor write: Verify chain from previous anchor to current anchor before writing to WORM

*Snapshot integrity (see <<sect-immutability>>):*

* When snapshots are materialized, compute snapshot hash and store in event stream
* Periodic re-materialization from events; compare against stored snapshot hash; mismatch = drift alert

[[sect-exam-pack-lifecycle-manager]]
=== Exam Pack Lifecycle Manager

*Responsibilities:*

* Maintain the exam pack as a continuously updated dossier throughout the audit lifecycle
* Create new bundle versions on material events (finding changes, approvals, evidence updates)
* Compute coverage and sampling statistics from audit data
* Collect and structure approval manifests from all three lines per the IIA Three Lines Model
* Track exceptions and overrides with justification
* Generate pipeline provenance from model registry and service versions
* Produce the exportable ZIP bundle with manifest and hash verification
* Serve bundle version history and version-to-version diffs via API

*Event-driven update triggers:*

The Exam Pack Lifecycle Manager subscribes to audit trail events via the Workflow Engine.
When a material event occurs, it creates a new bundle version:

[cols="40%,30%,30%",options="header"]
|===
|Event Type |Version Bump |Action

|`audit_started`
|Major (v1.0)
|Create initial bundle with audit metadata and pack configuration

|`pre_scan_complete`
|Major (v2.0)
|Populate findings register, coverage statistics, confidence distributions

|`finding_created` / `finding_closed` / `finding_reopened`
|Major
|Update findings register, recalculate coverage stats, append to change log

|`finding_triaged` / `finding_escalated`
|Minor
|Update finding status in register, append to change log

|`corrective_action_completed`
|Minor
|Update corrective action log, append to change log

|`evidence_added` / `evidence_superseded`
|Minor
|Update source document index, append to change log

|`approval_recorded`
|Major
|Add approval to manifest, update defense lines certification, check all-lines-approved

|`exception_recorded`
|Minor
|Append to exceptions log

|`bundle_exported`
|No bump
|Record export metadata (who, when, hash)
|===

*Error Handling:*

[cols="40%,60%",options="header"]
|===
|Error |Handling

|Bundle export requested before all AI narratives approved
|Return 422 with list of unapproved narratives; block export

|Bundle export requested with untriaged Critical findings
|Return 422 with list of untriaged Critical findings; block export

|Approval recorded by wrong defense line (e.g., QC manager approving audit-layer findings)
|Return 403; RBAC enforcement prevents cross-line approvals

|Version conflict (concurrent updates)
|Optimistic locking via bundle content hash; retry with latest version

|Bundle generation fails mid-export

*Determinism:* Bundle content is deterministic for a given audit state — same findings, evidence, and approvals always produce the same bundle content (hash-verified).

*Storage:* Generated bundles stored in Azure Blob Storage under `tenant-{tenant_id}/bundles/audit_{audit_id}/v{version}/`.
Retention follows the same 7-year lifecycle policy as source documents.

[[sect-ai-governance-approval-workflow]]
=== AI Governance & Approval Workflow

*Core principle:* Every AI output is advisory, non-finalized, and requires explicit human approval before inclusion in official records.

*AI output categories:*

[arabic]
. *Extracted values (OCR/Docling):* Labeled “AI-extracted (advisory)” in UI. Recommendation to use: > High confidence (≥0.85) can be pre-confirmed; lower values require manual verification.
. *Mapping suggestions:* System suggests field pairings. Reviewer approves, rejects, or overrides. Approved mappings versioned and reusable.
. *AI-generated narratives:* Audit summary, root cause analysis, remediation recommendations. Generated on-demand with explicit approval workflow:
* Draft labeled “[AI-DRAFT — REQUIRES APPROVAL]”
* Manager reviews and approves or rejects
* Approved version embedded in report
* Rejected version discarded; Manager can request new draft
. *Rule generation (TBD Phase 2):* System suggests new comparison rules from discovered patterns. Requires explicit pack author review before deployment.

[[sect-api-gateway]]
=== API Gateway

*Technology:* Azure API Management (APIM) — managed service, Azure-native.
Selected for integration with Azure AD, built-in rate limiting, and operational simplicity.

*Responsibilities:*

* Authenticate and authorize every inbound request (OAuth 2.0 / OIDC via Azure AD)
* Enforce rate limiting per tenant and per user
* Route requests to backend microservices
* Terminate TLS and enforce HTTPS-only
* Apply request/response transformation policies (header injection, CORS)
* Emit request telemetry to Azure Monitor / Application Insights

*Authentication flow:*

[mermaid]
----
sequenceDiagram
  participant Client as Web UI / API Consumer
  participant APIM as Azure API Management
  participant AAD as Azure Active Directory
  participant Backend as Backend Services

  Client->>AAD: Authenticate (OAuth 2.0 / OIDC)
  AAD-->>Client: Access token (JWT) + refresh token
  Client->>APIM: API request + Bearer token
  APIM->>APIM: Validate JWT (signature, expiry, audience, tenant claim)
  APIM->>APIM: Check rate limit (tenant_id + user_id)
  alt Rate limit exceeded
    APIM-->>Client: 429 Too Many Requests (Retry-After header)
  else Valid request
    APIM->>Backend: Forward request + X-Tenant-ID + X-User-ID headers
    Backend-->>APIM: Response
    APIM-->>Client: Response
  end
----

*Rate limiting policy:*

[cols="25%,25%,25%,25%",options="header"]
|===
|Scope |Limit |Window |Burst

|Per tenant
|10,000 requests
|1 minute
|500 requests/second

|Per user
|1,000 requests
|1 minute
|100 requests/second

|Document upload
|50 requests
|1 minute
|10 requests/second

|AI narrative generation
|20 requests
|1 minute
|5 requests/second
|===

*Burst semantics:* Burst limits allow short spikes above the sustained rate.
The burst allowance is a token bucket that refills at the sustained rate.
For example, the per-tenant burst of 500 requests/second allows a spike of 500 requests in a single second, but the bucket then takes ~3 seconds to refill to full capacity at the 10,000/minute sustained rate (~167/second).
A tenant cannot continuously consume at the burst rate.

*Quota window type:* Sliding window (not fixed-minute boundaries).
This prevents the "double-burst" problem where a client sends 10,000 requests at :59 and another 10,000 at :01 of the next minute.

*Multi-tenant fairness:* Rate limits are enforced per-tenant in APIM using separate rate-limit counters per `tenant_id` claim.
One tenant hitting its quota has zero impact on other tenants — no shared counters, no shared queues.
Backend services use separate Kubernetes resource quotas per tenant tier (Standard, Professional, Enterprise) to prevent noisy-neighbor effects at the compute layer.

*Rate-limit response headers:* All responses include standard rate-limit headers per the IETF RateLimit Header Fields draft:

....
RateLimit-Limit: 10000
RateLimit-Remaining: 7842
RateLimit-Reset: 34
Retry-After: 0
....

`RateLimit-Reset` is the number of seconds until the quota window resets.
`Retry-After` is populated only on 429 responses and indicates seconds to wait before retrying.

*API versioning:* URL-path versioning (`/api/v1/...`).
Breaking changes require a new version; old versions supported for 6 months minimum.

*Input contracts:*

[source,typescript]
----
// Headers injected by APIM into backend requests
interface GatewayHeaders {
  'X-Tenant-ID': string;       // Extracted from JWT tenant claim
  'X-User-ID': string;         // Extracted from JWT sub claim
  'X-User-Roles': string[];    // Extracted from JWT roles claim
  'X-Request-ID': string;      // Correlation ID (UUID, generated by APIM)
  'X-Forwarded-For': string;   // Client IP (for audit logging)
}
----

*CORS policy:*

* Allowed origins: configured per environment (production: app domain only)
* Allowed methods: GET, POST, PUT, PATCH, DELETE, OPTIONS
* Allowed headers: Authorization, Content-Type, X-Request-ID
* Max age: 3600 seconds

*Health check endpoint:* `GET /health` — bypasses authentication, returns `200 OK` with service status.

*Error responses:*

[cols="15%,40%,45%",options="header"]
|===
|Code |Condition |Response Body

|401
|Missing or invalid JWT
|`{"error": "unauthorized", "message": "Invalid or expired access token"}`

|403

|`{"error": "forbidden", "message": "Insufficient permissions for this resource"}`

|429

*Responsibilities:*

* Manage the complete audit lifecycle from creation to finalization
* Load and validate audit pack versions
* Enforce state transition rules (authoritative state machine)
* Emit audit events for every state change
* Coordinate between Ingestion, Document Processing, Comparison, and Reporting services
* Enforce concurrent audit limits per tenant

*Authoritative state machine:*

The state machine defined in <<sect-key-concepts, Diagram 1>> is the single source of truth.
The Orchestration Service enforces all transitions.

*Pack loading and versioning:*

* On `createAudit`, the Orchestration Service loads the specified pack version from the Audit Pack Manager.
* Pack versions are immutable once published.
If a pack needs changes, a new version is created.
* An in-progress audit always uses the pack version it was created with — no hot-swapping mid-audit.
* Pack validation runs on load: schema checks, rule syntax validation, evidence requirement checks.
If validation fails, audit creation is rejected.

*Retry and compensation for failed transitions:*

* *Transient failures* (network timeout, temporary service unavailability): Retry with exponential backoff (1s, 2s, 4s) up to 3 attempts.
State remains unchanged until transition succeeds or is abandoned.
* *Permanent failures* (validation error, business rule violation): No retry.
Return error to caller with specific failure reason.
State remains unchanged.
* *Compensation:* If a multi-step transition partially completes (e.g., pre-scan starts but comparison engine crashes), the Orchestration Service emits a `TransitionFailed` event with partial results and rolls the state back to the prior valid state.
The operator can retry or manually intervene.

*Partial batch failure handling:*

When a pre-scan processes N records and fails at record K:

[arabic]
. Records 1 through K-1: findings are committed and visible.
The audit state transitions to `REVIEW_READY` for the completed portion.
. Record K: marked as `processing_error` with error details.
A finding is created with `match_result = 'unable_to_determine'` and the error attached as metadata.
. Records K+1 through N: remain unprocessed.
The Orchestration Service emits a `PartialBatchComplete` event with `{processed: K-1, failed: 1, remaining: N-K}`.
. Manager sees a banner: "Pre-scan partially complete (K-1 of N records).
1 record failed processing.
{N-K} records not yet processed.
[Retry Remaining] [Review Completed]."
. *Retry Remaining* restarts processing from record K (not from the beginning).
Record IDs are checkpointed in the audit state to enable resume-from-failure.

*Rollback semantics for multi-step transitions:*

[cols="30%,35%,35%",options="header"]
|===
|Transition |Steps |Compensation on Failure

|`READY → PRE_SCAN_RUNNING → REVIEW_READY`
|1. Load pack rules 2. Execute comparison 3. Generate findings 4. Compute coverage stats
|If step 3 fails: findings from steps 1-2 are committed (partial results); state set to `REVIEW_READY` with `partial_scan = true` flag.
If step 1 fails: no side effects; state remains `READY`.

|`UNDER_REVIEW → REPORT_GENERATION → PENDING_APPROVAL`
|1. Aggregate findings 2. Generate AI narratives 3. Compile report PDF 4. Assemble bundle version
|If step 3 fails: findings and narratives are preserved; report marked `generation_failed`; state remains `UNDER_REVIEW`.
Manager can retry report generation without re-running review.

|`PENDING_APPROVAL → FINALIZED`
|1. Validate all approvals 2. Lock findings (immutable) 3. Generate final bundle version 4. Archive to cold storage
|If step 3 fails: approvals are preserved but findings remain mutable; state remains `PENDING_APPROVAL`.
Manager retries finalization.
|===

*Dead-letter handling:*

Failed operations that exhaust all retries are written to a dead-letter topic in Azure Service Bus.
Each dead-letter message includes: `operation_type`, `audit_id`, `record_id` (if applicable), `error_details`, `retry_count`, `original_timestamp`.
A background process reviews dead-letter messages hourly and either retries (for transient errors that have since resolved) or alerts the operator (for persistent failures).
Dead-letter messages older than 7 days are escalated to P1 severity.

*Embedding model:* Domain-fine-tuned sentence transformer (SBERT) shared across the Regulatory Citation Engine and the Policy Alignment Pipeline (see <<sect-embedding-domain-finetuning>>).
Fine-tuned on financial regulatory language for negation sensitivity, numeric threshold context, and deontic language precision.
Self-hosted on Azure Container Instances for cost predictability and data sovereignty.
No regulation text leaves the tenant's Azure environment.

*Vector database:* Qdrant (self-hosted on Azure Kubernetes Service).
Selected for: native multi-tenancy support, payload filtering, and no vendor lock-in.

*Chunking strategy:*

* Regulations are chunked at the paragraph level (typically 100–500 tokens).
* Each chunk retains metadata: regulation title, CFR part, section number, subsection, effective date, amendment history.
* Overlapping windows (50-token overlap) to preserve cross-paragraph context.

*Chunk size justification:*

[cols="20%,40%,40%",options="header"]
|===
|Chunk Size |Advantage |Disadvantage

|<100 tokens
|Higher precision per chunk; fine-grained citation
|Loses cross-sentence context; regulatory paragraphs often require full paragraph for meaning (e.g., "subject to the exceptions in paragraph (b)" is meaningless without paragraph (b))

|100–300 tokens (selected)
|Balances precision and context; aligns with typical CFR paragraph length; produces embeddings with high semantic density
|Some long paragraphs (e.g., commentary sections) may be split mid-thought

|300–500 tokens
|Preserves full paragraph context including commentary
|Embedding quality degrades — diverse topics within a single chunk reduce semantic specificity; recall drops for narrow queries

|>500 tokens
|Full section context preserved
|Embedding vector represents a blend of multiple concepts; precision drops significantly on targeted queries
|===

The 50-token overlap was empirically chosen during Phase 0 prototyping: overlaps <30 tokens lost critical cross-paragraph references (e.g., "as defined in the preceding paragraph"), while overlaps >80 tokens created near-duplicate embeddings that diluted search results.

*Query flow:*

[arabic]
. Finding context (rule violation narrative + field name + severity) is embedded using the same domain-fine-tuned sentence transformer.
. Semantic search retrieves top-20 candidate paragraphs from Qdrant (over-retrieve to allow re-ranker to filter).
. Re-ranker (cross-encoder) re-scores all 20 candidates and returns top-k (default: 5) by relevance.
. Pack-defined primary citations are always included and ranked first regardless of re-ranker score.
. AI-suggested secondary citations are labeled `[AI-SUGGESTED]` and require reviewer confirmation before inclusion in reports.

==== Re-Ranker Specification

*Model:* Cross-encoder fine-tuned on regulatory question-paragraph pairs.
Base model: `ms-marco-MiniLM-L-12-v2` (lightweight cross-encoder with strong out-of-box performance on passage reranking benchmarks).

*Fine-tuning (Phase 1b):*

* *Training data:* 2,000+ (query, paragraph, relevance) triples constructed from:
** HMDA examination procedure questions paired with the CFR paragraphs they reference (extracted from FFIEC examiner guides — these guides explicitly cite specific CFR sections).
** Synthetic negatives: for each positive pair, 5 hard negatives sampled from the same CFR part (same regulation, wrong section).
* *Loss function:* Cross-entropy on binary relevance (relevant / not relevant).
* *Evaluation:* 80/20 train/test split, stratified by regulation.

*Phase 0 (pre-fine-tuning):* Use `ms-marco-MiniLM-L-12-v2` out of the box.
Expected NDCG@5 ~0.65 on regulatory text (acceptable for MVP; pack-defined primary citations compensate for search quality gaps).

*Phase 1b target (post-fine-tuning):* NDCG@5 ≥ 0.80 on held-out regulatory question-paragraph test set.

==== Search Quality Metrics & Monitoring

[cols="25%,25%,25%,25%",options="header"]
|===
|Metric |Definition |Target |Monitoring Cadence

|NDCG@5
|Normalized Discounted Cumulative Gain at rank 5, measured on a held-out test set of 200 regulatory queries
|≥ 0.80 (Phase 1b); ≥ 0.65 (Phase 0)
|Re-evaluated on every corpus update and every model fine-tuning iteration

|MRR (Mean Reciprocal Rank)
|Average of 1/rank of the first relevant result across all test queries
|≥ 0.75
|Re-evaluated on corpus update

|Precision@5
|Fraction of top-5 results that are relevant
|≥ 0.60
|Re-evaluated on corpus update

|Citation acceptance rate
|% of AI-suggested secondary citations that reviewers confirm as relevant
|≥ 70%
|Tracked per audit; aggregated monthly

|Citation rejection rate
|% of AI-suggested citations that reviewers explicitly reject
|< 15%
|Tracked per audit; if > 15% for 3 consecutive audits, trigger re-ranker evaluation

|Corpus drift indicator
|Semantic similarity between new corpus embeddings and previous version (measured as cosine similarity of mean pool vectors per regulation)
|> 0.90 (no significant drift)
|Computed on every quarterly corpus update; if < 0.85, flag for human review of changed regulations
|===

*Citation verification:* To address the confabulation risk identified in <<sect-ai-governance>>, every AI-suggested citation is verified before display:

[arabic]
. The cited CFR reference (e.g., "12 CFR § 1003.4(a)(1)") is validated against the metadata store — the paragraph must exist in the corpus at the cited location.
. If the cited paragraph text does not match the stored text (hash comparison), the citation is flagged as `[CITATION MISMATCH]` and suppressed from the report.
. Citations with `relevance_score` < 0.60 are suppressed entirely (below `min_relevance` threshold).

*Corpus update cadence:*

* CFR text updated quarterly (aligned with Federal Register publication cycle).
* Agency guidance updated within 30 days of publication.
* Embeddings re-computed on corpus update; old vectors retained for audit reproducibility (versioned by `corpus_version`).
* On corpus update: re-ranker evaluation suite runs automatically.
If NDCG@5 drops by >0.05, alert ML team and hold the update until investigated.

=== Document Store (Azure Blob Storage)

*Responsibilities:*

* Store all source documents (PDFs, images), evidence files, generated reports, and audit export bundles
* Enforce encryption at rest and in transit
* Implement lifecycle policies (hot → cool → archive)
* Support legal hold and immutability for regulatory compliance
* Provide tenant-isolated access via Shared Access Signatures (SAS)

*Container structure:*

[cols="25%,35%,40%",options="header"]
|===
|Container |Contents |Access Pattern

|`{tenant_id}-documents`
|Source documents (PDF, TIFF, JPEG, PNG)
|Write-once on upload; read during review and extraction

|`{tenant_id}-evidence`
|Evidence files linked to findings
|Write-once; read during review, reporting, and exam export

|`{tenant_id}-reports`
|Generated reports (PDF, CSV, XLSX, JSON)
|Write on generation; read on download and exam export

|`{tenant_id}-exports`
|Exam-ready bundles (ZIP)
|Write on export; read on download; archive after 90 days

|`{tenant_id}-quarantine`
|Files flagged by malware scan
|Write-once; read by security team only; deleted after investigation
|===

*Encryption:*

* *At rest:* Azure Storage Service Encryption (SSE) with Customer-Managed Keys (CMK) stored in Azure Key Vault.
* *In transit:* TLS 1.2+ enforced; HTTP access disabled at the storage account level.
* *Key rotation:* Automatic annual rotation; manual rotation on-demand via Key Vault policy.

*Retention and lifecycle:*

[cols="20%,20%,20%,20%,20%",options="header"]
|===

|Hot

* Documents in finalized audits are moved to Cool tier 12 months after finalization.
* Documents are moved to Archive tier 36 months after finalization.
* Documents are deleted (with compliance confirmation) after 7 years.
* *Legal hold:* Azure Immutable Blob Storage with legal hold policy.
When a legal hold is placed on a tenant, all documents in that tenant's containers are protected from modification and deletion regardless of lifecycle policy.

*Malware scanning:*

* All uploaded files are scanned by Microsoft Defender for Storage before being moved from `quarantine` to the target container.
* Scanning is asynchronous; files are not available for processing until scan completes (typically <30 seconds).
* Infected files remain in quarantine; an alert is sent to the IT Admin role.

*Access control:*

* No public access — all access via SAS tokens scoped to tenant and container.
* SAS tokens are short-lived (1 hour) and generated by the API Gateway on authenticated requests.
* Backend services use Managed Identity (no stored credentials).

[[sect-multi-tenancy-isolation]]
=== Multi-Tenancy & Tenant Isolation

AuditIQ is a multi-tenant SaaS platform where every layer enforces tenant isolation independently.
No single layer is trusted to be the sole isolation boundary — defense in depth ensures that a failure in one layer cannot leak data across tenants.

==== Isolation by Layer

[cols="20%,35%,45%",options="header"]
|===
|Layer |Isolation Mechanism |Enforcement Detail

|*API Gateway*
|JWT `tenant_id` claim extraction; per-tenant rate-limit counters
|APIM policy validates `tenant_id` claim exists in JWT.
If missing, request rejected with 401.
`X-Tenant-ID` header injected for all downstream services.
Rate-limit counters are keyed on `tenant_id` — no shared buckets.

|*Application services*
|`X-Tenant-ID` header required on every request; services scope all queries by `tenant_id`
|Every database query includes `WHERE tenant_id = :tenant_id`.
Services reject requests without `X-Tenant-ID` header with 400 Bad Request.
Integration tests verify that cross-tenant queries return zero results.

|*PostgreSQL*
|Row-Level Security (RLS) policies on all tables
|RLS policies enforce `tenant_id = current_setting('app.tenant_id')` on SELECT, INSERT, UPDATE.
Application sets `SET app.tenant_id = :tenant_id` at the start of every transaction.
Even if application code omits a `WHERE tenant_id` clause, RLS prevents cross-tenant access.
DELETE is revoked on audit trail tables (immutable log).

|*Elasticsearch*
|Per-tenant indices (`{index}-{tenant_id}`)
|Search service constructs index name from `X-Tenant-ID` header.
A request from tenant A can never query tenant B's index.
The shared `regulations` index is read-only and contains no tenant-specific data.

|*Qdrant (Vector DB)*
|Payload-level filtering by `tenant_id` for institution-specific policy embeddings; shared corpus for regulations
|Regulation embeddings are shared (public regulatory text — no tenant data).
Institution policy embeddings use `tenant_id` as a required payload filter.
Even if all tenant embeddings coexist in one collection, Qdrant's payload filter ensures only the requesting tenant's vectors are returned.

|*Azure Blob Storage*
|Per-tenant containers (`{tenant_id}-documents`, `{tenant_id}-evidence`, etc.)
|SAS tokens are scoped to a specific tenant's container.
Backend services use Managed Identity with container-level RBAC.
No cross-tenant container access is possible without a tenant-specific SAS token or Managed Identity scope.

|*Azure Service Bus*
|Per-tenant topic subscriptions for event routing
|Dead-letter topics and event routing use `tenant_id` as a partition key.
Messages are never routed to the wrong tenant's subscription.

[[sect-policy-documents-table]]
=== Policy Document Library (Data Model)

*Strategic rationale:* The Institution Policy Profile (<<sect-institution-policy-profile>>) currently embeds a `policy_document_registry` array inside the profile JSON.
This table promotes policy documents to a first-class entity, decoupled from any specific audit or policy profile.
This decoupling enables three future capabilities at near-zero marginal cost:

. *Standalone Policy Library:* Institutions can manage their policy document inventory, review cycles, and version chains without creating an audit or a policy profile — delivering immediate value during onboarding, before the first audit runs.
. *Document Control Workflows:* The `status` lifecycle and `next_review_date` field enable a policy review cycle engine using the same state-machine and event-publishing infrastructure as audit runs and findings.
. *Cross-Audit Policy Reuse:* Multiple audit packs and policy profiles can reference the same policy document without duplication, maintaining a single source of truth for policy version chains.

[source,sql]
----
CREATE TABLE compliance.policy_documents (
  document_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id UUID NOT NULL,
  institution_id UUID NOT NULL,

  -- Document identity
  title VARCHAR(512) NOT NULL,                 -- e.g., 'ABC Bank HMDA Data Integrity Policy v3.2'
  document_type VARCHAR(50) NOT NULL,          -- 'board_policy', 'compliance_policy', 'operating_procedure', 'job_aid', 'board_resolution', 'risk_appetite_statement'
  regulations_addressed TEXT[] NOT NULL,        -- e.g., '{12 CFR 1003, Reg C}'

  -- Version chain
  version VARCHAR(50),
  effective_date DATE NOT NULL,
  supersedes_document_id UUID,                 -- Previous version in the chain
  superseded_by_document_id UUID,              -- Next version (populated when superseded)

  -- Ownership & review cycle
  owner VARCHAR(255),                          -- Department or individual responsible
  department VARCHAR(255),
  review_cycle_months INT DEFAULT 12,          -- Board-mandated review cadence
  next_review_date DATE,                       -- Computed: effective_date + review_cycle_months; triggers review workflow when approaching
  last_reviewed_date DATE,
  last_reviewed_by UUID,

  -- Lifecycle
  status VARCHAR(50) DEFAULT 'draft',          -- 'draft', 'under_review', 'approved', 'published', 'expired', 'retired'

  -- Storage & extraction
  storage_reference VARCHAR(512),              -- URI to document in Azure Blob / institution's document store
  sha256_hash VARCHAR(64),                     -- Integrity verification
  ai_extraction_confidence DECIMAL(3,2),       -- Overall confidence from AI extraction pipeline

  -- Audit trail link
  audit_event_id UUID,                         -- Link to event store for full lifecycle history

  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

  FOREIGN KEY (tenant_id) REFERENCES tenants(tenant_id),
  FOREIGN KEY (supersedes_document_id) REFERENCES compliance.policy_documents(document_id),
  FOREIGN KEY (superseded_by_document_id) REFERENCES compliance.policy_documents(document_id)
);
CREATE INDEX idx_policy_docs_tenant ON compliance.policy_documents(tenant_id, institution_id);
CREATE INDEX idx_policy_docs_status ON compliance.policy_documents(status);
CREATE INDEX idx_policy_docs_review ON compliance.policy_documents(next_review_date) WHERE status = 'published';
CREATE INDEX idx_policy_docs_regulation ON compliance.policy_documents USING GIN (regulations_addressed);
----

*Relationship to Institution Policy Profile:* The Policy Profile's `policy_document_registry` array references `document_id` values from this table.
This table is the source of truth for document metadata; the profile captures the point-in-time snapshot used for a specific audit overlay.

*Review cycle engine (Phase 2+):* When `next_review_date` approaches (configurable alert window, default 30 days), the Workflow Engine creates a review workflow event.
The review follows the same dual-approval pattern as Policy Profile review (<<sect-ai-policy-pipeline>>, Phase 5).
Every review event is recorded in `audit_trail.events` with `entity_type = 'policy_document'`.

* MAJOR schema changes require audit run completion or explicit migration

*Approach:* Stream findings, status transitions, and metrics to a separate analytics database (Aurora or Redshift) via Change Data Capture or nightly batch export.

[[sect-pack-rule-engine]]
==== Rule Engine Specification

The rule engine interprets comparison rules defined in the pack manifest and produces findings.
This section specifies the full semantics of each comparison type, including complex scenarios required for real-world regulatory audits.

===== Comparison Types

====== Basic Types

*exact_match:* Target value must exactly equal source value after normalization.
Used for enum fields (Action Taken codes), identifiers (loan numbers), and boolean flags.

*tolerance_absolute:* Target value must be within plus-or-minus N of source value.
Tolerance specified in the rule's `tolerance` object.
Used for currency amounts (e.g., loan amount within plus-or-minus $50 of CD amount).

*tolerance_percentage:* Target value must be within plus-or-minus N% of source value.
Used for rates and ratios (e.g., interest rate within plus-or-minus 0.125% of CD rate).

*lookup:* Target value must be present in a reference lookup table defined in the pack.
Used for enum validation (e.g., Action Taken code must be one of 1-8 per HMDA spec).

*cross_field:* Target field value is validated against a computed relationship between two or more other fields.
Used for derived values (e.g., LTV ratio = loan_amount / appraised_value).

===== Advanced Types

[[sect-multi-doc-corroboration]]
====== Multi-Document Corroboration

A single target field is validated by cross-referencing values from two or more source documents.
This is the core audit pattern for HMDA compliance -- the same field (e.g., loan amount) appears on the Closing Disclosure, the Loan Estimate, the URLA, and the LOS record, and all must agree.

*Rule structure:*

[source,json]
----
{
  "rule_id": "hmda-loan-amount-corroboration",
  "rule_name": "Loan Amount Multi-Document Corroboration",
  "target_field": "loan_amount",
  "comparison_type": "multi_doc_corroboration",
  "source_doc_types": ["closing-disclosure", "loan-estimate", "urla-1003"],
  "tolerance": { "value": 0.50, "unit": "absolute" },
  "corroboration_policy": {
    "minimum_sources": 2,
    "conflict_resolution": "flag_for_review",
    "agreement_threshold": 0.95
  },
  "severity": "high",
  "regulation_citation": "12 CFR 1003.4(a)(7)"
}
----

*Execution semantics:*

. Extract the target field value from each specified source document type.
. For each pair of extracted values, apply the specified tolerance to determine agreement.
. If fewer than `minimum_sources` documents are available, flag as `unable_to_determine` with reduced rule confidence (x0.7).
. If all available sources agree within tolerance, produce a `match` finding with confidence boosted by the number of corroborating sources.
. If sources disagree, apply `conflict_resolution` policy: `flag_for_review` routes to human reviewer; `majority_wins` accepts the majority value; `highest_confidence_wins` accepts the value from the source with the highest extraction confidence (AFC).

*Confidence aggregation for multi-doc rules:*

....
corroboration_confidence = min(afc_per_source) * (1 + 0.05 * (num_agreeing_sources - 1))
rule_confidence = min(base_confidence, corroboration_confidence)
....

The more independent sources that agree, the higher the corroboration signal -- capped at 1.0.

[[sect-conditional-rule-chains]]
====== Conditional Rule Chains

Rules can depend on the outcome of other rules.
This enables branching audit logic: if the loan is FHA, apply additional FHA-specific validation rules that do not apply to conventional loans.

*Rule structure:*

[source,json]
----
{
  "rule_id": "fha-pmi-validation",
  "rule_name": "FHA PMI/MIP Validation",
  "target_field": "mortgage_insurance_premium",
  "comparison_type": "expression",
  "depends_on": ["hmda-loan-type-check"],
  "expression": "results['hmda-loan-type-check'].value == 'FHA' && target.mortgage_insurance_premium > 0",
  "severity": "high",
  "regulation_citation": "24 CFR 203.259a"
}
----

*Execution semantics:*

. Rules are topologically sorted by `depends_on` references before execution.
Circular dependencies are rejected at pack validation time.
. A rule with `depends_on` executes only if all prerequisite rules have completed.
. The `expression` field can reference `results['rule_id']` to access the outcome of prerequisite rules.
. If a prerequisite rule produced `unable_to_determine`, the dependent rule's confidence is reduced (x0.5) because its precondition is uncertain.

*Maximum chain depth:* 5 levels.
Packs with deeper chains are rejected at validation time.
This prevents runaway complexity while supporting real-world audit logic.

[[sect-stage-2-rules]]
====== Stage 2 Substantive Testing Rules

Stage 2 rules evaluate whether reported data reflects the underlying transaction correctly -- beyond field-level comparison.
These rules require professional judgment and are typically executed by Second Line (Compliance) reviewers.
Stage 2 rules are tagged with `testing_stage: "stage_2"` and `defense_line_scope: "second_line_compliance"`.

*Rule structure -- Action Taken Code Validation:*

[source,json]
----
{
  "rule_id": "hmda-action-taken-substantive",
  "rule_name": "Action Taken Code Substantive Validation",
  "target_field": "action_taken",
  "comparison_type": "expression",
  "expression": "target.action_taken == '1' && source.application_status != 'originated'",
  "testing_stage": "stage_2",
  "defense_line_scope": "second_line_compliance",
  "severity": "critical",
  "base_confidence": 0.85,
  "regulation_citation": "12 CFR 1003.4(a)(8)",
  "key_data_field": true
}
----

*Execution semantics:*

. Stage 2 rules execute after Stage 1 comparison is complete for the record.
. The rule evaluates whether the action-taken code is consistent with the application's full disposition history -- not just whether the code matches a source document.
. If the action-taken code is "1" (originated) but underwriting notes or application status indicate the application was withdrawn or denied, the rule flags a substantive finding.
. Stage 2 findings always require human review regardless of confidence score, because they involve professional judgment.

[[sect-temporal-rules]]
====== Temporal and Date Logic

Regulatory compliance frequently involves date calculations, deadlines, and timing requirements.
The rule engine supports temporal comparison types for these scenarios.

*Rule structure:*

[source,json]
----
{
  "rule_id": "trid-3-day-rule",
  "rule_name": "TRID Closing Disclosure 3-Business-Day Rule",
  "target_field": "closing_date",
  "comparison_type": "temporal",
  "source_field": "cd_received_date",
  "source_doc_types": ["closing-disclosure"],
  "temporal_config": {
    "operator": "gte",
    "offset_value": 3,
    "offset_unit": "business_days",
    "calendar": "us_federal",
    "description": "Closing date must be at least 3 business days after CD receipt"
  },
  "severity": "critical",
  "regulation_citation": "12 CFR 1026.19(f)(1)(ii)"
}
----

*Supported temporal operators:*

* `gte` -- target date >= source date + offset (e.g., closing >= CD receipt + 3 business days)
* `lte` -- target date <= source date + offset (e.g., rate lock expiration <= application date + 60 days)
* `eq` -- target date = source date + offset (exact date match with offset)
* `within` -- target date is within plus-or-minus offset of source date (date proximity)

*Calendar support:*

* `us_federal` -- US federal business day calendar (excludes weekends + federal holidays)
* `calendar_days` -- simple calendar day count
* `state:{code}` -- state-specific calendar (for state-level regulatory deadlines)

*Edge cases:*

* If either date field is missing or unparseable, the rule produces `unable_to_determine` with confidence x0.5.
* Date parsing supports ISO 8601, MM/DD/YYYY, and common US date formats.
The parser normalizes to ISO 8601 before comparison.

[[sect-pack-expression-language]]
====== Expression Language (CEL)

Decision D4 established declarative JSON as the pack interface.
As pack complexity grows, pure JSON configuration becomes insufficient for expressing real-world regulatory logic -- particularly conditional chains, cross-field validations, and computed thresholds.

The pack expression language extends D4 with a controlled, sandboxed expression evaluator.
AuditIQ adopts the *Common Expression Language (CEL)* -- a non-Turing-complete expression language designed by Google for security-critical policy evaluation.

*Why CEL:*

* *Non-Turing-complete:* CEL expressions always terminate.
No infinite loops, no recursion, no unbounded computation.
This is a hard security requirement for packs.
* *Sandboxed:* CEL has no I/O, no network access, no file system access, no side effects.
It evaluates expressions against provided data only.
* *Auditable:* CEL expressions are human-readable strings embedded in JSON.
Examiners can review them directly.
* *Proven:* CEL is used in Google Cloud IAM, Kubernetes admission webhooks, Firebase security rules, and Envoy proxy.
It is production-hardened for policy evaluation.

*Available variables in CEL expressions:*

* `target` -- the target record object (all fields accessible by field_id)
* `source` -- the source document object (all extracted fields)
* `results` -- map of prior rule results (keyed by rule_id), each containing `value`, `status`, `confidence`
* `metadata` -- audit context (audit_id, pack_version, run_date)

*Expression constraints (enforced at pack validation):*

* Maximum expression length: 1,024 characters.
* No function definitions -- expressions only.
* Only CEL standard functions + AuditIQ custom functions (see below).
* Expressions must type-check at pack validation time.

*AuditIQ custom CEL functions:*

* `businessDaysBetween(date1, date2, calendar)` -- returns business day count
* `normalize(value, type)` -- applies normalization rules (e.g., currency formatting, address standardization)
* `lookupEnum(value, table_name)` -- validates value against a reference enum table defined in the pack
* `withinTolerance(actual, expected, tolerance, unit)` -- returns boolean for tolerance comparison

*Example expressions:*

[source]
----
// FHA-specific PMI validation
target.loan_type == 'FHA' && source.mortgage_insurance_premium > 0

// TRID 3-day rule
businessDaysBetween(source.cd_received_date, target.closing_date, 'us_federal') >= 3

// Cross-field LTV validation
abs(target.ltv_ratio - (target.loan_amount / source.appraised_value * 100)) <= 0.5

// Conditional severity escalation
results['loan-amount-check'].status == 'mismatch' && target.loan_amount > 500000
----

===== Rule Precedence and Conflict Resolution

When multiple rules evaluate the same target field and produce conflicting results, the engine applies the following precedence:

. *Highest severity wins:* If rule A produces a `critical` finding and rule B produces a `low` finding for the same field, the critical finding is reported.
The low finding is attached as a supplementary note.
. *Most specific rule wins:* A conditional rule that matches its precondition takes precedence over a generic rule for the same field.
. *Explicit conflict flag:* If two rules of the same severity produce contradictory findings (one says match, one says mismatch), both are flagged for human review with a `rule_conflict` indicator.

===== Rule Execution Order

Rules are executed in dependency-aware order:

. All rules are topologically sorted by `depends_on` references.
. Rules with no dependencies execute first, in parallel where possible.
. Rules with dependencies execute after their prerequisites complete.
. The engine enforces a maximum total execution time of 30 seconds per record.
If this budget is exceeded, remaining rules are marked `timed_out` and routed to manual review.

[[sect-pack-ai-boundary]]
==== AI-Pack Boundary Specification

The pack is declarative; the AI pipeline is probabilistic.
The boundary between these two domains is one of the most critical design decisions in AuditIQ.
Getting it wrong means either: (a) packs that are too dumb to handle real documents, or (b) AI that makes unauditable decisions.
This section draws the line precisely.

===== The Boundary Principle

*Deterministic operations live in the pack.
Probabilistic operations live in the AI pipeline.
Every crossing of the boundary is explicitly labeled, scored, and auditable.*

[cols="30%,35%,35%",options="header"]
|===
|Operation |Owner |Rationale

|Target data validation (schema check)
|Pack (deterministic)
|Target data is structured; validation is rule-based; no uncertainty.

|Source document extraction (OCR, layout parsing)
|AI pipeline (probabilistic)
|Unstructured documents introduce OCR uncertainty; extraction confidence (AFC) quantifies this.

|Schema mapping (target field -> source field)
|AI pipeline (probabilistic) with pack constraints
|Mapping requires semantic similarity judgment; confidence scored; pack constrains candidate space via source schemas.

|Comparison execution (target value vs. source value)
|Pack (deterministic)
|Once values are extracted and mapped, comparison is rule-based.
The pack defines what "equal" or "within tolerance" means.

|Scoring (severity assignment)
|Pack (deterministic)
|Severity is defined by the pack rule.
Confidence is computed from the three-layer formula.

|Narrative generation (finding descriptions, report summaries)
|AI pipeline (probabilistic)
|Natural language generation is inherently probabilistic.
All AI narratives require human approval before finalization.

|Rule suggestion (during pack authoring)
|AI pipeline (advisory only)
|AI can suggest new rules based on regulatory text analysis, but a human compliance expert must approve and add them to the pack.
|===

===== AI-Assisted Rule Creation

When RiskExec compliance experts author packs, AI assists by:

. *Regulatory text analysis:* Given a regulation citation (e.g., 12 CFR 1003.4), AI extracts candidate audit rules -- field requirements, validation conditions, and severity suggestions.
. *Rule draft generation:* AI generates draft comparison rules in the pack JSON schema format.
These are suggestions, not final rules.
. *Gap detection:* AI compares a draft pack against the full regulatory text and flags coverage gaps -- fields or requirements that have no corresponding rule.

*Guardrail:* AI-suggested rules are always created in `draft` status.
They cannot be published until a human compliance expert reviews and approves each rule.
The pack manifest records `ai_suggested: true` on any rule that originated from AI assistance, and this flag is visible in exam-ready bundles.

===== AI in the Rule Execution Hot Path

The general principle is that rule execution is deterministic.
However, there are specific scenarios where the AI pipeline is invoked during rule execution:

*Scenario 1: Extraction confidence below threshold.*
If a source document field has AFC < 0.50, the value is unreliable.
The rule engine can invoke AI re-extraction: a second-pass extraction using a different model (e.g., LLM-based extraction via Mistral) to produce an alternative value with its own AFC score.
If the re-extracted value's AFC exceeds the threshold, it is used.
If not, the rule produces `unable_to_determine`.

*Scenario 2: Fuzzy/semantic matching for unstructured fields.*
Some fields (e.g., property address, borrower name) cannot be compared with exact_match due to formatting variations.
The rule engine delegates these to a fuzzy matching service that uses NLP-based similarity scoring.
The fuzzy match confidence is recorded as an additional signal in the rule's confidence calculation.

*Scenario 3: AI-assisted normalization.*
When standard normalization rules (address standardization, currency formatting) fail, AI-based normalization can be invoked as a fallback.
The AI normalization result is flagged with `ai_normalized: true` and its confidence is reduced (x0.7) to reflect the additional uncertainty.

*In all three scenarios:*

* The AI invocation is logged as an audit event with model version, input, output, and confidence.
* The finding is tagged with `ai_assisted: true` in the finding record.
* Human-in-loop triggers apply: any AI-assisted finding with severity >= `high` requires mandatory human review regardless of confidence score.

===== Confidence Propagation Model

The three-layer confidence model (<<sect-scoring-framework>>) computes combined confidence as:

....
combined_confidence = min(mapping_confidence, extraction_confidence, rule_confidence)
....

This section specifies precisely how each layer is computed in the context of pack rule execution.

*Extraction confidence (AFC):*
Produced by the Document Processing Service.
Per-field composite score from 6 signals (see <<sect-glossary>>).
When extraction is not applicable (record-to-record audit), defaults to 1.0.

*Mapping confidence:*
Produced by the Schema Mapping Service.
5-signal hybrid heuristic (name similarity, synonym match, type compatibility, document section context, historical acceptance).
When mapping is pre-defined in the pack (no ambiguity), defaults to 1.0.

*Rule confidence:*
Starts at `base_confidence` from the pack rule definition.
Reduced by the following modifiers:

* Precondition not satisfied (dependent rule unclear): x0.7
* Normalization failed: x0.5
* Fewer than `minimum_sources` available (multi-doc rules): x0.7
* AI-assisted extraction used: x0.85
* AI-assisted normalization used: x0.7

Multiple modifiers stack multiplicatively:

....
rule_confidence = base_confidence * modifier_1 * modifier_2 * ... * modifier_n
rule_confidence = max(0.0, min(1.0, rule_confidence))
....

*Combined confidence routing:*

* >=0.85: Auto-confirm eligible.
No human review required.
* 0.50-0.84: Human review required.
Reviewer validates finding.
* <0.50: Finding rejected.
Routed to manual re-review if severity is critical or high.

These thresholds are pack-configurable via `scoring_rules.confidence_thresholds`.
The values above are the platform defaults and the values used by the HMDA Origination pack.

For the full regulatory compliance hierarchy (five layers from regulation to audit rule set), institution policy profiles, AI-driven policy profile creation pipeline, and effective rule set merge logic, see <<sect-policy-management-engine>>.

==== Pack Versioning and Distribution

*Version format:* Semantic Versioning (MAJOR.MINOR.PATCH)

* *MAJOR:* Breaking changes to schemas or rule logic (requires migration).
Increment when: target field added/removed, comparison rule logic fundamentally changes, severity assignments revised across multiple rules.
* *MINOR:* New rules or schemas added (backward-compatible).
Increment when: new comparison rules added for existing fields, new source document type supported, regulatory citation updated.
* *PATCH:* Bug fixes, typo corrections (no functional changes).
Increment when: rule description typo fixed, test case corrected, no change to rule evaluation behavior.

*Distribution channel:*

* *Phase 1:* RiskExec ships HMDA Origination QC pack v1.0 in the platform binary.
Pack is embedded in the deployment artifact and versioned with the platform release.
* *Phase 2+:* Packs distributed via internal pack registry API with version pinning and rollback capability.
Customers pin to a specific pack version per audit run.
Rollback to a prior pack version is supported for new runs.

[[sect-pack-testing]]
==== Pack Testing and Validation Harness

===== Harness Requirements

* Load a pack manifest and validate its JSON schema compliance.
* Validate all CEL expressions for syntax and type correctness.
* Detect circular dependencies in rule chains.
* Load and execute each test case against the pack rules.
* Compare actual findings to expected findings.
* Report pass/fail, coverage metrics, rule execution time, and expression evaluation results.

===== Test Case Requirements

*Minimum test cases per rule:*

* 1 positive case (values match -- no finding expected)
* 1 negative case (values mismatch -- finding expected with specified severity)
* 1 edge case (missing field, boundary value, format variation, or multi-document disagreement)

*Golden dataset requirements:*

Each regulation-specific pack must include a golden dataset: a curated set of anonymized loan records with known-correct findings, constructed from production data.
The golden dataset serves as the regression test baseline.

* *HMDA Origination pack:* Golden dataset of >=200 anonymized loan records from >=5 lenders, covering all loan types (conventional, FHA, VA, USDA), property types (1-4 family, multifamily, manufactured), and action taken codes (1-8).
* *Per-document-type:* Golden documents with pre-verified field extractions for each source document type the pack supports.

===== Success Criteria (Before Publication)

* All test cases pass (0 failures).
* Rule coverage >=90% of pack rules have at least one test case that exercises them.
* No test case execution exceeds 5 seconds per record.
* All CEL expressions evaluate without runtime errors.
* No circular dependencies detected.
* All AI suggestions (if any) are labeled and traced to model/version.
* Regression: No prior-version test case that previously passed now fails (unless the failure is intentional and documented in the change log).

===== Continuous Regression Testing

Published packs are retested nightly against their golden datasets.
If a regression is detected (e.g., due to platform engine changes), an alert is generated and the pack is flagged for review.
This ensures that platform updates do not silently break pack behavior.

*Backward compatibility:* The `profileId` and `profileVersion` parameters are optional.
When no Institution Policy Profile is provided, the Effective Rule Set equals the base Regulatory Pack -- all rules are tagged `rule_origin: regulatory`.
This ensures existing audit runs and institutions without policy profiles continue to work without modification.

[[sect-embedding-architecture]]
==== Sentence Transformer Embedding Architecture

The AI-Driven Policy Profile Creation Pipeline (<<sect-ai-policy-pipeline>>) depends on a two-layer embedding architecture that bridges unstructured policy language and the structured compliance knowledge graph.
Neither layer works well alone: the graph tells you what concepts exist and how they relate; the embeddings tell you which sentences in messy, unstructured policy documents are _about_ those concepts.

[[sect-embedding-domain-finetuning]]
===== Domain-Fine-Tuned Sentence Transformer

All embedding operations -- regulatory citation retrieval, policy sentence indexing, and policy-to-regulation alignment -- share a single sentence transformer (SBERT) fine-tuned on financial regulatory language.
This model is registered once in the Model Registry (see <<sect-model-registry>>) and versioned alongside the regulatory corpus.

General-purpose embedding models (e.g., Nomic Embed, BGE-large) are insufficient for compliance text because they fail on three critical dimensions:

* *Negation sensitivity.* "Loan amount shall match" and "loan amount shall not exceed" have opposite compliance implications but high cosine similarity in general embedding space.
A fine-tuned model learns that negation inverts the compliance vector.
* *Numeric threshold context.* "$25" and "$50" are near-identical to a general model but represent materially different compliance postures.
A fine-tuned model encodes threshold magnitude as a salient feature.
* *Deontic language.* "Must," "shall," "may," and "should" carry precise legal meanings -- mandatory obligation, permissive option, aspirational guidance -- that general models treat as soft synonyms.
A fine-tuned model places these terms far apart in compliance embedding space.

The fine-tuning corpus and training signal are bootstrapped from the graph itself (see <<sect-training-data-bootstrap>>).

[[sect-two-layer-embeddings]]
===== Two Embedding Layers, One Graph

.Two-Layer Embedding Architecture
[mermaid]
----

  F --> G
  G --> H["Graph anchoring edge<br/>PolicySentence ──implements──→ GraphNode"]

*Layer 1 -- Ontology node embeddings (structural):* Each node in the unified compliance graph -- every FIBO class, every FinRegOnt regulation section, every MISMO field definition -- receives an embedding that encodes both its _position in the graph_ (subclass hierarchy, equivalence relations, restriction edges) and its _label and textual description_.
OWL2Vec* generates these embeddings by walking the OWL graph structure and combining structural features with label semantics.
The on2vec approach extends this by integrating GNN-based graph structure with sentence transformer text features into a single embedding.

*Layer 2 -- Document sentence embeddings (textual):* Every sentence extracted from a regulation, guidance document, institution policy, or procedure is embedded using the domain-fine-tuned sentence transformer.
Embeddings are stored in Qdrant (the same vector store used by the Regulatory Citation Engine -- see <<sect-regulatory-citation-engine>>) with metadata tagging the source document, sentence position, and document type.
Sentence-level granularity is required because individual policy sentences carry distinct obligations that paragraph-level chunking would conflate.

*Cross-space anchoring:* When a policy sentence is embedded, it is not stored as a free-floating vector.
It is _anchored_ to the nearest graph node(s) via cosine similarity between the sentence embedding (Layer 2) and the ontology node embeddings (Layer 1).
Top-_k_ candidate nodes are identified by similarity, then an LLM reranker confirms the anchor ("is this sentence genuinely about this regulatory concept?").
Confirmed anchors create a graph edge: `PolicySentence ──implements──→ GraphNode`.

This anchoring is what makes the graph _useful_ rather than merely _correct_.
The graph provides the structural scaffold; the embeddings provide the glue that attaches messy, unstructured institutional language to precise regulatory concepts.

[[sect-embedding-boundary-guardrail]]
===== Design Guardrail -- Embedding Boundaries

[IMPORTANT]
====
Embeddings are used _only_ at the boundary where unstructured text meets structured knowledge.

Inside the graph, relationships -- subclass, equivalence, `owl:sameAs`, FinRegOnt-to-FIBO mappings -- are traversed via SPARQL and graph queries, not approximated as embedding distances.
Structural relationships are _exact_; embedding similarity is _approximate_.
Substituting one for the other introduces false positives where embedding similarity says two concepts are "close" but the graph knows they are structurally unrelated.
====

===== Phase 2 -- AI-Driven Policy Requirement Extraction

AI reads each classified policy document and extracts *policy-level requirements* -- not loan-level field data.

|Requirement Type |Description |Example

|Thresholds and tolerances

|Procedural requirements
|Steps or approvals mandated by the institution's procedures

|Documentation requirements
|Documents or records that must be present in the loan file

|Conditions that trigger internal review or escalation

Each extracted requirement carries:

* A source citation (document_id, section, page, verbatim quote)
* An extraction confidence score
* A preliminary classification (threshold, procedural, documentation, exception, escalation, timing)
* A *graph anchor reference* -- the ontology node(s) to which the requirement's source sentence was anchored in Phase 1, with anchoring confidence scores

The graph anchor reference is critical because it links the extracted requirement to a precise concept in the compliance knowledge graph, not just to a document location.
A requirement whose source sentence anchors strongly to a known regulatory concept (e.g., `fibo:LoanAmount` linked to `finregont:§1003.4(a)(7)`) is inherently more trustworthy than one that floats unanchored in the vector store.
The anchoring confidence is factored into the alignment confidence in Phase 3.

*Pass 1 -- Structural graph traversal:* Each base pack rule maps to one or more ontology nodes via the compliance knowledge graph (e.g., rule `hmda-loan-amount-tolerance` -> `finregont:§1003.4(a)(7)` -> `fibo:LoanAmount` -> `mismo:LoanPrincipalAmount`).
The system queries the graph for any `PolicySentence ──implements──→` edges attached to these nodes -- sentences that were anchored during Phase 1 intake.
If anchored policy sentences exist, alignment classification proceeds directly from the structured relationship.
This is the preferred path because it produces _structurally grounded_ alignments, not probabilistic retrieval results.

*Pass 2 -- Vector fallback:* If the graph traversal finds no anchored policy sentences for a given rule's concept nodes, the system falls back to semantic search.
A query is generated from the rule's description and regulatory citation, and RAG retrieval searches the full policy sentence vector store in Qdrant.
Results from the fallback path carry lower alignment confidence because they lack structural grounding.

*Alignment categories:*

* *Match:* The policy restates or directly implements the regulatory requirement.
No override is needed, but the policy citation is captured in the `policy_to_regulation_traceability` array for exam defensibility.
* *Tightening:* The policy imposes a stricter threshold or higher severity than the regulation requires.
A `threshold_override` entry is generated.
* *Extension:* The policy adds requirements that go beyond the scope of the base pack rule (e.g., additional approval steps, documentation checks).
A `supplemental_rule` or `procedural_checkpoint` entry is generated.
* *Gap:* Neither structural traversal nor vector fallback found corresponding policy language.
A `gap_analysis` entry is generated with the full graph path that was inspected -- which regulatory concept was checked, which FIBO/MISMO nodes were traversed, and what the search yielded.
This dual-pass failure constitutes _evidence of absence_, not just _absence of evidence_, because the graph provides structural completeness: the system knows every regulatory concept that should have coverage.

[[sect-alignment-confidence]]
*Alignment confidence:* Each alignment carries a composite confidence score incorporating four signals:

[cols="30%,15%,55%",options="header"]
|===
|Signal |Weight |Description

|Graph anchor strength
|0.35
|Cosine similarity between the policy sentence embedding and the ontology node embedding at anchoring time.
Higher values indicate the sentence is semantically close to the formal concept.

|LLM reranker confirmation
|0.25
|Confidence from the LLM reranker that confirmed the anchor in Phase 1.
Measures whether the sentence genuinely addresses the regulatory concept, not just superficially resembles it.

|Structural distance
|0.20
|Inverse of the hop count between the regulation node (the pack rule's origin) and the anchored concept node in the graph.
Direct matches (zero hops) score 1.0; multi-hop traversals decay.

|Semantic similarity (RAG)
|0.20
|Cosine similarity from the vector store retrieval.
For Pass 1 (graph-resolved) alignments, this signal is computed but weighted lower.
For Pass 2 (fallback) alignments, this is the primary signal.
|===

Low-confidence alignments (composite score < 0.70) are flagged for mandatory human review in Phase 5.

Additionally, AI performs an *unanchored scan* -- searching for policy sentences that were embedded but did not anchor to any graph node above the confidence threshold.
These become candidate supplemental rules with no regulatory anchor, classified as `beyond_regulatory_scope` in the traceability mapping.

===== Phase 4 -- Draft Profile Generation

AI assembles the structured draft Institution Policy Profile from the Phase 2 and Phase 3 outputs:

* `policy_document_registry` -- populated from Phase 1 classification
* `threshold_overrides` -- generated from Phase 3 tightenings
* `supplemental_rules` -- generated from Phase 2 extensions and Phase 3 unanchored findings
* `procedural_checkpoints` -- generated from Phase 2 procedural/documentation/timing requirements
* `policy_to_regulation_traceability` -- generated from Phase 3 alignments
* `gap_analysis` -- generated from Phase 3 gaps

Every entry is tagged with `ai_suggested: true` and `approved: false`.
No AI-generated entry becomes effective until a human sets `approved: true`.

===== Phase 6 -- Ongoing Policy Change Detection

When an institution updates its policies or procedures, the profile must be re-evaluated.

* The institution uploads the updated policy documents.
* AI performs a diff analysis against the prior version of the policy documents.
* AI identifies which profile entries are affected by the policy changes.
* A draft profile update is generated containing only the deltas -- new overrides, modified thresholds, removed entries.
* The update goes through the same human review and dual approval process.
* The profile version is incremented (MINOR for additions/tightenings, MAJOR for structural changes).

*Trigger: Base pack update.*
When RiskExec publishes a new version of the base Regulatory Pack, all Institution Policy Profiles linked to that pack are automatically flagged for re-validation.
The AI pipeline re-runs Phase 3 (alignment) against the new pack rules and generates a delta report.

[[sect-training-data-bootstrap]]
==== Training Data Bootstrapping from the Graph

The domain-fine-tuned sentence transformer (see <<sect-embedding-domain-finetuning>>) requires training pairs that teach the model how compliance language clusters.
The compliance knowledge graph generates its own training signal, creating a self-improving loop.

===== Pair Generation Strategy

[cols="20%,40%,40%",options="header"]
|===
|Pair Type |Construction |Example

|Positive (same concept)
|Two sentences anchored to the same graph node -- one from a regulation, one from a policy.
The model learns that these sentences, despite different surface language, address the same compliance concept.
|Regulation: "The loan amount reported shall be verified against the closing disclosure within a tolerance of $50." +
Policy: "Loan principal discrepancies exceeding $25 require supervisor escalation." +
Label: same_concept

|Negative (different concept)
|Two sentences anchored to different graph nodes.
The model learns that superficial similarity (both mention loans, both mention dates) does not imply conceptual equivalence.
|Regulation: "Action taken date must be reported within 30 calendar days." +
Policy: "QC review must be completed within 15 business days of closing." +
Label: different_concept

|Hard negative (same domain, different obligation)
|Sentences anchored to sibling nodes in the ontology (e.g., two different HMDA fields under the same CFR section).
These are the most valuable training pairs because they force the model to distinguish between closely related but distinct compliance obligations.
|Regulation: "The applicant's ethnicity shall be reported as observed or stated." +
Policy: "Demographic data collection follows applicant self-identification procedures for race." +
Label: different_concept (ethnicity != race in HMDA)
|===

===== Self-Improving Loop

As more institution policy documents are processed through the pipeline, the graph accumulates more anchored sentences.
Each new anchoring event generates additional training pairs.
The fine-tuning cycle operates as follows:

. *Accumulate:* Policy documents from multiple institutions are processed; graph anchoring edges accumulate.
. *Extract pairs:* Positive pairs from same-node anchors; negative and hard-negative pairs from different-node anchors within the same regulatory domain.
. *Fine-tune:* When the training corpus reaches a minimum threshold (>=500 labeled pairs -- aligned with the calibration thresholds in <<sect-scoring-framework>>), the sentence transformer is fine-tuned and a new version is registered in the Model Registry.
. *Re-anchor:* Optionally, existing policy sentences are re-embedded with the improved model and re-anchored, improving alignment quality for previously processed institutions.
. *Repeat:* Each cycle produces a model that anchors more accurately, which generates higher-quality training pairs for the next cycle.

*Cold-start strategy:* Before the system has processed enough institution documents to generate 500 training pairs, the sentence transformer operates with a warm-start fine-tune on publicly available regulatory text (CFR corpus, FFIEC examination procedures, published consent orders).
This provides baseline deontic and threshold sensitivity while the institution-specific training signal accumulates.

----

*Merge rules:*

. *Threshold overrides are applied in one direction only.*
A policy profile can reduce a tolerance (e.g., $50 -> $25), escalate a severity (e.g., medium -> high), or raise a confidence threshold.
It cannot widen a tolerance, downgrade a severity, or lower a confidence threshold.
If a profile attempts to weaken a regulatory rule, the merge rejects the override and logs a validation error.
. *Supplemental rules are injected alongside base pack rules.*
Supplemental rule IDs are prefixed with `policy-` to distinguish them from regulatory rules in findings and reports.
Supplemental rules are executed by the same rule engine with the same confidence propagation model.
. *Procedural checkpoints are injected as a distinct rule category.*
Checkpoints may use `ai_document_detection` (the AI pipeline checks for the presence of a required document), `field_check` (deterministic field comparison), or `manual_attestation` (requires human attestation during the audit).
. *Every rule in the Effective Rule Set is tagged with its origin:*
* `regulatory` -- rule from the base pack, unmodified
* `policy_tightened` -- base pack rule with a policy-driven threshold override
* `policy_supplemental` -- supplemental rule from the policy profile
* `procedural_checkpoint` -- procedural checkpoint from the policy profile
. *The Effective Rule Set is immutable for the duration of the audit run.*
If either the pack or the profile is updated mid-audit, the running audit continues with the original Effective Rule Set.
New audit runs pick up the latest versions.

*Finding attribution:* Every finding produced by the audit engine includes an `origin` field indicating whether it was triggered by a regulatory rule, a policy-tightened rule, a policy-supplemental rule, or a procedural checkpoint.
This allows examiners to clearly distinguish regulatory violations from policy adherence issues in the exam-ready bundle.
