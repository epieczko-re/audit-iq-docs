=== Document Processing Service (Docling + OCR)

*Responsibilities:*

* Process source documents through Docling (document conversion + layout understanding substrate)
* Build AuditIQ Field Confidence (AFC) scores per extracted field (see below)
* Extract text, tables, key-value pairs via deterministic field extractor layer on top of Docling
* Fall back to cloud OCR for pages graded POOR or FAIR by Docling
* Store extracted text, AFC metadata, and Docling page-level confidence grades

____
*Critical Architecture Clarification (per Docling Technical Validation Report v1):* Docling is a *document conversion + layout understanding engine*, not a per-field extraction system. Its confidence scores (`layout_score`, `ocr_score`, `parse_score`) are *conversion-quality indicators* (POOR/FAIR/GOOD/EXCELLENT) that grade how well the page was converted—they are NOT per-field extraction confidence. AuditIQ must build its own *deterministic field extractor* that consumes Docling’s `DoclingDocument` output and produces per-field AFC scores. Do not use Docling’s numeric scores directly for field-level gating.
____

==== Docling-Graph Integration

*Overview.* AuditIQ adopts *Docling-Graph* as the Phase 1 baseline for document extraction and structured output. Docling-Graph combines Docling’s document conversion and VLM extraction capabilities with a Pydantic-based schema enforcement layer and a directed knowledge graph. This transforms free-text and OCR outputs into validated, strongly-typed Pydantic objects and a directed graph of entities and relationships that the Schema Mapping Service and Comparison Engine consume.

*Primary pipeline (Phase 1):*

....
Docling (document conversion & VLM)
  → Docling-Graph (Pydantic templates & structured extraction)
    → Knowledge Graph (directed graph with stable node IDs & edge metadata)
      → Schema Mapping Service (graph-aware mapping)
        → Comparison & Scoring Engine
....

*Why Docling-Graph:*

* Deterministic, schema-validated outputs (Pydantic objects) — extraction results are typed and validated at extraction time, not downstream
* Cross-document corroboration through explicit graph edges — improves `cross_doc_redundancy` and adds a new `graph_corrob_score` AFC signal
* Exportable graph artifacts for exam-ready bundles (CSV, Cypher, Neo4j) — supports Section 21 evidence requirements
* Supports both VLM and LLM extraction backends — flexible inference routing
* Trace/debug exports for provenance and troubleshooting
* *MIT-licensed* — lower licensing risk than GPL-linked alternatives

_(Source: Docling-Graph README, https://github.com/docling-project/docling-graph)_

*LLM Integration (LiteLLM / Mistral):* Docling-Graph integrates with *LiteLLM* — a single interface to local and remote LLM runtimes (vLLM, Ollama) and cloud providers (Mistral, OpenAI). For Phase 1 the recommended LLM inference path is *self-hosted Mistral* using vLLM or Ollama orchestrated via LiteLLM, with `provider_override` set to `mistral` in Docling-Graph configs. This provides high throughput, lower per-call cost at scale, and improved data residency controls while preserving the option to use cloud LLMs during early validation. See Section 25 (Model Deployment: Mistral Self-Hosted) for hardware and operational guidance.

*Integration guidance:*

* *Templates:* Implement Pydantic templates for each key document type (Closing Disclosure, Loan Estimate, 1003/URLA) and version them. Templates must include `model_config` fields for stable graph IDs.
* *Fallback strategy:* Prefer Docling-Graph LLM extraction (LiteLLM → self-hosted Mistral) when Docling page grade or initial AFC is borderline. Use cloud OCR fallback only for documents where Docling conversion fails at the page level (e.g., encrypted PDFs, severely degraded scans). See updated fallback logic in Section 10.2.3.
* *Provenance:* Every extracted Pydantic object records `pydantic_template_id`, `template_version`, `graph_node_id`, and `graph_export_version` in the extraction metadata (see Section 22.3 for provenance fields).
* *Legal:* Docling-Graph is MIT-licensed. If using self-hosted Mistral model weights, obtain legal sign-off confirming the chosen Mistral model license permits self-hosting under your deployment terms.

==== AuditIQ Field Confidence (AFC)

AFC is a deterministic, per-field composite score built from six signals (v2 — updated to include Docling-Graph corroboration):

[cols="20%,10%,30%,40%",options="header"]
|===
|Signal |Weight |Source |Description
|`docling_page_grade` |0.12 |Docling conversion output |Page-level conversion quality (POOR=0.2, FAIR=0.5, GOOD=0.8, EXCELLENT=1.0)
|`ocr_segment_confidence` |0.23 |Docling OCR or cloud OCR fallback |Character-level OCR confidence for the text segment containing this field
|`pattern_validity` |0.20 |AuditIQ field extractor |Regex/parse validation (e.g., date format, currency pattern, HMDA enum membership)
|`layout_anchor_consistency` |0.18 |AuditIQ field extractor |Does the field appear in the expected document region with expected label proximity?
|`cross_doc_redundancy` |0.12 |AuditIQ field extractor |Does the same value appear in a corroborating document (e.g., loan amount on both CD and LE)?
|`graph_corrob_score` |0.15 |Docling-Graph knowledge graph |Graph-based corroboration: does the extracted value appear as a graph node with matching normalized values and correct linking edges across documents? Exploits Docling-Graph's directed knowledge graph for explicit entity-relationship validation.
|===

NOTE: AFC v1 (5-signal) remains valid during Phase 0 POC and early Phase 1a until Docling-Graph pipeline is integrated. Systems must support both v1 and v2 formulas and log which version was used.

*AFC formula (v2 — with Docling-Graph):*

....
afc = (docling_page_grade * 0.12) +
      (ocr_segment_confidence * 0.23) +
      (pattern_validity * 0.20) +
      (layout_anchor_consistency * 0.18) +
      (cross_doc_redundancy * 0.12) +
      (graph_corrob_score * 0.15)
....

NOTE: Weights are calibration targets — tune during Phase 1a using labeled extraction data. See Canonical Glossary (Appendix) for routing thresholds (unchanged: ≥0.85 auto-confirm, 0.50–0.85 human review, <0.50 manual entry).

*AFC routing thresholds:*

* `afc >= 0.85`: Auto-confirm (no human review required)
* `0.50 <= afc < 0.85`: Human review required (field highlighted in UI)
* `afc < 0.50`: Flag as extraction failure; route to manual entry queue

==== Fallback Logic (Refined — Docling-Graph-First)

*Phase 1 fallback chain:* Docling-Graph (VLM) → Docling-Graph (LLM via LiteLLM/Mistral) → Cloud OCR → Manual entry

....
for each document:
  // Step 1: Attempt Docling-Graph structured extraction (VLM backend)
  context = docling.convertDocument(document)
  graph_result = docling_graph.run_pipeline(
    context, template=pydantic_template, backend="vlm"
  )
  compute afc including graph_corrob_score

  if graph_result.success and afc >= 0.50:
    persist extraction (Pydantic objects + graph)
    recordExtractionEvent(auditId, documentId, 'docling_graph_vlm')
  else:
    // Step 2: Attempt LLM extraction via LiteLLM (self-hosted Mistral/vLLM)
    llm_result = docling_graph.run_pipeline(
      context, template=pydantic_template, backend="llm",
      provider="litellm", provider_override="mistral"
    )
    recompute afc

    if llm_result.success and afc >= 0.50:
      persist extraction (Pydantic objects + graph)
      recordExtractionEvent(auditId, documentId, 'docling_graph_llm_mistral')
    else:
      // Step 3: Fall back to cloud OCR or manual entry
      triggerCloudOCRFallback(document)
      recordFallbackEvent(auditId, documentId, 'cloud_ocr_fallback')
    end
  end
end
....

*Legacy page-level fallback (still applies within each step):*

....
for each page in document:
  grade = docling.convertPage(page).grade  // POOR | FAIR | GOOD | EXCELLENT

  if grade in ['POOR', 'FAIR']:
    triggerCloudOCRFallback(page)
    recordFallbackEvent(auditId, documentId, pageNumber, grade)
  end

  // Even for GOOD/EXCELLENT pages, if >30% of key fields have AFC < 0.50:
  if (countKeyFieldsBelowThreshold(page, 0.50) / totalKeyFields(page)) > 0.30:
    triggerCloudOCRFallback(page)
    recordFallbackEvent(auditId, documentId, pageNumber, 'key_field_threshold')
  end
end
....

*Cloud OCR fallback providers (ranked by cost):* 1. AWS Textract `DetectDocumentText` — $0.0015/page 2. Google Enterprise Document OCR — $0.0015/page ($1.50/1000 pages) 3. AWS Textract `AnalyzeDocument` — $0.01/page (tables/forms; use only when needed)

==== Known Extraction Failure Modes

The Docling validation report identified 10 failure modes AuditIQ must handle:

[cols="25%,25%,25%,25%",options="header"]
|===
|# |Failure Mode |Likely Impact |Mitigation
|1 |Scanned PDF with degraded OCR quality |Low AFC on text-heavy fields |Cloud OCR fallback; AFC flags for review
|2 |Multi-column layouts mis-parsed by Docling |Fields assigned to wrong columns |Layout anchor consistency check in AFC
|3 |Table cells merged/split incorrectly |Numeric values extracted from wrong cell |Table structure validation + cross-doc redundancy
|4 |Handwritten annotations overlaid on printed text |OCR garbles both layers |Fallback to cloud OCR; manual review flag
|5 |Non-standard document templates |Field extractor patterns don’t match |Pattern library per document type; fallback to manual
|6 |Watermarks / stamps obscuring text |Reduced OCR segment confidence |Pre-processing (watermark removal); AFC flags
|7 |Mixed-orientation pages (landscape tables in portrait doc) |Layout analysis failure |Docling orientation detection; page rotation preprocessing
|8 |Extremely long documents (>50 pages) |Timeout / memory exhaustion |Chunked processing; page-level parallelism
|9 |Encrypted or password-protected PDFs |Docling cannot open |Pre-check + user notification; manual upload path
|10 |Digital signatures / form fields (XFA) |Docling may skip interactive form data |Flatten PDF before processing; extract XFA data separately
|===

=== Schema Mapping Service

*Responsibilities:*

* Suggest field-level mappings between MISMO-aligned canonical target schema (see Section 11.2) and source schema (customer-uploaded data from LOS, CSV, or document extraction)
* Validate that target field references exist in the `mismo_dictionary` (or use `custom:` namespace)
* Produce two-layer confidence scores for each suggestion (raw signals + calibrated decision score)
* Clearly demarcate CD-supported fields (~10–20% auto-mappable at ≥0.85) vs. fields requiring additional document types or manual entry
* Provide UI for human review and override with confidence-based routing
* Version and store approved mappings; feed override data back into calibration

____
*Scope constraint (per Schema Mapping Strategy Report):* HMDA LAR spans 48 data points → ~110 fields, with 37 “key HMDA data fields” (Fed CA Letter 18-9). Closing Disclosure (CD) documents alone cover only ~10–20% of these at ≥0.85 auto-mappable confidence. MVP must explicitly label fields as “CD-supported,” “requires 1003/URLA,” “requires appraisal,” or “manual entry required.”
____

==== Matching Algorithm (Phase 1 — Hybrid Heuristic)

The matcher uses a COMA-style combination of multiple independent signals, weighted to produce a raw mapping confidence:

* *Name similarity (0.30):* Levenshtein distance + Jaccard token overlap on field names, boosted by MISMO field description matching (uses `mismo_dictionary.description`)
* *HMDA/CD synonym dictionary (0.25):* Curated domain-specific equivalence table mapping known HMDA field names to CD label variants, abbreviations, and industry synonyms (e.g., `loan_type` ↔ `mortgage_product`, `prop_addr` ↔ `property_address`)
* *Type compatibility (0.20):* Target (MISMO) and source types must be compatible; MISMO enum fields validated against `mismo_dictionary.enum_values`
* *Document-section context (0.15):* Boost confidence when source field appears in expected document section (e.g., loan amount in CD page 1, Section J)
* *Historical acceptance (0.10):* If this mapping has been approved before by any reviewer in this tenant, confidence += up to 0.10 (decayed by recency)

*Mapping Pattern Taxonomy (5 types):*

[cols="34%,33%,33%",options="header"]
|===
|Pattern |Example |Signal Used
|Exact name match |`loan_amount` → `loan_amount` |Name similarity = 1.0
|Abbreviation mismatch |`prop_addr` → `property_address` |Synonym dictionary
|Synonym |`loan_type` → `mortgage_product` |Synonym dictionary
|Derived field |`ltv_ratio` (computed) |Rule engine; flagged as “derived” not “mapped”
|Multi-source field |`total_income` (W-2 + 1003 + pay stubs) |Flagged as “Phase 2: requires additional documents”
|===

==== Two-Layer Confidence Design

*Layer 1 — Raw signals (produced per mapping suggestion):*

....
conf_mapping = (
  (name_sim * 0.30) +
  (synonym_match * 0.25) +
  (type_compat * 0.20) +
  (doc_section_context * 0.15) +
  (historical_acceptance * 0.10)
)
....

where each component is normalized to [0.0, 1.0].

Additional raw signals passed to Layer 2: - `conf_extraction`: AFC score from Section 10.2 (how confident we are in the extracted value) - `conf_rule`: Business rule pass/fail from Section 10.5 (does the mapped value pass validation?)

*Layer 2 — Calibrated decision score:*

* Combines `conf_mapping`, `conf_extraction`, and `conf_rule` into a single calibrated probability
* Phase 1: Temperature scaling on logistic regression of Layer 1 signals (requires ≥200 labeled mapping decisions for initial calibration)
* Phase 2: Isotonic regression for non-parametric calibration (requires ≥500 labeled decisions)
* Calibration validation: Reliability diagrams (predicted confidence vs. actual accuracy in 10 bins); Expected Calibration Error (ECE) must be ≤0.05

*Routing thresholds (calibrated score):*

* ≥0.95: Auto-accept (reviewer sees green checkmark; ~10s visual confirmation). NOTE: Initial conservative threshold per Decision D2 (Section 19.3); may be lowered to ≥0.90 after calibration validation on ≥500 mappings.
* 0.70–0.95: Human review required (reviewer evaluates suggestion; ~45s per field)
* <0.70: Manual mapping required (no suggestion shown; reviewer selects from field list)

=== Findings Management Service

*Responsibilities:*

=== Audit Trail & Event Store (Immutable Log)

*Responsibilities:*

* Record every state change as an immutable, hash-chained event with RFC 8785 canonical JSON serialization
* Maintain global monotonic sequence IDs (per-tenant) for gap and reorder detection
* Produce cryptographic anchor points at configurable intervals for tamper resistance
* Support querying events by audit, finding, user, date range
* Enable audit replay for investigations and regulator exam response
* Support litigation defensibility via FRE Rule 902(13)/(14) self-authentication

____
*Critical design requirement (per Audit Trail Defensibility Report):* Hash chaining alone is necessary but NOT sufficient. An attacker with database write access can rewrite the entire chain. This design adds three mandatory layers: (1) RFC 8785 canonical JSON serialization for repeatable hash verification, (2) global monotonic sequence IDs for gap/reorder detection, (3) cryptographic anchor points that cannot be silently rewritten. See Section 19.4 (Decision D3) for design tier rationale.
____

==== Event Schema

[source,typescript]
----
interface AuditEvent {
  event_id: string;                    // UUID v7 (time-ordered)
  tenant_id: string;                   // Tenant isolation boundary
  sequence_id: number;                 // Global monotonic per tenant (NOT per audit)
  timestamp: string;                   // Server-assigned ISO 8601; client timestamps rejected

  event_type: string;                  // e.g., 'FindingCreated', 'FindingTriaged', 'FindingClosed'
  actor_id: string;                    // userId or system principal
  actor_role: string;

  payload: object;

  // Tamper evidence (hash chain)
  previous_event_hash: string;         // SHA-256 of RFC 8785-canonicalized predecessor event
  event_hash: string;                  // SHA-256 of RFC 8785-canonicalized THIS event

  // Cryptographic anchor (populated on anchor events only)
  is_anchor: boolean;                  // True every N events (configurable; default N=100)
  anchor_hash?: string;                // If is_anchor: hash written to WORM storage

  // Metadata
  source_system: string;               // e.g., 'comparison_engine', 'audit_ui', 'remediation_workflow'
  api_version: string;
  schema_version: string;              // Event schema version for forward compatibility
}
----

==== RFC 8785 Canonical JSON Serialization

*MANDATORY:* All event payloads MUST be serialized using RFC 8785 (JSON Canonicalization Scheme) before hashing. This ensures: - Deterministic key ordering (lexicographic) - Consistent number formatting (no trailing zeros, no positive sign) - Consistent Unicode normalization - No optional whitespace

....
canonical_json = RFC8785.canonicalize(event_without_hash_fields)
event_hash = SHA-256(canonical_json)
....

*Implementation requirement:* Use a well-tested RFC 8785 library (e.g., `canonicalize` npm package or equivalent). Do NOT hand-roll canonical JSON — edge cases (Unicode escaping, number precision, key ordering with special characters) are subtle and produce silent hash mismatches.

*Write-time validation:* Every event MUST pass round-trip verification before persistence:

....
assert(RFC8785.canonicalize(JSON.parse(RFC8785.canonicalize(event))) === RFC8785.canonicalize(event))
....

==== Hash Chain Computation

....
event_hash = SHA-256(
  RFC8785.canonicalize({
    event_id, tenant_id, sequence_id, timestamp, audit_id,
    entity_type, entity_id, event_type, actor_id, actor_role,
    payload, previous_event_hash, source_system, api_version, schema_version
  })
)
....

*Chain rules:*

* First event in tenant chain: `previous_event_hash = SHA-256("GENESIS:" + tenant_id)`
* Each subsequent event: `previous_event_hash = event_hash` of the immediately preceding event (by sequence_id)
* Gap in sequence_id → chain is broken → alert + investigation required

==== Cryptographic Anchor Points

Every N events (default N=100, configurable per tenant), the system writes an *anchor event*: - `is_anchor = true` - `anchor_hash` = SHA-256 of the event_hash at this point - Anchor hash is written to *WORM storage* (e.g., AWS S3 Object Lock, Azure Immutable Blob) — cannot be overwritten or deleted during retention period - Anchor verification: compare stored anchor hash against recomputed hash from event chain; mismatch = tampering detected

*Phase 2 upgrade:* Replace WORM anchors with KMS-signed checkpoint events (AWS KMS, Azure Key Vault) for non-repudiation and litigation support under FRE 902(13)/(14).

==== Operational Integrity Controls (NIST AU-9 / FFIEC)

Per NIST SP 800-53 Rev 5 AU-9 and FFIEC IT Examination Handbook:

* *No admin delete:* Database roles for audit_trail schema must NOT include DELETE permission; enforced at PostgreSQL role level
* *Separate storage:* Audit trail stored in dedicated schema/database; not co-located with application data that admins routinely modify
* *Read-only access for audit:* Regulator/auditor access via read-only database replica or export; no write access to production audit trail
* *Retention:* Minimum 7 years (configurable per tenant; aligned with NIST AU-11); events never purged during retention window
* *Backup verification:* Weekly automated restore-and-verify of audit trail backup; hash chain re-verified on restored data

==== Tamper Detection & Verification

*Chain verification tool (MVP — Phase 1a):*

* CLI tool that reads entire tenant event chain and verifies: 1. Every `event_hash` matches recomputed SHA-256 of RFC 8785-canonicalized event 2. Every `previous_event_hash` matches `event_hash` of preceding event (by sequence_id) 3. No gaps in sequence_id 4. All anchor hashes match WORM-stored values
* Output: “OK” or list of corrupted/missing events with sequence_id range

*Periodic integrity check (automated):*

* Daily: Verify last 24 hours of events (chain continuity + hash correctness)
* Weekly: Full chain verification for each tenant
* On anchor write: Verify chain from previous anchor to current anchor before writing to WORM

*Snapshot integrity (see Section 20):*

* When snapshots are materialized, compute snapshot hash and store in event stream
* Periodic re-materialization from events; compare against stored snapshot hash; mismatch = drift alert

=== AI Governance & Approval Workflow

*Core principle:* Every AI output is advisory, non-finalized, and requires explicit human approval before inclusion in official records.

*AI output categories:*

[arabic]
. *Extracted values (OCR/Docling):* Labeled “AI-extracted (advisory)” in UI. Recommendation to use: > High confidence (≥0.85) can be pre-confirmed; lower values require manual verification.
. *Mapping suggestions:* System suggests field pairings. Reviewer approves, rejects, or overrides. Approved mappings versioned and reusable.
. *AI-generated narratives:* Audit summary, root cause analysis, remediation recommendations. Generated on-demand with explicit approval workflow:
* Draft labeled “[AI-DRAFT — REQUIRES APPROVAL]”
* Manager reviews and approves or rejects
* Approved version embedded in report
* Rejected version discarded; Manager can request new draft
. *Rule generation (TBD Phase 2):* System suggests new comparison rules from discovered patterns. Requires explicit pack author review before deployment.
