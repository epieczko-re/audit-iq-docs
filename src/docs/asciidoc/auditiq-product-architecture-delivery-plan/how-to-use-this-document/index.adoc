== How to Use This Document

This document is organized into seven parts.
Read Part II (Operational Model) and Part III (Product Workflows & Requirements) before estimating or sprinting.
Those two parts define _what the system does_ and _how users interact with it_ — every requirement and every component in Part IV traces back to a workflow and FR table in Part III.

=== For the 4-Person Engineering Team

[arabic]
. *Start with Part II* — understand the audit run state machine, the finding remediation lifecycle, and the Three Lines Model RBAC framework.
These are the conceptual primitives that every service and screen is built around.
. *Read Part III cover to cover* — the user journeys (Upload & Ingestion, Extraction & Mapping, Finding Review, Remediation & Reporting, Dashboards, Admin & Registry) define the sprint-planning units.
Each journey contains workflow diagrams, screen wireframes, functional requirements (epics), architecture details (components, data flows, API contracts), and acceptance criteria in one place.
. *Use Part III's epic sections as the acceptance contract* — the FR tables are embedded inside each journey.
When you pick up a story, find the FR ID in the relevant journey's Functional Requirements subsection for the authoritative acceptance criteria and the embedded wireframe.
. *Reference Part IV for deep architecture detail* — the Architecture Overview and Component-Level Design sections provide cross-cutting architectural context beyond what is covered in each journey's Architecture subsection.
. *Check Part V before any AI-related work* — AI governance red lines, model registry requirements, and provenance logging are non-negotiable.
Read <<sect-ai-governance>> before writing any LLM integration code.
. *Plan sprints with Part VI* — the Workflow Delivery Matrix (<<sect-workflow-delivery-matrix>>) maps screens to services to sprints to owners.
The Quick Start (<<sect-quick-start-team>>) gives you the 10-screen MVP scope and a 6-sprint suggestion.

=== Document Structure Overview

* *Part I — Strategic Context:* Executive Summary, this reading guide, Problem Statement, and Product Vision.
Start here for business context, market positioning, and guidance on navigating the rest of the document.
* *Part II — Operational Model:* Concept of Operations (audit modes, state machines, finding lifecycle, RBAC).
The conceptual foundation for everything that follows.
* *Part III — Product Workflows & Requirements:* End-to-end workflow diagrams, all 26 MVP screen wireframes, functional requirements (epics with FR tables), and architecture context — all organized by user journey.
This is the primary sprint-planning reference.
* *Part IV — Architecture & Design:* Architecture Overview (including Audit Pack concepts, Compliance Hierarchy, and Exam-Ready Bundle philosophy), Component-Level Design (including Audit Pack Technical Specification, Policy Management & Compliance Engine, and Exam Pack Lifecycle Manager with full bundle specs).
* *Part V — AI & Governance:* AI Governance & Model Risk Controls, Model Registry, Model Deployment, Docling-Graph Integration, Immutability vs. Correctability.
* *Part VI — Delivery & Business:* Pricing, Quick Start for the 4-person team, Delivery Plan with Workflow Delivery Matrix, Risks & Mitigations, Future Roadmap.
* *Part VII — Reference:* Appendices (including Canonical Glossary and HMDA Field Coverage Matrix), Competitive Intelligence, Architectural Decisions (including MVP Decisions Locked), Document Control.

=== Audiences

* *Engineering team:* Parts II, III, IV (sprint-plan against Part III's user journeys which contain workflows, screens, FR tables, and architecture context; use Part IV for cross-cutting architecture detail).
* *Product / leadership:* Part I for context; Part VI for investment decisions and delivery timeline; Part VII for Competitive Intelligence.
* *Compliance / regulatory stakeholders:* Part V (AI Governance, Immutability) and Part VII (Canonical Glossary) for regulatory defensibility.

=== Conventions

* *TODO* = work item that must be completed before implementation starts.
* *ASSUMPTION* = a belief stated explicitly; risk if wrong is documented.
* *DECISION NEEDED* = requires stakeholder input before proceeding.
* *DECISION LOCKED* = decided for MVP; revisit criteria documented in <<sect-mvp-decisions>>.
* *Canonical Glossary:* All confidence metrics, routing thresholds, severity definitions, and calibration criteria are defined in <<sect-glossary>>.
If a value elsewhere in this document conflicts with the Glossary, the Glossary governs.
