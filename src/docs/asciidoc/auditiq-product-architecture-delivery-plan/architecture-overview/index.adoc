=== Key Architectural Concepts

This section introduces the foundational concepts that underpin every downstream component.

==== Audit Packs: The "Brain" Plugin Model

An audit pack encapsulates everything needed to audit a specific regulatory domain.
It is a versioned, declarative configuration artifact — not executable code.
The platform interprets the pack to drive ingestion validation, schema mapping, comparison rules, scoring, and report generation.
This makes AuditIQ extensible: adding a new regulation means creating a new pack, not modifying platform code.

*Phase 1:* One pack — HMDA Origination QC.
Pre-built by RiskExec. +
*Phase 2:* CRA, SBL (Section 1071).
Pre-built by RiskExec. +
*Future:* Client-customizable packs, potentially a third-party pack marketplace.

[[sect-target-source-schemas]]
==== Target vs. Source: Schema Definitions

AuditIQ distinguishes two data domains throughout the audit workflow.
Understanding this distinction is critical for every downstream component — ingestion, mapping, comparison, and findings.

*Target dataset* (also: "target data", "target records")::
The structured loan records under audit.
These are the institution's own data — the records they filed, originated, or reported.
+
--
* *Format:* Structured tabular data (CSV upload, API endpoint, or cross-module transfer from existing RiskExec modules such as HMDA or CRA).
* *Schema:* Defined by the audit pack (e.g., the HMDA LAR schema specifies required fields like `loan_amount`, `action_taken`, `property_address`).
* *Example:* A 5,000-record HMDA LAR CSV exported from the institution's Loan Origination System (LOS).
* *Determinism:* Target data is validated deterministically during ingestion — no AI is involved.
--

*Source documents* (also: "source data", "evidence documents")::
The evidence documents against which target records are compared.
These are the original loan file artifacts — the documents that prove (or disprove) the accuracy of the target data.
+
--
* *Format:* Unstructured or semi-structured documents (PDF, TIFF, JPEG, PNG) — or, in record-to-record audits, a second structured dataset.
* *Schema:* No predefined schema.
The Document Processing Service (Docling) extracts text, tables, and key-value pairs from source documents.
The Schema Mapping Service then maps extracted fields to the target schema.
* *Example:* A Closing Disclosure PDF, a Loan Estimate image, a URLA form.
* *Confidence:* Extraction from source documents produces confidence scores (AFC) because OCR and layout parsing introduce uncertainty.
--

*How they relate in the audit workflow:*

[cols="20%,40%,40%",options="header"]
|===
|Stage |Target Data |Source Data

|*Ingestion*
|Validated against pack schema (deterministic)
|Stored in Document Store; classified by document type

|*Extraction*
|N/A (already structured)
|Docling extracts values with extraction confidence (AFC)

|*Schema Mapping*
|Pack defines the canonical field names
|Extracted fields are mapped to target schema fields with mapping confidence

|*Comparison*
|Target value is the "expected" value
|Source value (extracted or manual) is the "actual" value

|*Finding*
|Discrepancy between target and source produces a finding
|Source document is linked as evidence
|===

*Record-to-record audits* (e.g., HMDA LAR vs. LOS database extract): both target and source are structured datasets.
In this case, extraction confidence defaults to 1.0 (no OCR uncertainty) and the audit focuses on mapping and rule confidence.

==== Three-Layer Confidence Scoring

AuditIQ applies confidence scoring at three independent layers — mapping, extraction, and rule application — which combine via a `min()` formula to produce a combined confidence score.
For full scoring formulas, layer weights, and threshold definitions, see <<sect-scoring-framework>> and the <<sect-glossary, Canonical Glossary>>.

==== Immutable Audit Trail with Regulator-Safe Semantics

Every state change in the system is recorded as an immutable event.
Events are append-only and hash-chained for tamper evidence.
The system differentiates between truly immutable artifacts (evidence, events) and correctible artifacts (findings, mappings) where corrections are modeled as new events that supersede prior state.
See <<sect-immutability>> for the full immutability/correctability specification.

=== Architectural Principles

[arabic]
. *Separation of concerns:* Ingestion → Processing → AI → Storage → Reporting.
. *Immutable audit trail:* Every state change is recorded as an event; events are append-only and hash-chained.
. *Declarative configuration:* Packs are JSON, not code.
The platform interprets them.
. *Confidence scoring:* Every output carries a quantified confidence score (extraction, mapping, rule application).
. *Human-in-loop by default:* AI outputs are advisory and non-finalized until explicitly approved by a human with the right role.
. *Multi-tenancy:* Row-level security at the database level.
No cross-tenant data leakage.

*Isolation enforcement by layer:*

* *Database (PostgreSQL RLS):* Every table containing tenant data has an RLS policy that filters rows by `current_setting('app.tenant_id')`.
The tenant ID is set on each database connection by the API middleware before any query executes.
Even if application code omits a `WHERE tenant_id = ...` clause, RLS silently filters rows.
RLS policies are enforced for all roles except the superuser migration role, which is used only for schema migrations and is not accessible to application services.
* *Blob Storage (per-tenant containers):* Each tenant's documents are stored in a dedicated Azure Blob container named `tenant-{tenant_id}`.
SAS tokens are scoped to the tenant's container with a 1-hour expiry.
Container-level access policies prevent cross-tenant enumeration.
* *Search (per-tenant Elasticsearch indices):* Each tenant has a dedicated index with a naming convention of `auditiq-{tenant_id}-{entity_type}`.
Index-level security aliases ensure queries are routed to the correct tenant index.
Cross-index search is disabled at the API layer.
* *API Gateway (JWT tenant extraction):* The API Gateway extracts `tenant_id` from the JWT `org_id` claim and injects it as a request header.
Backend services trust this header (Gateway is the only ingress point).
Requests without a valid `tenant_id` claim are rejected with 401 Unauthorized.
* *Vector Store (Qdrant):* Regulation embeddings are shared across tenants (public regulatory corpus).
Institution-specific policy embeddings are stored in per-tenant collections, with payload-level filtering by `tenant_id`.
[start=7]
. *Reproducibility:* Identical inputs + pack version + configuration = identical deterministic outputs.

==== Phase 1 Deployment (East US 2)

*Hosting model:* RiskExec-hosted multi-tenant SaaS.
All infrastructure runs in RiskExec's Azure subscription.
Tenant isolation is enforced via PostgreSQL Row-Level Security (RLS), per-tenant Blob containers, and per-tenant Elasticsearch indices.

*Compute:*

* *AKS cluster:* Azure Kubernetes Service (Standard tier).
3 node pools:
** *System pool:* 3× Standard_D4s_v5 (4 vCPU, 16 GB) — core services, orchestration, auth.
** *Pipeline pool:* 3–6× Standard_D8s_v5 (8 vCPU, 32 GB) — document processing, schema mapping, comparison engine.
Autoscales based on queue depth.
** *Infrastructure pool:* 3× Standard_D4s_v5 — Elasticsearch, Qdrant, Redis.

*Database:*

* *Azure Database for PostgreSQL Flexible Server:* General Purpose tier, 8 vCPU, 64 GB RAM.
* Multi-AZ deployment with automatic failover.
* Automated backups: daily full, continuous WAL archiving.
Point-in-time recovery up to 35 days.
* Read replicas: 1 replica for dashboard queries and search indexing.

*Networking:*

* All services deployed within a single VNet.
* AKS, PostgreSQL, and Blob Storage communicate over Private Endpoints (no public internet).
* Azure API Management is the only publicly accessible endpoint.
* Azure Front Door provides WAF, DDoS protection, and TLS termination.
* Network Security Groups (NSGs) restrict traffic between subnets.

*Region:* Azure East US 2 (Virginia).
US data residency guaranteed — no data leaves US Azure regions.

==== Disaster Recovery

*Strategy:* Multi-AZ within East US 2 (primary), with async replication to West US 2 (secondary) for PostgreSQL and Blob Storage.

*Failover procedure — AZ-level failure (automated, RTO < 5 minutes):*

[arabic]
. Azure Monitor detects AZ-level degradation via health probes and triggers P1 alert to on-call engineer.
. PostgreSQL Flexible Server performs automatic failover to standby replica in the surviving AZ (< 60 seconds).
Connection strings do not change; PgBouncer reconnects transparently.
. AKS scheduler reschedules pods from the failed AZ onto healthy nodes in surviving AZs (< 2 minutes).
Pod disruption budgets ensure minimum replicas remain available during rescheduling.
. On-call engineer verifies service health via Platform Health dashboard.
Confirms replication lag has caught up (< 30 seconds after failover).
. Incident logged in event store with `disaster_recovery_failover` event type for post-incident review.

*Failover procedure — full-region failure (manual, RTO < 1 hour):*

[arabic]
. Azure Monitor detects sustained East US 2 region failure (no healthy AZs for > 10 minutes).

==== Database Scaling Roadmap

This roadmap defines the scaling strategy for PostgreSQL as tenant count and data volume grow.

*Phase 1 — Vertical Scaling (1–50 tenants):*

* Single PostgreSQL Flexible Server instance: 8 vCPU, 64 GB RAM.
* PgBouncer connection pooling: transaction-mode, max 200 connections per pool, 20 reserved per tenant.
* 1 read replica dedicated to dashboard queries, search indexing, and report generation.
* Estimated cardinalities: ~50 tenants, ~500 users, ~2,000 audits/year, ~10M findings/year, ~50M events/year.
* Table partitioning by `tenant_id` on `findings`, `events`, and `audit_records` tables for query isolation.

*Phase 2 — Horizontal Sharding (50–500 tenants):*

* Citus extension for distributed PostgreSQL.
* Sharding key: `tenant_id` (colocates all tenant data on the same shard for join-local queries).
* Distribution strategy: hash-based distribution across 32 initial shards, rebalancing via `rebalance_table_shards()`.
* Reference tables (non-tenant-specific): `regulations`, `pack_definitions`, `system_config` replicated to all nodes.
* Connection pooling scales to per-coordinator routing with Citus-aware PgBouncer.

*Phase 3 — Multi-Region (500+ tenants):*

* Geo-sharded Citus deployment across Azure regions.
* Tenant affinity routing: tenants assigned to the nearest region at onboarding.
* Cross-region reads via read replicas; writes pinned to tenant's home region.

==== Failure Modes & Resilience Patterns

AuditIQ implements defense-in-depth resilience to ensure audit processing continues under degraded conditions.

*Circuit Breaker Pattern:*

All inter-service calls use circuit breakers with three states:

* *Closed (normal):* Requests flow through.
Failure count tracked per 60-second window.
* *Open (tripped):* After 5 consecutive failures or > 50% error rate in a window, the circuit opens.
All requests immediately return a fallback response for 30 seconds.
* *Half-Open (probing):* After the cooldown, one probe request is sent.
Success closes the circuit; failure reopens it.

*Retry Policies:*

* *Idempotent operations* (reads, status checks): Retry up to 3 times with exponential backoff (1s, 2s, 4s) and jitter.
* *Non-idempotent operations* (writes, state transitions): No automatic retry.
Failed operations are logged and surfaced to the Audit Orchestration Service for manual or workflow-driven retry.

*Component Failure Scenarios:*

[cols="25%,35%,40%",options="header"]
|===
|Failure Scenario |Impact |Mitigation

|Document Processing (Docling) unavailable
|New document extraction halts; in-flight extractions fail
|Circuit breaker trips after 5 failures.
Queued documents retry when service recovers.
Already-extracted documents unaffected.

|PostgreSQL primary failure
|All write operations fail

|Elasticsearch cluster degraded
|Search and full-text queries slow or unavailable
|Search results degrade gracefully: API returns cached results or a "search temporarily unavailable" indicator.

|Qdrant unavailable
|Regulatory citation lookups fail
|Citation Engine returns "citations pending" status.

|Azure Blob Storage throttled
|Document uploads/downloads slow
|Exponential backoff with retry.

Lifecycle tiering paused during throttling.

|API Gateway (APIM) overloaded
|User requests rejected (429 Too Many Requests)
|Rate limiting protects backend services.
Azure Front Door absorbs traffic spikes.

|AKS node failure
|Pods on failed node restart on healthy nodes
|Pod disruption budgets ensure minimum replicas remain available during rescheduling.
Stateless services recover in < 30 seconds.

|===

*Graceful Degradation Hierarchy:*

When multiple components degrade simultaneously, AuditIQ prioritizes in this order:

[arabic]
. *Data integrity:* Immutable event store and audit trail are never compromised.
. *In-progress audits:* Active audit runs continue processing with available data.
. *New audit creation:* New audits may be queued but not lost.
. *Search and reporting:* Non-critical features degrade last and recover first.

==== Configuration & Deployment Management

*Deployment Strategy:*

AuditIQ uses a blue-green deployment model on AKS to achieve zero-downtime releases.

* *Blue environment:* Current production workload.
* *Green environment:* New release deployed to a parallel set of pods.
* *Cutover:* After health checks pass on green, the API Gateway routes traffic to green.
Blue remains available for instant rollback for 1 hour.
* *Rollback:* If post-deployment health checks fail, traffic reverts to blue within 30 seconds via APIM routing update.

*Canary Releases (Phase 2):*

* New versions deployed to 5% of traffic initially.
* Automated analysis of error rates and latency for 15 minutes.
* Progressive rollout: 5% → 25% → 50% → 100% over 1 hour if metrics remain healthy.
* Automatic rollback if error rate exceeds baseline by > 2×.

*Configuration Management:*

* *Application configuration:* Stored in Kubernetes ConfigMaps and Secrets.
Changes applied via rolling restart (zero downtime).
* *NFR targets:* Defined in ConfigMaps.
Alert thresholds, rate limits, and scaling parameters are updatable without redeployment.
* *Audit Pack configuration:* Managed by the Audit Pack Manager service.
Pack updates are versioned and deployed independently of platform releases.
* *Infrastructure configuration:* Terraform modules in a dedicated `infra` repository.
All changes go through PR review and `terraform plan` approval before apply.

*Configuration Drift Detection:*

* Azure Policy enforces compliance rules on all resources (e.g., encryption enabled, public access disabled).
* Weekly automated drift scan compares live infrastructure state against Terraform state.
* Drift alerts route to the `#auditiq-ops` Slack channel for investigation.

*Dependency Upgrade Procedures:*

* Dependabot monitors all runtime dependencies for security vulnerabilities.
* Critical CVEs (CVSS ≥ 9.0): patch within 24 hours.
* High CVEs (CVSS 7.0–8.9): patch within 7 days.
* AKS Kubernetes version: upgrade within 30 days of new stable release.
* PostgreSQL major version: upgrade during quarterly maintenance window with full regression testing.
