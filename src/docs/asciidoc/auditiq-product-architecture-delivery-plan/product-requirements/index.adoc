*Deterministic boundary:* All ingestion operations are deterministic.
The same input file always produces the same validated output.
No AI is involved in ingestion.

===== Ingestion Workflow

The ingestion workflow covers audit setup, dataset upload, and schema validation.
These are the first steps in both manual and OCR-assisted audit modes (see <<sect-concept-of-operations>>).

[mermaid]
----
sequenceDiagram
  participant Admin
  participant System
  participant DB as Staging DB

  Admin->>System: Create audit run + select pack
  Admin->>System: Upload target dataset (CSV/API/cross-module)
  System->>System: Validate schema against pack definition
  alt Validation passes
    System->>DB: Store validated records in staging area
    System->>System: Compute per-record SHA-256 checksums
    System-->>Admin: Validation summary (UI + JSON/CSV download)
  else Validation fails
    System-->>Admin: Error report (missing columns, malformed rows)
    Admin->>System: Upload corrected dataset
  end
  Admin->>System: Upload source documents (PDF, TIFF, JPEG, PNG)
  System->>System: Classify documents (CD, LE, URLA, etc.)
  System->>System: Link documents to target records via loan number
  System-->>Admin: Document classification summary
----

===== Extraction Architecture

*Component:* Document Processing Service (<<sect-doc-processing>>)

*3-Tier Fallback Chain:*

....
for each document:
  // Step 1: Attempt Docling-Graph structured extraction (VLM backend)
  context = docling.convertDocument(document)
  graph_result = docling_graph.run_pipeline(
    context, template=pydantic_template, backend="vlm"
  )
  compute afc including graph_corrob_score

  if graph_result.success and afc >= 0.50:
    persist extraction (Pydantic objects + graph)
    recordExtractionEvent(auditId, documentId, 'docling_graph_vlm')
  else:
    // Step 2: Attempt LLM extraction via LiteLLM (self-hosted Mistral/vLLM)
    llm_result = docling_graph.run_pipeline(
      context, template=pydantic_template, backend="llm",
      provider="litellm", provider_override="mistral"
    )
    recompute afc

    if llm_result.success and afc >= 0.50:
      persist extraction (Pydantic objects + graph)
      recordExtractionEvent(auditId, documentId, 'docling_graph_llm_mistral')
    else:
      // Step 3: Fall back to cloud OCR or manual entry
      triggerCloudOCRFallback(document)
      recordFallbackEvent(auditId, documentId, 'cloud_ocr_fallback')
    end
  end
end
....

*Known Extraction Failure Modes:*

[cols="10%,25%,25%,40%",options="header"]
|===
|# |Failure Mode |Likely Impact |Mitigation
|1 |Scanned PDF with degraded OCR quality |Low AFC on text-heavy fields |Cloud OCR fallback; AFC flags for review
|2 |Multi-column layouts mis-parsed |Fields assigned to wrong columns |Layout anchor consistency check in AFC
|3 |Table cells merged/split incorrectly |Numeric values from wrong cell |Table structure validation + cross-doc redundancy
|4 |Handwritten annotations on printed text |OCR garbles both layers |Fallback to cloud OCR; manual review flag
|5 |Non-standard document templates |Field extractor patterns don't match |Pattern library per document type; fallback to manual
|===

*Performance targets:* ~175,000 pages/day on single 8-vCPU / 32GB instance.
Per-page: ~0.5–2s (standard), ~3–8s (VLM).
Cloud OCR fallback adds 1–3s/page.

===== Schema Mapping Architecture

*Component:* Schema Mapping Service (<<sect-schema-mapping-service>>)

*Matching Algorithm (Hybrid Heuristic — 5 weighted signals):*

* *Name similarity (0.30):* Levenshtein distance + Jaccard token overlap, boosted by MISMO field description matching
* *HMDA/CD synonym dictionary (0.25):* Curated domain-specific equivalence table (e.g., `loan_type` ↔ `mortgage_product`, `prop_addr` ↔ `property_address`)
* *Type compatibility (0.20):* Target (MISMO) and source types must be compatible
* *Document-section context (0.15):* Boost when source field appears in expected document section (e.g., loan amount in CD page 1)
* *Historical acceptance (0.10):* Previously approved mappings for this tenant, decayed by recency

*Two-Layer Confidence Design (<<sect-two-layer-confidence>>):*

* *Layer 1 — Raw signals:* `conf_mapping`, `conf_extraction` (AFC), `conf_rule` computed independently
* *Layer 2 — Calibrated score:* Temperature scaling (Phase 1, ≥200 labeled decisions) or isotonic regression (Phase 2, ≥500 decisions); ECE must be ≤0.05

*Routing thresholds (calibrated score):*

* *≥0.95:* Auto-accept (green checkmark, ~10s visual confirmation)
* *0.70–0.95:* Human review required (~45s per field)
* *<0.70:* Manual mapping required (no suggestion shown)

*Reviewer Burden Model:*

|===
|Scenario |Fields |Override Rate |Est. Session Time
|CD-only, good extraction |~15–20 CD fields |10–15% |8–12 min
|CD-only, mixed extraction |~15–20 CD fields |20–30% |15–25 min
|Full HMDA (Phase 2) |~110 fields |15–25% |30–45 min
|===

===== Evidence Architecture

*Component:* Evidence & Attachment Service (<<sect-component-design>>)

*Chain-of-custody metadata:*

[source,typescript]
----
interface EvidenceRecord {
  evidence_id: string;
  audit_id: string;
  finding_id?: string;
  uploaded_by: userId;
  uploaded_at: timestamp;
  filename: string;
  file_size_bytes: number;
  mime_type: string;
  sha256_hash: string;

  supersedes?: evidence_id;
  full_text_indexed: boolean;
  blob_path: string;
  audit_event_id: string;
}

*Append-only semantics:* When a user "replaces" an evidence file, the system: (1) stores the new file with a new evidence_id, (2) creates a "supersedes" link from new to old, (3) marks the original as visible in history (never deleted), (4) audit trail records both the original and the replacement.

===== RBAC & Tenancy Architecture

*Components:* Auth & RBAC Service, PostgreSQL Row-Level Security (<<sect-component-design>>)

*Roles (Three-Lines-of-Defense aligned):*

* *First Line (QC):* `qc_reviewer`, `qc_manager` — loan-level data integrity testing
* *Second Line (Compliance):* `compliance_analyst`, `compliance_officer` — regulatory adherence review
* *Third Line (Internal Audit):* `auditor`, `audit_director` — independent assessment of first and second lines
* *Cross-Line:* `executive` (read-only dashboards), `it_admin` (tenant administration)

See <<sect-rbac-permission-matrix>> for the full permission matrix.

*Multi-tenant isolation:* PostgreSQL Row-Level Security (RLS) ensures that every query is automatically filtered by the authenticated user's `tenant_id`.
Cross-tenant data access is architecturally impossible at the database level.

*Line-of-defense isolation:* Each finding carries a `defense_line` tag (1, 2, or 3).
Roles within a defense line can only create, edit, and close findings tagged to their own line.
Cross-line visibility is read-only: a `compliance_analyst` can view First Line QC findings for context but cannot modify them.

*Audit trail integration:* Every access control decision — grants, denials, segregation-of-duty violations, and cross-line isolation violations — is recorded as an immutable audit event (<<sect-audit-trail-event-store>>).
`it_admin` role provides searchable, filterable, exportable access to the complete audit log.

*SSO:* SAML 2.0 and OIDC integration.
No AuditIQ-specific passwords.
Session management: 30-minute idle timeout, 8-hour absolute timeout.
