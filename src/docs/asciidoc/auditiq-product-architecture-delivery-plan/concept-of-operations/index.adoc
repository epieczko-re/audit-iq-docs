[[sect-concept-of-operations]]
== Concept of Operations (ConOps)

AuditIQ supports three distinct audit modes — manual, OCR-assisted, and fully automated pre-scan — because different audits demand different levels of human involvement.
A reviewer manually comparing a Closing Disclosure against a HMDA LAR record needs a different workflow than a system automatically validating 5,000 structured records against a LOS extract.
The platform adapts; the reviewer stays in control.

=== Three-Lines-of-Defense Operating Model

AuditIQ serves all three lines of defense from a single platform — but it does not merge them.
Each line maintains independent judgment, independent conclusions, and an independent audit trail, consistent with the IIA Three Lines Model (see <<sect-problem-statement>>).
What the platform shares is the _evidence base_: source documents, extracted data, confidence scores, and the immutable event log.

The following table shows how each line interacts with the platform differently:

[cols="18%,27%,27%,28%",options="header"]
|===
|Dimension |First Line (QC) |Second Line (Compliance) |Third Line (Internal Audit)

|*Primary question*
|"Does this loan file match its source documents?"
|"Are our origination practices adhering to HMDA reporting standards?"
|"Are the first and second lines performing their functions effectively?"

|*Typical workflow*
|Record-by-record field comparison (Manual or OCR-Assisted mode)
|Sample-based thematic review using Automated Pre-Scan; policy-overlay analysis
|Independent re-review of first-line work product; control testing across audit runs

|*Evidence access*
|Full access to assigned loan files and source documents
|Read access to all QC findings and evidence; write access to compliance-layer findings
|Read access to all QC and compliance findings; write access to audit-layer findings

|*Finding authority*
|Creates QC findings; cannot modify compliance or audit findings
|Creates compliance findings; can reference but not alter QC findings
|Creates audit findings; can reference but not alter QC or compliance findings

|*Approval path*
|QC Manager approves closure of QC findings
|Compliance Officer approves closure of compliance findings
|Audit Director approves closure of audit findings

|*RBAC role(s)*
|`qc_reviewer`, `qc_manager`
|`compliance_analyst`, `compliance_officer`
|`auditor`, `audit_director`
|===

This separation is enforced at the RBAC level: a user with `qc_reviewer` permissions cannot create or modify compliance-layer findings, and vice versa.
The exam pack dossier (<<sect-exam-ready-bundle>>) includes all three layers — with a structured three-lines-of-defense certification artifact — allowing examiners to see how each line independently evaluated the same evidence.

=== Two-Stage Testing Model (HMDA)

The FFIEC Examiner Transaction Testing Guidelines prescribe a two-stage methodology that AuditIQ mirrors in its workflow design:

* *Stage 1 — Data Integrity Testing:* Compare HMDA LAR fields against source documents to verify accuracy.
The HMDA audit pack defines 37 Key Data Fields (<<sect-glossary>>) as the comparison target.
This is the "stare-and-compare" work that AuditIQ automates — field-by-field comparison with confidence-scored results.
* *Stage 2 — Substantive / Transactional Testing:* Evaluate whether reported data reflects the underlying transaction correctly.
Stage 2 findings require professional judgment — for example, determining whether an action-taken code accurately reflects the disposition of an application given its full history.
AuditIQ surfaces the evidence and flags anomalies; the reviewer applies judgment.

When Stage 1 error rates breach the Initial Sample Threshold, examiners expand to the full sample.
AuditIQ tracks Stage 1 error rates in real time and alerts the reviewer when the threshold is approached, enabling proactive sample expansion before an examiner demands it.

[[sect-finding-remediation]]
=== Finding Remediation Lifecycle

Once findings are generated (in any workflow mode), each finding follows a remediation lifecycle from identification through closure.
This lifecycle is orthogonal to the audit run state machine — an audit run may be in `FindingsGenerated` while individual findings are in different remediation states.

*Actors:* Rules Engine (creation), Reviewer (triage + retest), Remediation Owner (correction), Manager (escalation + closure approval)

*States:*

[arabic]
. *Identified* — Finding created by rules engine or reviewer; AI narrative drafted (if applicable)
. *Triaged* — Reviewer confirms severity and assigns remediation owner; AI-generated findings must pass effective challenge before triage
. *Remediation Assigned* — Owner notified; corrective action expected within SLA (Critical: 1 day, High: 3 days, Medium: 7 days, Low: 14 days)
. *Corrected* — Owner marks finding as corrected and attaches evidence of remediation
. *Retest Passed* — Reviewer independently verifies correction against source documents; finding cleared
. *Retest Failed* — Reviewer determines correction is insufficient; finding returns to Remediation Assigned with additional notes
. *Closed* — Manager approves closure (required for Critical/High findings); finding is finalized and immutable
. *Escalated* — Manager escalates finding (e.g., pattern detected across multiple loans); triggers additional review or regulatory notification workflow
. *Reopened* — Finding reopened via compensating event (<<sect-immutability>>); original finding record preserved; new event captures reopening rationale

*Approval gates:*

* *Triage → Remediation Assigned:* Reviewer must complete effective challenge for AI-generated findings
* *Corrected → Retest Passed:* Reviewer (different from remediation owner) must independently verify
* *Retest Passed → Closed:* Manager approval required for Critical and High severity findings
* *Closed → Reopened:* Manager or Compliance Officer only; compensating event required with documented rationale

*Audit trail:* Every state transition emits an immutable audit event with `actor_id`, `timestamp`, `disposition`, and `rationale`.
The complete finding remediation history is included in the exam pack dossier (<<sect-exam-ready-bundle>>).

=== Exam Pack Dossier: Continuous Update Model

The exam pack is not generated at the end of the audit — it is a *living dossier* that the platform maintains from the moment the audit begins.
This operational model ensures the institution is always examination-ready, not just at finalization.

*How the dossier evolves through the audit lifecycle:*

[arabic]
. *Audit created (v1.0):* Bundle initialized with audit metadata, pack configuration snapshot, and sampling methodology.
Coverage statistics show 0% tested; no findings.
. *Pre-scan complete (v2.0):* Findings register populated from automated pre-scan.
Coverage statistics show rules tested, confidence distributions, and error rate.
Approval manifest is empty — no reviews yet.
. *QC review in progress (v2.x):* Each finding triage, closure, or escalation triggers a minor version bump.
The change log tracks every status transition.
Exceptions log captures any reviewer overrides of confidence or severity.
. *First-line approval (major bump):* QC Manager reviews all findings and records approval.
Approval manifest shows first-line sign-off with scope, findings reviewed, and digital signature.
. *Compliance review (major bump):* Compliance Officer reviews all findings, adds compliance-layer observations.
Second-line approval recorded with independent assessment.
. *Audit Director review (major bump):* Third-line independent assurance sample reviewed.
Three-lines-of-defense certification becomes complete.
`all_lines_approved` flag set to true.
. *Export to examiner:* Bundle exported as ZIP with all 14 artifacts.
Export event recorded with hash and recipient.
If findings change after export, a new version is created and the previous version is marked `superseded`.
The version chain provides full traceability.

*Post-export updates:* If a finding is reopened or a corrective action completes after the initial export, the Exam Pack Lifecycle Manager automatically creates a new bundle version.
Examiners (or the institution) can request a diff between any two versions to see exactly what changed.
This eliminates the risk of examiners reviewing stale data.

[[sect-remediation-extensibility]]
==== Platform Extensibility: Complaint Management and Beyond

The 9-state remediation lifecycle is intentionally designed as a generic workflow primitive — not an audit-specific construct.
The state machine (Identified → Triaged → Assigned → Corrected → Retested → Closed), the SLA-driven due dates, the severity classification, the evidence attachments, and the immutable event trail are structurally identical to what a complaint management system, a corrective action tracker, or a regulatory response workflow requires.

*Finding origin extensibility:* Every finding is tagged with an `origin` field indicating what triggered it (see <<sect-effective-rule-merge>>).
The MVP defines four origins: `regulatory`, `policy_tightened`, `policy_supplemental`, and `procedural_checkpoint`.
A fifth origin — `complaint` — is reserved for future complaint management integration.
This enables bi-directional linking: a finding can trigger a complaint (e.g., pattern of HMDA errors affecting borrowers suggests a fair lending issue), and a complaint can trigger an audit (e.g., customer alleges rate discrimination, triggering a targeted HMDA review using existing packs).

*Architectural implication:* Because the remediation lifecycle, audit trail, evidence store, and approval workflow are all entity-type-agnostic (see <<sect-policy-documents-table>> and the event store's extensible `entity_type` discriminator), extending to complaint management requires new UI screens and a Complaint Pack — not new backend infrastructure.
The same Workflow Engine, Event Store, and RBAC model serve both audit findings and complaint records.
