== Concept of Operations (ConOps)

AuditIQ supports three distinct audit modes — manual, OCR-assisted, and fully automated pre-scan — because different audits demand different levels of human involvement.
A reviewer manually comparing a Closing Disclosure against a HMDA LAR record needs a different workflow than a system automatically validating 5,000 structured records against a LOS extract.
The platform adapts; the reviewer stays in control.

=== Audit Workflow: Manual Mode

[arabic]
. *Setup:* Admin creates an audit run, selects the audit pack (e.g., HMDA), uploads or links the target dataset.
. *Ingestion:* System validates the target dataset schema against the pack definition, flags validation errors.
. *Document linking:* Reviewer or batch process associates source documents (PDFs, images) with target records.
. *Review:* Reviewer opens a record in the side-by-side viewer: target data fields on the left, source document on the right. Reviewer compares each field, marks as confirmed/discrepancy/unable to determine.
. *Finding creation:* When a discrepancy is identified, reviewer creates a finding with: correct source value, root cause selection, severity (auto-suggested by pack rules, overridable), and comment.
. *Remediation:* Findings route to responsible parties. Corrective actions are tracked with due dates and evidence attachments.
. *Second review:* Configurable sampling of completed reviews for quality assurance.
. *Reporting:* Manager reviews dashboards, generates audit summary report, exports exam-ready bundle.

  Admin->>System: Upload target dataset

=== Audit Workflow: OCR-Assisted Mode (Phase 1+)

Steps 1–3 as above, then:

[arabic, start=4]
. *OCR extraction:* Document processing pipeline extracts text, tables, and key-value pairs from source documents. Each extracted value carries an extraction confidence score.
. *Automated pre-scan:* Comparison engine applies pack rules to match target values against extracted source values. Generates preliminary findings with combined confidence scores (extraction × rule application).
. *Human review:* Reviewer sees pre-populated comparison results. High-confidence matches are pre-confirmed (reviewer can override). Low/medium-confidence items are flagged for manual verification. Reviewer focuses time on genuinely uncertain cases.
. Steps 6–8 as manual mode.

=== Finding Remediation Lifecycle

Once findings are generated (in any workflow mode), each finding follows a remediation lifecycle from identification through closure.
This lifecycle is orthogonal to the audit run state machine — an audit run may be in `FindingsGenerated` while individual findings are in different remediation states.

*Actors:* Rules Engine (creation), Reviewer (triage + retest), Remediation Owner (correction), Manager (escalation + closure approval)

*States:*

[arabic]
. *Identified* — Finding created by rules engine or reviewer; AI narrative drafted (if applicable)
. *Triaged* — Reviewer confirms severity and assigns remediation owner; AI-generated findings must pass effective challenge before triage
. *Remediation Assigned* — Owner notified; corrective action expected within SLA (Critical: 1 day, High: 3 days, Medium: 7 days, Low: 14 days)
. *Corrected* — Owner marks finding as corrected and attaches evidence of remediation
. *Retest Passed* — Reviewer independently verifies correction against source documents; finding cleared
. *Retest Failed* — Reviewer determines correction is insufficient; finding returns to Remediation Assigned with additional notes
. *Closed* — Manager approves closure (required for Critical/High findings); finding is finalized and immutable
. *Escalated* — Manager escalates finding (e.g., pattern detected across multiple loans); triggers additional review or regulatory notification workflow
. *Reopened* — Finding reopened via compensating event (<<sect-immutability>>); original finding record preserved; new event captures reopening rationale

*Approval gates:*

* *Triage → Remediation Assigned:* Reviewer must complete effective challenge for AI-generated findings
* *Corrected → Retest Passed:* Reviewer (different from remediation owner) must independently verify
* *Retest Passed → Closed:* Manager approval required for Critical and High severity findings
* *Closed → Reopened:* Manager or Compliance Officer only; compensating event required with documented rationale

[mermaid]
----

  Triaged --> RemediationAssigned: Owner notified

*Audit trail:* Every state transition emits an immutable audit event with `actor_id`, `timestamp`, `disposition`, and `rationale`.
The complete finding remediation history is included in the exam-ready bundle (<<sect-exam-ready-bundle>>).

=== Audit Workflow: Automated Pre-Scan (“Target-to-Source” / “Record-to-Record”)

For structured data audits (e.g., HMDA LAR against LOS database extract), the full audit can be automated:

[arabic]
. Both target and source datasets are uploaded as structured data (CSV, API).
. Schema mapping applies (auto-suggested, human-approved).
. Comparison engine runs all pack rules automatically.
. Output: full findings register with confidence scores.
. Human review is required for: all Critical/High findings, all findings below confidence threshold, all AI-generated narratives.

.Automated Pre-Scan Workflow
[mermaid]
----
sequenceDiagram
  participant Admin
  participant System
  participant CompEngine as Comparison Engine
  participant Reviewer
  participant Manager

  Admin->>System: Upload target dataset (CSV/API)
  Admin->>System: Upload source dataset (CSV/API)
  System->>System: Auto-suggest schema mapping
  Admin->>System: Approve schema mapping
  System->>CompEngine: Run all pack rules automatically
  CompEngine->>System: Full findings register + confidence scores
  System->>Reviewer: Critical/High findings for review
  System->>Reviewer: Below-threshold findings for review
  Reviewer->>System: Approve / override findings
  System->>Manager: Escalated findings
  Manager->>System: Final approval
  Manager->>System: Generate report
----
