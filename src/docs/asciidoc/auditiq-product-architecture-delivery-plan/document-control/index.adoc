=== How to Use This Document

This document is organized into seven parts:

* *Part I — Strategic Context:* Executive Summary, Problem Statement, Product Vision, Competitive Intelligence.
* *Part II — Operational Concept:* Concept of Operations (ConOps) — audit modes, workflow states, finding lifecycle, PII redaction.
* *Part III — Requirements & Specifications:* Product Requirements, Audit Pack Specification, Regulatory Compliance & Policy Management.
* *Part IV — Architecture & Design:* Architecture Overview (including Key Concepts), Component-Level Design, Data Model, Scoring Framework.
* *Part V — AI & Governance:* AI Governance & Model Risk Controls, Model Registry, Model Deployment, Docling-Graph Integration, Immutability vs. Correctability.
* *Part VI — Delivery & Business:* Pricing & Packaging, Delivery Plan, Risks & Mitigations, Future Roadmap, UI Workflows & User Journeys.
* *Part VII — Reference:* Architectural Decisions (including MVP Decisions Locked), Exam-Ready Bundle Specification, Canonical Glossary, Appendices.

This document serves three audiences simultaneously:

* *Engineering team:* Sprint-plan against Parts III and IV (<<sect-product-requirements, Product Requirements>>, <<sect-audit-pack-spec, Audit Pack Specification>>, <<sect-architecture-overview, Architecture Overview>>, <<sect-component-design, Component-Level Design>>, <<sect-data-model, Data Model>>).
Part V (<<sect-ai-governance, AI Governance>>, <<sect-model-registry, Model Registry>>, <<sect-model-deployment, Model Deployment>>, <<sect-docling-graph, Docling-Graph>>, <<sect-immutability, Immutability>>) provides AI/ML architecture.
Each section contains explicit contracts, schemas, and acceptance criteria that can be decomposed into user stories and tasks.
* *Product / leadership:* Read Part I for context and competitive positioning.
Review Part VI (<<sect-pricing, Pricing>>, <<sect-delivery-plan, Delivery Plan>>) and Part VII (<<sect-decisions, Decisions>>) for investment decisions.
<<sect-risks-constraints, Risks, Constraints, and Mitigations>> provides risk context.
* *Compliance / regulatory stakeholders:* Focus on Part V (<<sect-ai-governance, AI Governance>>, <<sect-immutability, Immutability>>) and Part VII (<<sect-exam-ready-bundle, Exam-Ready Bundle>>, <<sect-glossary, Canonical Glossary>>) for regulatory defensibility.
<<sect-regulatory-compliance, Regulatory Compliance & Policy Management>> specifies the five-layer compliance hierarchy.

*Canonical Glossary:* All confidence metrics (AFC, mapping_confidence, extraction_confidence, rule_confidence, combined_confidence), routing thresholds, severity definitions, and calibration criteria are defined in a *single source of truth*: the *<<sect-glossary, Canonical Glossary>>*.
If a threshold value elsewhere in this document conflicts with the Glossary, the Glossary governs.

*Conventions used in this document:*

* *TODO* = work item that must be completed before implementation starts
* *ASSUMPTION* = a belief stated explicitly; risk if wrong is documented
* *DECISION NEEDED* = requires stakeholder input before proceeding
* *DECISION LOCKED* = decided for MVP; revisit criteria documented in <<sect-mvp-decisions>>
* Reference documents are listed in <<sect-appendices, Appendix E>>.
