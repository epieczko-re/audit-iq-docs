=== How to Use This Document

This document serves three audiences simultaneously:

* *Engineering team:* Sprint-plan against <<sect-key-concepts, Key Architectural Concepts>>, <<sect-product-requirements, Product Requirements>>, <<sect-audit-pack-spec, Audit Pack Specification>>, <<sect-architecture-overview, Architecture Overview>>, <<sect-component-design, Component-Level Design>>, and <<sect-data-model, Data Model>>.
Supplementary architecture sections include <<sect-mvp-decisions, MVP Decisions Locked>>, <<sect-immutability, Immutability vs. Correctability>>, <<sect-exam-ready-bundle, Exam-Ready Bundle>>, <<sect-ai-governance, AI Governance & Model Risk Controls>>, <<sect-ui-workflows, UI Workflows & User Journeys>>, <<sect-model-registry, Model Registry>>, <<sect-model-deployment, Model Deployment>>, and <<sect-docling-graph, Docling-Graph Integration>>.
Each section contains explicit contracts, schemas, and acceptance criteria that can be decomposed into user stories and tasks.
* *Product / leadership:* Read <<sect-document-control, Document Control>> through <<sect-concept-of-operations, Concept of Operations>> for context and competitive positioning.
Review <<sect-pricing, Pricing and Packaging>>, <<sect-delivery-plan, Delivery Plan>>, <<sect-mvp-decisions, MVP Decisions Locked>>, and the MVP Cutline for investment decisions.
<<sect-risks-constraints, Risks, Constraints, and Mitigations>> and <<sect-decisions, Decisions and Open Questions>> provide risk and decision context.
* *Compliance / regulatory stakeholders:* Focus on <<sect-scoring-framework, Scoring Framework>>, <<sect-immutability, Immutability vs. Correctability>>, <<sect-exam-ready-bundle, Exam-Ready Bundle>>, and <<sect-ai-governance, AI Governance & Model Risk Controls>> for regulatory defensibility.
<<sect-model-registry, Model Registry>> and the <<sect-appendices, Glossary>> provide supporting reference material.

*Canonical Glossary:* All confidence metrics (AFC, mapping_confidence, extraction_confidence, rule_confidence, combined_confidence), routing thresholds, severity definitions, and calibration criteria are defined in a *single source of truth*: the *Canonical Glossary* (<<sect-appendices, Appendix A>>).
If a threshold value elsewhere in this document conflicts with the Glossary, the Glossary governs.

*Conventions used in this document:*

* *TODO* = work item that must be completed before implementation starts
* *ASSUMPTION* = a belief stated explicitly; risk if wrong is documented
* *DECISION NEEDED* = requires stakeholder input before proceeding
* *DECISION LOCKED* = decided for MVP; revisit criteria documented in <<sect-mvp-decisions>>
* Reference documents are listed in <<sect-appendices, Appendix E>>.
