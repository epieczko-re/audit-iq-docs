==== Happy Path

[arabic]
. Finding enters `Identified` state → reviewer triages (confirms severity, assigns remediation owner).
. Owner receives notification → submits corrective action with evidence within SLA.
. Independent reviewer verifies correction → retest passed → manager approves closure (required for Critical/High).
. All findings closed → manager generates audit summary report.
. Exam-ready bundle exported as versioned ZIP (14 artifacts) → Three Lines Model certification.

==== Key Edge Cases

* *SLA breach:* Remediation owner misses SLA → aging dashboard alerts manager → automatic escalation.
* *Retest failed:* Reviewer determines correction insufficient → finding returns to `Remediation Assigned` with notes.
* *Post-export update:* Finding reopened after bundle export → Exam Pack Lifecycle Manager auto-creates new bundle version; previous version marked `superseded`.

==== Architecture

===== Evidence Chain-of-Custody Workflow

[mermaid]
----
sequenceDiagram
    participant Reviewer
    participant System
    participant BlobStore as Azure Blob Storage
    participant EventStore as Event Store

    Reviewer->>System: Attach evidence to finding
    System->>System: Compute SHA-256 hash
    System->>System: Classify document (CD, LE, URLA, etc.)
    System->>BlobStore: Store file (append-only, immutable)
    System->>EventStore: EvidenceCreated {evidence_id, finding_id, hash, uploader, timestamp}
    System-->>Reviewer: Evidence attached with chain-of-custody metadata

    Note over Reviewer: Later — reviewer uploads replacement

    Reviewer->>System: Upload replacement evidence
    System->>System: Compute SHA-256 hash for new file
    System->>BlobStore: Store new file (new evidence_id)
    System->>System: Create "supersedes" link (new → original)
    System->>EventStore: EvidenceSuperseded {new_id, original_id, reason}
    Note over BlobStore: Original file remains unchanged and visible in history
----

===== Evidence Architecture

*Component:* Evidence & Attachment Service (<<sect-component-design>>)

*Chain-of-custody metadata:*

[source,typescript]
----
interface EvidenceRecord {
  evidence_id: string;
  audit_id: string;
  finding_id?: string;
  uploaded_by: userId;
  uploaded_at: timestamp;
  filename: string;
  file_size_bytes: number;
  mime_type: string;
  sha256_hash: string;
  document_classification: 'closing_disclosure' | 'loan_estimate' | 'urla' | 'other';
  superseded_by?: evidence_id;
  supersedes?: evidence_id;
  full_text_indexed: boolean;
  blob_path: string;
  audit_event_id: string;
}
----

*Append-only semantics:* When a user "replaces" an evidence file, the system: (1) stores the new file with a new evidence_id, (2) creates a "supersedes" link from new to old, (3) marks the original as visible in history (never deleted), (4) audit trail records both the original and the replacement.
See <<sect-immutability>> for the full immutability/correctability specification.
