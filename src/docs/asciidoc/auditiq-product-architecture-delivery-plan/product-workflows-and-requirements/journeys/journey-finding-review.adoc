=== Journey: Finding Review & Approval

This journey covers rules engine execution, finding generation, AI narrative creation, effective challenge review, and manager escalation.
The comparison engine applies pack rules against mapped data; the LLM drafts finding narratives; reviewers exercise effective challenge.

==== End-to-End Workflow Diagram

===== Findings Review & Approval (AI Governance Controls)

This workflow details the human oversight process for AI-generated findings, enforcing the AI governance red lines from <<sect-ai-governance>>.

*Actors:* Rules Engine, AI (LLM), Reviewer (effective challenge) *Trigger:* Rules engine generates exceptions from scored data *End State:* Finding finalized with full provenance and effective challenge documentation

*RED LINES (<<sect-ai-governance>>):*

* No AI output enters the official audit record without explicit human approval
* AI output must never be represented as definitive conclusions
* Reviewer is responsible for accuracy of any approved finding

*Required Disclaimers (<<sect-ai-governance>>):*

* "This content was drafted by an AI assistant and requires human verification before use"
* "AI-generated content may contain inaccuracies — reviewer is responsible for verifying accuracy"
* "This finding has been reviewed and approved by [reviewer name] on [date]"

*Design Reference:* Findings Management, Reporting Service, <<sect-ai-governance>>, <<sect-decision-d1, Decision D1>>–<<sect-decision-d3, D3>>

==== Happy Path

[arabic]
. Rules engine applies pack rules against mapped data for all 37 Key Data Fields → generates preliminary findings with combined confidence scores.
. LLM drafts finding narratives with citations → AI provenance logged (21-field schema).
. Findings enter review queue sorted by severity and confidence.
. Reviewer opens each finding → exercises effective challenge (SR 11-7) → approves, edits, or rejects.
. Critical/High findings auto-escalate to manager → manager approves or requests re-review.
. Approved findings enter remediation lifecycle.

==== Key Edge Cases

* *False positive:* Reviewer rejects AI-generated finding → must provide rationale → finding marked as false positive with audit trail event.
* *Stage 1 error rate breach:* Error rate approaches pack-defined threshold (e.g., 5% for HMDA) → system displays threshold proximity alert → proactive sample expansion.
* *Automation bias:* Reviewer approves without effective challenge → system enforces mandatory challenge documentation; randomized challenge prompts (R26 mitigation).

*Deterministic boundary:* The comparison engine is fully deterministic.
Given identical normalized inputs and pack rules, it always produces identical findings.
AI-generated narratives, suggestions, and advisory outputs are explicitly non-deterministic and always labeled.

|FR-6.2 |Each finding shall capture: root cause, corrective action, owner, due date, evidence references, and all status transitions. |Given a finding with corrective action assigned, when viewed, then all metadata fields are populated and the complete transition history is visible in the audit trail. |P0

===== Finding Lifecycle & Escalation Architecture

*Component:* Findings Management Service (<<sect-component-design>>)

*Finding lifecycle state machine:*

....
IDENTIFIED
  ↓ (reviewer triages; AI findings must pass effective challenge)
TRIAGED
  ↓ (root cause identified, owner assigned)
REMEDIATION_ASSIGNED  ←──────────────────┐
  ↓ (owner corrects; attaches evidence)  │
CORRECTED                                │
  ↓ (reviewer retests)                   │
  ├── RETEST_PASSED                      │
  │     ↓ (manager approves closure)     │
  │   CLOSED (immutable)                 │
  └── RETEST_FAILED ─────────────────────┘
        (returns with additional notes)

(Any state → ESCALATED: manager escalates for pattern review)
(CLOSED → REOPENED: compensating event; original finding preserved)
....

*Finding lifecycle event sequence:*

[mermaid]
----
sequenceDiagram
    participant Engine as Comparison Engine
    participant Store as Event Store
    participant DB as State DB
    participant Reviewer
    participant Manager

    Engine->>Store: FindingCreated {id, record, field, severity, confidence}
    Engine->>DB: INSERT finding (status=IDENTIFIED)

    Reviewer->>Store: FindingTriaged {id, reviewer, root_cause}
    Reviewer->>DB: UPDATE finding (status=TRIAGED)

    Reviewer->>Store: RemediationAssigned {id, owner, due_date}
    Reviewer->>DB: UPDATE finding (status=REMEDIATION_ASSIGNED)

    Note over Reviewer: Owner corrects the issue

    Reviewer->>Store: FindingCorrected {id, corrective_action, evidence_refs}
    Reviewer->>DB: UPDATE finding (status=CORRECTED)

    Manager->>Store: FindingRetested {id, retest_result, retester}
    Manager->>DB: UPDATE finding (status=RETEST_PASSED or RETEST_FAILED)

    Manager->>Store: FindingClosed {id, resolution, approver}
    Manager->>DB: UPDATE finding (status=CLOSED)
