[[journey-extraction-mapping]]
=== Journey: Extraction & Mapping

This journey covers document processing through Docling-Graph, AFC scoring, field extraction review, schema mapping, and synonym dictionary management.
The system extracts structured data from source documents and maps extracted fields to the HMDA/MISMO target schema.

==== End-to-End Workflow Diagrams

===== Schema Mapping Review (Two-Layer Confidence)

This workflow details how extracted fields are mapped to the HMDA/MISMO target schema with AI assistance and human oversight.

*Actors:* System (matcher), AI (calibration), Reviewer (approval) *Trigger:* Extracted fields arrive from Document Processing Service *End State:* All fields mapped to target schema; approved mappings feed into rules engine

Docling extracts data from source documents and the comparison engine generates preliminary findings, with human review focused on uncertain cases.

*Actors:* Admin (setup), System (Docling, Comparison Engine), Reviewer (validation + findings), Manager (approval + reporting)
*Trigger:* Admin creates a new audit run in OCR-Assisted execution mode

==== Happy Path

[arabic]
. System processes uploaded documents through Docling-Graph pipeline → extracts text, tables, key-value pairs.
. AFC v2 scoring (6 signals including `graph_corrob_score`) routes each field: >=0.85 auto-confirm, 0.50-0.84 human review, <0.50 manual entry.
. Reviewer validates flagged fields in split-pane extraction review → accepts or overrides extracted values.
. Schema mapping service suggests field mappings with two-layer confidence → reviewer approves mapping set.
. Approved mappings feed into rules engine.

==== Key Edge Cases

* *Low-quality scan:* Page grade POOR/FAIR → 3-tier fallback: VLM → LLM (Mistral via LiteLLM) → cloud OCR (Azure).
Each fallback logged with `extraction_method` provenance.
* *CD-only coverage gap:* With CD-only documents, expect ~10-20% of HMDA fields at >=0.85 confidence.
Full coverage requires 1003, appraisal, and title documents (Phase 2).
* *High override rate:* If override rate >25%, triggers confidence recalibration alert (<<sect-glossary>>).

*Deterministic boundary:* OCR extraction is deterministic for a given model version and input.
The same document processed with the same model version always produces the same extraction.
Model version is logged as provenance metadata.

*DECISION LOCKED:* Docling-Graph pipeline as baseline extraction engine (MIT-licensed); self-hosted Mistral via LiteLLM as LLM extraction backend; cloud OCR as page-level fallback.

===== Extraction Architecture

*Component:* Document Processing Service (<<sect-doc-processing>>)

*3-Tier Fallback Chain:*

....
for each document:
  // Step 1: Attempt Docling-Graph structured extraction (VLM backend)
  context = docling.convertDocument(document)
  graph_result = docling_graph.run_pipeline(
    context, template=pydantic_template, backend="vlm"
  )
  compute afc including graph_corrob_score

  if graph_result.success and afc >= 0.50:
    persist extraction (Pydantic objects + graph)
    recordExtractionEvent(auditId, documentId, 'docling_graph_vlm')
  else:
    // Step 2: Attempt LLM extraction via LiteLLM (self-hosted Mistral/vLLM)
    llm_result = docling_graph.run_pipeline(
      context, template=pydantic_template, backend="llm",
      provider="litellm", provider_override="mistral"
    )
    recompute afc

    if llm_result.success and afc >= 0.50:
      persist extraction (Pydantic objects + graph)
      recordExtractionEvent(auditId, documentId, 'docling_graph_llm_mistral')
    else:
      // Step 3: Fall back to cloud OCR or manual entry
      triggerCloudOCRFallback(document)
      recordFallbackEvent(auditId, documentId, 'cloud_ocr_fallback')
    end
  end
end
....

[cols="10%,25%,25%,40%",options="header"]

|# |Failure Mode |Likely Impact |Mitigation

===== Schema Mapping Architecture

*Component:* Schema Mapping Service (<<sect-schema-mapping-service>>)

*Matching Algorithm (Hybrid Heuristic — 5 weighted signals):*

* *Name similarity (0.30):* Levenshtein distance + Jaccard token overlap, boosted by MISMO field description matching
* *HMDA/CD synonym dictionary (0.25):* Curated domain-specific equivalence table (e.g., `loan_type` ↔ `mortgage_product`, `prop_addr` ↔ `property_address`)
* *Type compatibility (0.20):* Target (MISMO) and source types must be compatible
* *Document-section context (0.15):* Boost when source field appears in expected document section (e.g., loan amount in CD page 1)
* *Historical acceptance (0.10):* Previously approved mappings for this tenant, decayed by recency

*Two-Layer Confidence Design (<<sect-two-layer-confidence>>):*

* *Layer 1 — Raw signals:* `conf_mapping`, `conf_extraction` (AFC), `conf_rule` computed independently
* *Layer 2 — Calibrated score:* Temperature scaling (Phase 1, ≥200 labeled decisions) or isotonic regression (Phase 2, ≥500 decisions); ECE must be ≤0.05

*Routing thresholds (calibrated score):*

* *≥0.95:* Auto-accept (green checkmark, ~10s visual confirmation)
* *0.70–0.95:* Human review required (~45s per field)
* *<0.70:* Manual mapping required (no suggestion shown)
