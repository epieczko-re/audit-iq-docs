==== Happy Path

[arabic]
. IT Admin configures SSO/SAML integration → creates users with defense line and role assignments → system enforces one-user-per-defense-line isolation.
. IT Admin sets confidence thresholds and SLA timelines in System Settings → changes logged to audit trail with before/after values.
. Manager creates a new audit pack (Draft) → defines rules, scoring weights, and evidence requirements → submits for test harness validation.
. Test harness executes against sample data → validation passes → Manager publishes pack → pack available for new audits; existing audits remain pinned to prior version.
. Model Risk Officer registers a new AI/ML model → uploads performance metrics → reviews against acceptance criteria → approves model → model available at call-time.
. IT Admin reviews audit trail for recent administrative activity → filters by event type and date range → confirms no anomalies.

==== Key Edge Cases

* *Pack validation failure:* Test harness detects unexpected findings or missing rules → Manager receives failure report with specifics → revises pack and resubmits; publication blocked until validation passes.
* *No approved model at call-time:* All models for a capability are deprecated or in Draft/Staging → system refuses AI calls (fail-closed gating) → Manager sees error "No approved model for extraction"; audits requiring that capability are blocked.
* *Cross-line role assignment attempt:* IT Admin attempts to assign a user to multiple defense lines → system enforces single-line constraint and rejects the change with "One user = one defense line" validation error.
* *Segregation of duties violation:* A user who created a finding attempts to close it → system blocks with "Segregation of duties violation" and logs the attempt as an immutable audit event.
* *Model deprecation with active audits:* Model Risk Officer deprecates a model version while audits are in progress → active audits continue with the current model; only new audits use the latest approved version.

===== Pack Architecture

*Component:* Audit Pack Manager (<<sect-audit-pack-spec>>)

*Pack contents:* A declarative JSON specification defining rules, scoring weights, evidence requirements, field normalizations, sampling methodology, and report templates for a compliance audit category.
Not executable code — the platform interprets the pack to drive ingestion validation, schema mapping, comparison rules, scoring, and report generation.

*Phase 1:* One pack — HMDA Origination QC (pre-built by RiskExec). *Phase 2:* CRA, SBL (Section 1071). *Future:* Client-customizable packs.

*Version format:* Semantic Versioning (MAJOR.MINOR.PATCH)

* *MAJOR:* Breaking changes to schemas or rule logic (requires migration)
* *MINOR:* New rules or schemas added (backward-compatible)
* *PATCH:* Bug fixes, typo corrections (no functional changes)

See <<sect-pack-manifest-schema>> for the full Pack Manifest JSON Schema and <<sect-audit-pack-spec>> for the complete pack specification.

===== Pack Authoring Model

RiskExec exclusively authors and maintains all packs.
Clients configure audit parameters (sample size, severity overrides) but do not create packs.
This is a deliberate product decision: pack quality is a platform obligation, and regulatory compliance packs require deep domain expertise that most client compliance teams cannot maintain.

*Step 1 -- Regulatory Analysis:* Compliance SME reads the regulation (e.g., 12 CFR 1003 for HMDA), identifies every auditable requirement, and documents field-level requirements with citations.
AI-assisted gap detection (see <<sect-pack-ai-boundary>>) can accelerate this step.

*Step 2 -- Field Coverage Matrix:* For each target field, document which source documents contain the value, the expected comparison type, and the severity of a mismatch.
This becomes the pack's coverage matrix (see <<sect-hmda-coverage-matrix>>).

*Step 3 -- Rule Authoring:* Pack engineer translates the coverage matrix into comparison rules in the pack JSON schema.
Complex conditions use CEL expressions.
Every rule includes a `regulation_citation`.

*Step 4 -- Test Case Creation:* For each rule, create at minimum: one passing test case (values match), one failing test case (values mismatch), and one edge case (missing data, boundary values, format variations).
Golden datasets are constructed from anonymized production loan files.

*Step 5 -- Validation & Testing:* Run the pack through the validation harness (see <<sect-pack-testing>>).
All test cases must pass.
Rule coverage must be >=90%.

*Step 6 -- Compliance Review:* An independent compliance reviewer (not the original author) reviews all rules for correctness, completeness, and regulatory alignment.
This is a mandatory gate before publication.

*Step 7 -- Publication:* Pack transitions from `draft` to `published`.
Existing audit runs are not affected (version pinning).
New audit runs use the latest published version.

*Step 8 -- Institution Policy Profile Intake (per-institution):* For each institution deploying the pack, the AI-driven policy profile creation pipeline (see <<sect-ai-policy-pipeline>>) is executed.
The institution uploads their relevant policy and procedure documents, AI generates a draft profile, and the dual-approval review process produces a published Institution Policy Profile.
This step is performed once per institution per pack, and repeated when the institution updates its policies or the base pack receives a MAJOR version update.

===== Pack Lifecycle States

.Pack Lifecycle State Machine
[mermaid]
----
stateDiagram-v2
    [*] --> Draft : Pack created
    Draft --> Published : Compliance review passed
    Draft --> Draft : Test failures / review feedback
    Published --> Deprecated : New version published OR regulation superseded
    Published --> Published : Patch update (bug fix)
    Deprecated --> Retired : Grace period expired
    Retired --> [*]
----

* *Draft:* Created by pack author, not available to audit runs.
Undergoes iterative testing and compliance review.
* *Published:* Available for new audit runs.
Existing runs remain pinned to their original version.
Only one version per pack_id can be in `published` state at a time.
* *Deprecated:* Marked for removal with migration path.
No new runs can be created.
Existing runs continue uninterrupted.
The `deprecation` object in the manifest specifies the replacement pack and reason.
Grace period: 90 days from deprecation date.
* *Retired:* No longer available.
Existing runs reference the archived pack version (read-only) for reproducibility.

When regulations change, pack updates are a platform obligation with SLA-backed timelines.

*Regulatory monitoring:*

RiskExec monitors the following sources for regulatory changes affecting active packs:

* Federal Register publications (CFPB final rules, proposed rules, amendments)
* CFPB guidance and bulletins
* FFIEC examination procedures updates
* State regulatory agency publications (for state-specific pack overlays)
* MISMO Reference Model version releases

*Update SLA:*

[cols="25%,25%,50%",options="header"]
|===
|Change Type |SLA |Example

|*Critical* (enforcement-impacting)
|Updated pack published within 30 days of effective date
|New HMDA data point added by final rule effective January 1

|*Standard* (annual filing changes)
|Updated pack published 60 days before filing season opens
|Annual HMDA LAR field changes (e.g., 2026 filing season pack ready by November 1, 2025)

|*Minor* (guidance clarifications)
|Updated pack published within 90 days
|CFPB FAQ clarifying property address formatting requirements
|===

*Version migration for regulatory changes:*

When a regulation changes, RiskExec publishes a new MAJOR or MINOR pack version:

. *MAJOR version* (breaking): Schema changes, field additions/removals, rule logic changes that affect finding outcomes.
Existing runs continue on the old version.
New runs use the new version.
The old version is deprecated with a 90-day grace period.
. *MINOR version* (backward-compatible): New rules added, existing rule severity adjustments, citation updates.
Existing runs can optionally upgrade (admin action).
New runs default to the latest minor version.
. *PATCH version*: Typo fixes, description updates, no functional changes.
Applied automatically to all runs using the same MAJOR.MINOR version.

===== RBAC & Tenancy Architecture

*Components:* Auth & RBAC Service (<<sect-rbac-permission-matrix>>), PostgreSQL Row-Level Security (<<sect-multi-tenancy-isolation>>)

*Roles (IIA Three Lines Model aligned):*

* *First-Line Roles (QC):* `qc_reviewer`, `qc_manager` — loan-level data integrity testing
* *Second-Line Roles (Compliance):* `compliance_analyst`, `compliance_officer` — regulatory adherence review
* *Third-Line Roles (Internal Audit):* `auditor`, `audit_director` — independent assurance of first- and second-line effectiveness
* *Cross-Line:* `executive` (read-only dashboards), `it_admin` (tenant administration)

See <<sect-rbac-permission-matrix>> for the full permission matrix.

*Multi-tenant isolation:* PostgreSQL Row-Level Security (RLS) ensures that every query is automatically filtered by the authenticated user's `tenant_id`.
Cross-tenant data access is architecturally impossible at the database level.

*Line isolation:* Each finding carries a `defense_line` tag (1, 2, or 3).
Roles within a line can only create, edit, and close findings tagged to their own line.
Cross-line visibility is read-only: a `compliance_analyst` can view first-line QC findings for context but cannot modify them.

*Audit trail integration:* Every access control decision — grants, denials, segregation-of-duty violations, and cross-line isolation violations — is recorded as an immutable audit event (<<sect-audit-trail-event-store>>).
`it_admin` role provides searchable, filterable, exportable access to the complete audit log.

*SSO:* SAML 2.0 and OIDC integration.
No AuditIQ-specific passwords.
Session management: 30-minute idle timeout, 8-hour absolute timeout.

==== Developer Context

*Services:* Auth Service (JWT + RBAC), Pack Engine, Model Registry, Audit Trail Event Store, System Configuration Service.
*RBAC enforcement:* Roles are `qc_reviewer`, `qc_manager`, `compliance_analyst`, `compliance_officer`, `auditor`, `audit_director`, `admin`.
Cross-line isolation enforced at DB level (RLS) and API level (middleware).
