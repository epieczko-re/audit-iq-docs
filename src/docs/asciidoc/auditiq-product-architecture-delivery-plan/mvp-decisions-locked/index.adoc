=== 19.2 Decision D1: Document Extraction Strategy

*Decision Statement:* Which document parsing and OCR technology will AuditIQ use for HMDA loan document extraction, and how will per-field extraction confidence be produced?

*Chosen Default:* *Docling (open-source conversion + layout engine) + deterministic AuditIQ Field Confidence (AFC) layer + AWS/GCP cloud OCR (page-level fallback)*

____
*Critical clarification (per Docling Technical Validation Report):* Docling is a _document conversion and layout understanding_ engine, NOT a field-level extraction engine with per-field confidence scores. Docling’s native confidence scores are _conversion-quality indicators_ (POOR/FAIR/GOOD/EXCELLENT), not extraction probabilities. AuditIQ must build a *deterministic field extractor layer* on top of Docling output, and produce its own per-field confidence via the *AuditIQ Field Confidence (AFC)* composite score. See Section 10.2 for full AFC design.
____

*Alternatives Considered:*

[width="100%",cols="25%,25%,25%,25%",options="header",]
|===
|Alternative |Pros |Cons |Rejected?
|Docling only (no AFC layer) |Open-source, free, no vendor lock-in |No per-field confidence; Docling confidence ≠ extraction accuracy; cannot drive auto-confirm routing |Yes — AFC layer required
|IronOCR, Tesseract |Proven, specialized for document types |Expensive (IronOCR), limited accuracy (Tesseract), no layout understanding |Yes
|Cloud-only (AWS Textract) |High accuracy, handles edge cases, has per-field confidence |Vendor lock-in, cost per page (~$0.01–$0.015), data residency concerns |Yes
|*Docling + AFC layer + cloud OCR fallback* |Open-source baseline + deterministic per-field confidence + fallback safety |Two-component system with custom field extractor (complexity) |*CHOSEN*
|===

*Rationale:* - Docling provides open-source document conversion with layout understanding (bounding boxes, reading order, table detection) — sufficient structural output for >80% of HMDA documents (primarily text-based, born-digital PDFs) - The *AFC layer* (Section 10.2.4) builds deterministic per-field confidence from 5 signals: `docling_page_grade` (0.15), `ocr_segment_confidence` (0.25), `pattern_validity` (0.25), `layout_anchor_consistency` (0.20), `cross_doc_redundancy` (0.15) - AFC routing thresholds: ≥0.85 auto-confirm, 0.50–0.85 human review, <0.50 manual entry - Cloud OCR fallback is *page-level* (triggered when Docling page-grade < threshold), not document-level — preserving Docling for pages it handles well - MIT-licensed Docling core avoids vendor lock-in; *WARNING:* `docling-surya` plugin wraps SuryaOCR under GPL — do NOT adopt without legal review (see Risk #4 below) - MVP timeline: Docling integration + AFC v0 by end of Phase 0; cloud OCR as operational fallback

* *Risks of This Choice:* 1. *AFC precision insufficient on HMDA corpus:* If Phase 0 POC reveals <85% precision on key HMDA fields at AFC ≥0.85, the auto-confirm tier is unusable and all fields route to human review, negating the value proposition.
* *Mitigation:* Phase 0 POC test plan (Section 10.2.7) with explicit go/kill criteria; sample HMDA corpus (≥200 loans, 5+ lenders) 2. *Cloud OCR fallback rate too high:* If >40% of pages require cloud OCR fallback, cost model breaks ($0.20–$0.50+ per document) and latency degrades
* *Mitigation:* Page-level fallback routing (not document-level); usage monitoring; cost cap per customer per month 3. *Latency issues:* Docling <2s/page target; cloud OCR adds 3–5s per page for API calls + network roundtrips
* *Mitigation:* Async processing; queue-based fallback (not blocking); fallback pages processed in parallel 4. *GPL licensing contamination:* `docling-surya` wraps SuryaOCR under GPL — if adopted, may require open-sourcing AuditIQ or create legal exposure for commercial SaaS
* *Mitigation:* Do NOT use `docling-surya` in any build. Stick to MIT-licensed Docling core. If OCR accuracy requires SuryaOCR-class models, use cloud OCR fallback instead. 5. *Docling beta extraction API instability:* Docling’s extraction API is marked beta and may change between releases, breaking field extractor logic
* *Mitigation:* Pin Docling version; wrap in abstraction layer; use conversion + layout API only (not beta extraction API)

*Go/Kill Criteria (Phase 0 POC — see Section 10.2.7):*

* *Go:* ≥85% precision on critical HMDA fields (borrower name, loan amount, property address, action taken, loan type) for AFC ≥0.85 slice; deterministic reproducibility (same input → same output); bounding-box alignment ≥90% overlap with ground truth
* *Kill:* <70% precision at AFC ≥0.85 OR fallback rate >40% of pages OR non-deterministic output across runs
* *Pivot:* If killed, evaluate AWS Textract as primary with Docling relegated to pre-processing only

*Do Not Do (per Docling Validation Report):* - Do NOT treat Docling’s native confidence scores as per-field extraction probabilities - Do NOT adopt GPL-licensed Docling plugins (`docling-surya`) in any commercial build - Do NOT rely on Docling’s beta extraction API for production field extraction

*Conditions to Revisit/Override:*

* *Override to cloud-primary:* If Phase 0 POC fails kill criteria (precision <70% or fallback >40%) AND AFC layer cannot compensate
* *Override to specialized tool (e.g., IronOCR):* If cloud OCR cost exceeds $10K/month by mid-Phase 1a
* *Revisit AFC weights:* After Phase 1a production data (≥1,000 loans) allows empirical recalibration of the 5-signal AFC formula
* *Revisit in Phase 2:* When pack marketplace includes non-English or specialized document types, reassess Docling + cloud coverage

=== 19.3 Decision D2: Schema Mapping Approach

*Decision Statement:* How will AuditIQ map extracted HMDA field values to audit pack schema fields with confidence scoring, given that HMDA LAR structure spans 48 data points expanding to ~110 fields with 37 ``key HMDA data fields'' (per Fed CA Letter 18-9)?

*Chosen Default:* *Hybrid heuristic matcher v1 (string similarity + type compatibility + document-section context + curated HMDA/CD synonym dictionary + human-in-the-loop) with two-layer confidence design*

____
*Critical clarification (per Schema Mapping Strategy Report):* Closing Disclosure (CD) documents alone cover only ~10–20% of the 110 HMDA fields at ≥0.85 auto-mappable confidence. Full HMDA coverage requires multi-document extraction (1003/URLA, CD, appraisal, title) in later phases. MVP must clearly demarcate CD-supported fields vs. fields requiring manual entry or additional document types.
____

*Alternatives Considered:*

[width="100%",cols="20%,20%,20%,20%,20%",options="header",]
|===
|Alternative |Approach |Precision (est.) |Latency |Rejected?
|*Hybrid heuristic + synonym dictionary (chosen)* |String similarity + type compat + doc-section context + HMDA/CD curated dictionary + COMA-style signal combination |85–92% on CD-supported fields |<50ms |*CHOSEN*
|Pure string matching (Levenshtein/Jaccard) |Name similarity only |70–80% (fails on synonyms like ``loan_type'' vs. ``mortgage_product'') |<10ms |Yes — insufficient
|LogMap / ontology-based |Formal ontology alignment |90–95% (requires ontology) |100ms–1s |Parallel research track
|Embedding-based (NLP) |Semantic vector similarity |88–94% (requires training data) |50–200ms |Deferred Phase 2
|LLM-based (zero-shot) |GPT-4 / Claude mapping |92–97% (zero-shot) |1–3s; $0.01–0.05/mapping |Deferred Phase 2 reranking
|===

*Mapping Pattern Taxonomy (5 types the matcher must handle):*

[width="100%",cols="34%,33%,33%",options="header",]
|===
|Pattern |Example |Heuristic Approach
|Exact name match |`loan_amount` → `loan_amount` |String equality; trivial
|Abbreviation mismatch |`prop_addr` → `property_address` |Abbreviation expansion dictionary
|Synonym |`loan_type` → `mortgage_product` |HMDA/CD curated synonym dictionary
|Derived field |`ltv_ratio` (computed from loan_amount / appraised_value) |Rule-based derivation; not pure mapping
|Multi-source field |`total_income` (W-2 + 1003 + pay stubs) |Phase 2; flag as ``requires additional documents'' in MVP
|===

*Two-Layer Confidence Design (see Section 10.3 for implementation):*

* *Layer 1 — Raw signals:* `conf_extraction` (from AFC), `conf_mapping` (heuristic matcher output), `conf_rule` (business rule pass/fail)
* *Layer 2 — Calibrated decision score:* Combines Layer 1 signals using reliability diagrams + temperature scaling (or isotonic regression in Phase 2) to produce a calibrated probability that the mapping is correct
* *Routing:* ≥0.90 auto-accept, 0.60–0.90 human review, <0.60 manual mapping required

*Rationale:* - Hybrid heuristic with curated synonym dictionary is MVP-fast, interpretable, and auditable (regulator requirement) - COMA-style combination of multiple signals (name similarity, type compatibility, document-section context) outperforms any single signal - Two-layer confidence separates raw signal production from calibrated decision-making — enables recalibration without retraining - Curated HMDA/CD synonym dictionary captures domain-specific equivalences that generic string matching misses - Human-in-the-loop for 0.60–0.90 band ensures safety while building training data for Phase 2 ML upgrade - CD-only coverage reality (~10–20% at ≥0.85) is explicitly communicated to stakeholders — no over-promising

*Reviewer Burden Model:*

* Accept (conf ≥0.90): ~10 seconds per field (visual confirmation)
* Override (conf 0.60–0.90): ~45 seconds per field (evaluate suggestion, select correct mapping)
* Per-session estimate: 8–45 minutes depending on document scope and override rate
* Target: ≤25% override rate for CD-supported fields by end of Phase 1b

* *Risks of This Choice:* 1. *Precision too low for auto-accept tier:* If Phase 1b testing shows <95% precision at conf_mapping ≥0.90, auto-accept is unsafe and all mappings route to human review.
* *Mitigation:* Phase 1b UAT with 5+ reviewers on ≥50 loan files; if precision <95% at ≥0.90, narrow auto-accept to ≥0.95 2. *Override rate too high:* If >40% of CD-supported mappings are overridden, reviewer fatigue makes the product worse than manual mapping.
* *Mitigation:* Monitor override rate weekly; if >25% after tuning, escalate synonym dictionary expansion; if >40%, trigger kill criteria 3. *Confidence miscalibration:* Layer 2 calibration requires sufficient production data; early-phase scores may be poorly calibrated.
* *Mitigation:* Start with conservative thresholds (auto-accept ≥0.95); lower to ≥0.90 only after calibration validation on ≥500 mappings 4. *CD-only coverage gap under-communicated:* Stakeholders expect ``HMDA mapping'' but CD covers only ~10–20% of fields — expectation mismatch.
* *Mitigation:* Explicit MVP scope communication; UI clearly labels ``CD-supported'' vs. ``requires additional documents'' fields 5. *Synonym dictionary maintenance burden:* Curated dictionary requires ongoing updates as new pack definitions and document types are added.
* *Mitigation:* Dictionary stored as versioned JSON; pack creators can propose additions via PR; reviewed quarterly

*Go/Kill Criteria (Phase 1b UAT):*

* *Go:* ≥95% precision at conf_mapping ≥0.90 on CD-supported fields; override rate ≤25%; reviewer session time ≤30 min for standard CD
* *Kill:* >40% override rate on CD-supported fields OR confidence scores uncalibratable (no separation between correct and incorrect mappings) OR unable to distinguish CD-supported vs. non-CD fields in UI
* *Pivot:* If killed, evaluate embedding-based matcher as primary; if that fails, fall back to manual mapping with template pre-fill

*Phase 2 Upgrade Path:* - Embedding-based candidate expansion (vector similarity for synonym discovery) - LLM-based reranking of top-N candidates (zero-shot, batch processing for cost control) - LogMap ontology track (if academic partnership delivers) - Isotonic regression for Layer 2 calibration (replaces temperature scaling)

*Conditions to Revisit/Override:*

* *Escalate embedding matcher to Phase 1b:* If Phase 1a synonym dictionary fails to achieve ≥85% recall on known HMDA synonyms
* *Adopt LLM reranking:* If cost model improves (batch LLM pricing <$0.005/mapping) and latency is acceptable (<500ms)
* *Expand to multi-document:* Phase 2 when 1003/URLA and appraisal extraction are available; requires new mapping patterns for derived and multi-source fields
* *Recalibrate confidence:* After ≥1,000 production mappings with reviewer feedback; adjust Layer 2 weights and thresholds

=== 19.4 Decision D3: Audit Trail Immutability & Correctability Semantics

*Decision Statement:* How will AuditIQ balance the need for immutable, cryptographically defensible audit trails (regulator + litigation requirement) with the business need to correct mistakes after finalization (operational reality)?

*Chosen Default:* *Append-only event log with RFC 8785 canonical JSON serialization + SHA-256 hash chaining + global monotonic sequence IDs + cryptographic anchor points + periodic snapshots + immutability rules per artifact type (Section 20)*

____
*Critical clarification (per Audit Trail Defensibility Report):* Hash chaining alone is necessary but NOT sufficient for defensibility. An attacker with database write access can rewrite the chain and recompute all subsequent hashes. The design MUST include: (1) RFC 8785 canonical JSON for repeatable hash verification, (2) global monotonic sequence IDs for gap/reorder detection, (3) cryptographic anchor points (signed checkpoints or WORM-stored anchors) that cannot be silently rewritten.
____

*Per-Artifact Immutability Rules:*

* Evidence files: Append-only; never deleted
* Audit events: Append-only; corrections emit compensating events; each event includes `sequence_id` (global monotonic), `previous_hash` (SHA-256 of RFC 8785-canonicalized predecessor), `event_hash`
* Findings: Mutable via status transitions; each transition is an immutable event
* Pack versions: Immutable once published
* Mappings: Versioned; old versions preserved
* Reports: Immutable once finalized; new versions supersede
* Reviewer comments: Append-only with ``supersedes'' pointers - *``Undo'' definition:* New compensating event that resets state pointer to prior snapshot (NOT deletion)

*Three Design Tiers (per Audit Trail Defensibility Report):*

[width="100%",cols="25%,25%,25%,25%",options="header",]
|===
|Tier |Components |Use Case |MVP?
|*Minimal Defensible* |Hash chain + RFC 8785 canonical JSON + monotonic sequence + WORM-stored anchor hashes |Regulatory exam defense; satisfies FFIEC + NIST AU-9 |*YES — MVP target*
|Enterprise-grade |Above + KMS-signed checkpoint events every N events + HSM key management |Litigation support (FRE 902(13)/(14)); SOC 2 Type II |Phase 2
|Overkill |Above + Certificate Transparency–style Merkle tree log |Public verifiability; multi-party audit |Not planned
|===

*Alternatives Considered:*

[width="100%",cols="25%,25%,25%,25%",options="header",]
|===
|Alternative |Pros |Cons |Rejected?
|*Append-only + hash chain + RFC 8785 + anchoring (chosen)* |Regulator-friendly; litigation-defensible; gap/tamper detectable |Requires canonical serialization discipline + anchor storage |*CHOSEN*
|Full immutability (no corrections) |Simplest; regulator clear favorite |Operationally unworkable; mistakes cannot be fixed |Yes
|Soft delete (deleted_at flag) |Simple DB implementation |Doesn’t meet immutability requirement; regulators will reject |Yes
|Simple hash chain (no anchoring) |Easy to implement |Attacker with DB access can rewrite entire chain undetected |Yes — insufficient
|Mutable with change log |Maximum flexibility; classic audit log pattern |Doesn’t guarantee immutability; deletion still possible |Deferred
|===

*Rationale:* - Regulators (FFIEC IT Examination Handbook) expect log controls with read-only media and isolated storage; append-only + WORM anchors is the gold standard - RFC 8785 (JSON Canonicalization Scheme) ensures identical JSON input always produces identical hash output — mandatory for repeatable verification (non-canonical JSON causes hash mismatches on logically identical data) - Global monotonic sequence IDs (per-tenant minimum) detect event gaps, reordering, and insertion attacks - Cryptographic anchor points (WORM-stored hashes, or KMS-signed checkpoints in Phase 2) prevent silent chain rewriting - Compensating events preserve complete audit history while allowing operational corrections - FRE Rule 902(13)/(14) allows self-authentication of electronic records via hash values — our design directly supports litigation defensibility - NIST SP 800-53 Rev 5 controls: AU-9 (protection of audit information, WORM + crypto), AU-10 (non-repudiation), AU-11 (audit record retention)

*Failure Modes (from Audit Trail Defensibility Report):*

[width="100%",cols="25%,25%,25%,25%",options="header",]
|===
|Failure Mode |Likelihood |Impact |Mitigation
|Non-canonical JSON breaks hash verification |High |High |RFC 8785 mandatory; reject non-canonical events at write time
|DB compromise rewrites hash chain |Medium |High |WORM-stored anchor hashes; periodic anchor validation
|Sequence gaps (events lost) |Medium |High |Monotonic sequence; gap detection on read; alert on gaps
|Key compromise (Phase 2 signing) |Low–Med |High |KMS with automatic rotation; HSM for production
|Cross-tenant event access |Low–Med |High |Tenant isolation at storage layer; sequence IDs per tenant
|Time skew between nodes |High |Medium |NTP sync; accept server timestamp only; reject client timestamps
|Snapshot integrity drift |Medium |Med–High |Hash snapshots; store hash in event stream; periodic re-materialization
|===

* *Risks of This Choice:* 1. *Complexity introduces bugs:* State machine with multiple artifact types + canonical serialization + hash chain + sequence management is hard to test; incorrect implementation could produce unverifiable chains.
* *Mitigation:* Extensive state machine testing (Phase 1a); round-trip verification tests (serialize → hash → deserialize → re-hash → compare); compensating event validation 2. *Regulator still rejects:* If regulator interprets ``immutable'' as ``no changes ever,'' compensating events are not sufficient.
* *Mitigation:* Early regulator engagement (Q1); whitepaper explaining compensating event pattern; cite NIST AU-9 and FFIEC precedent 3. *RFC 8785 implementation bugs:* Canonical JSON has subtle edge cases (Unicode normalization, number formatting, key ordering) that can silently break hash reproducibility.
* *Mitigation:* Use well-tested RFC 8785 library (not hand-rolled); round-trip verification on every event write; regression test suite with edge cases 4. *Snapshot storage cost:* If snapshots require 100MB each and we have 10K audits/year, storage = 1TB/year.
* *Mitigation:* Snapshot compression; tiering (snapshots only for finalized audits); hash snapshot in event stream for integrity 5. *WORM storage operational overhead:* WORM-stored anchors cannot be corrected if written incorrectly.
* *Mitigation:* Validate anchor data before WORM write; dual-write pattern (primary DB + WORM); alert on anchor mismatch

*Conditions to Revisit/Override:*

* *Upgrade to Enterprise-grade (Tier 2):* When litigation support is required or SOC 2 Type II audit demands KMS-signed checkpoints; target Phase 2
* *Override to full immutability:* If regulator explicitly rejects compensating events AND requires zero corrections post-finalization
* *Revisit WORM strategy:* If cloud provider WORM costs exceed $2K/month or operational burden is unsustainable; evaluate signed checkpoints as alternative anchor mechanism
* *Revisit in Phase 2:* Merkle tree evaluation if multi-party audit or public verifiability becomes a requirement

=== 19.5 Decision D4: Pack Interface – Declarative vs. Executable

*Decision Statement:* Should audit packs be declarative (JSON configuration) or executable (code with logic)?

*Chosen Default:* *Declarative JSON v1 (no executable logic)*

*Alternatives Considered:*

[width="100%",cols="25%,25%,25%,25%",options="header",]
|===
|Alternative |Pros |Cons |Rejected?
|*Declarative JSON (chosen)* |Secure (no code injection); versioning simple; portable; audit trail clear |Limited expressiveness; complex rule logic hard to encode |Partially
|Executable code (Python/JavaScript) |Flexible; supports arbitrarily complex rules |Security risk (code injection); validation hard; audit trail murky |Deferred Phase 2 research
|Hybrid (JSON + embedded expressions) |More expressive than pure JSON; somewhat safer |Still injection risk; validation complex |Evaluated
|Domain-specific language (DSL) |Security + expressiveness balance |Custom DSL requires design + parser (time cost) |Deferred Phase 2
|===

*Rationale:* - MVP scope requires HMDA + CRA + ECOA packs; all expressible in declarative JSON with rule definitions + confidence weights - No code execution = no code injection = no security audit headaches - Versioning and comparison (e.g., ``what changed in pack v2.1?'') are trivial with JSON - Regulator approval of pack logic is easier when pack is declarative (rules are explicit, not hidden in code) - Community pack marketplace (Phase 3) must be secure; declarative packs are safer for distribution

* *Risks of This Choice:* 1. *Customers demand executable logic:* Custom pack requirements include complex conditional rules (e.g., ``if loan_type == `ARM' AND rate > 5%, then find TRID violations'').
* *Mitigation:* JSON rule schema supports nested conditions & weights; covers 90% of use cases. For remaining 10%, Phase 2 DSL or API-based custom rules 2. *Declarative JSON becomes unwieldy:* As pack complexity grows (50+ rules), JSON becomes hard to maintain.
* *Mitigation:* Pack authoring UI (Phase 2); validation + linting tools; versioning support 3. *Performance: Interpreting JSON rules slower than compiled code*
* *Mitigation:* Rule evaluation is fast (<100ms per rule); not a bottleneck in MVP

*Conditions to Revisit/Override:*

* *Adopt executable packs:* If customer demand for custom logic is >30% AND we hire a security engineer to design safe sandboxing (Phase 2)
* *Implement DSL:* If community pack marketplace shows >20% DSL requests in Phase 3; prioritize DSL design for Phase 4
* *Hybrid approach:* If phase 2 customer feedback shows JSON limitations prevent >3 custom packs; adopt JSON + expression language (lower risk than full code)

=== 19.6 Decision D5: Tenant Isolation – Row-Level Security (RLS) for Phase 1

*Decision Statement:* What is the data isolation strategy for the multi-tenant AuditIQ system in Phase 1?

*Chosen Default:* *PostgreSQL Row-Level Security (RLS) for functional isolation + encryption + audit logging (Phase 1c)*

*Alternatives Considered:*

[width="100%",cols="25%,25%,25%,25%",options="header",]
|===
|Alternative |Pros |Cons |Rejected?
|*RLS only (Phase 1a–b); + encryption Phase 1c (chosen)* |RLS prevents data queries across tenants; simple to implement in PostgreSQL; audit logging for compliance |RLS alone not sufficient for SOC 2; encryption deferred to Phase 1c risk |Partially
|Database-per-tenant |Highest isolation; regulator favorite |Expensive (hosting 100+ DBs); operational complexity |Yes
|Application-level filtering |Works without RLS; flexible |Prone to IDOR bugs; requires discipline; harder to audit |Yes
|*RLS + encryption-at-rest (full Phase 1)* |Strongest security; meets SOC 2 + PCI DSS out of the box |Encryption key management complex; latency overhead; schedule slip |Parallel track
|===

*Rationale:*

* RLS (role-based GRANT policies) prevents accidental data query leaks; if RLS policy is set correctly, queries automatically filter by tenant
* Phase 1c adds encryption (TDE or Transparent Data Encryption) + audit logging to meet SOC 2 / regulatory requirements
* Timeline: RLS in Phase 1a (quick); encryption in Phase 1c (buffer time if key mgmt is complex)
* Cost: RLS is free (PostgreSQL feature); encryption licensing depends on strategy (pgcrypto free, customer keys optional)

* *Risks of This Choice:* 1. *RLS policy bugs expose customer data:* If role definition is incorrect, one customer can query another’s audit logs.
* *Mitigation:* RLS policy review by security partner (Phase 1c); automated tests for IDOR (Section 5.3); regular penetration testing 2. *Encryption deferred to Phase 1c introduces schedule risk:* If key management is complex, Phase 1c deadline at risk.
* *Mitigation:* Early PoC on encryption key rotation (Phase 0); third-party key management service (AWS KMS, HashiCorp Vault) considered 3. *Performance overhead:* Encryption adds ~5–10% latency per query (research needed).
* *Mitigation:* Benchmarking in Phase 1b; encryption only for sensitive tables (audit_events, findings) not high-frequency reads

*Conditions to Revisit/Override:*

* *Escalate encryption to Phase 1a:* If security audit (Phase 1c) identifies RLS policy gaps or if customer contracts require encryption-at-rest for Phase 1a
* *Adopt database-per-tenant:* If technical assessment shows RLS + encryption insufficient for SOC 2 (unlikely) OR if customer contracts explicitly require DB isolation
* *Add application-layer filtering:* If RLS policy proves too restrictive (rare); phase 2 research on RLS + app-layer hybrid
* *Upgrade encryption strategy:* If performance testing shows >10% latency; consider targeted encryption (only sensitive columns) vs. full TDE

=== 19.7 Decision D6: Canonical Data Schema — MISMO Reference Model (Flattened Subset)

*Decision Statement:* What naming convention and field definition standard should AuditIQ use for its canonical loan-level data model — the schema to which source records are mapped during ingestion and against which audit rules are evaluated?

*Chosen Default:* *MISMO Reference Model v3.x (flattened subset per regulation domain) for all loan-level attributes; proprietary `custom:` namespace for non-MISMO fields*

*Alternatives Considered:*

[width="100%",cols="25%,25%,25%,25%",options="header",]
|===
|Alternative |Pros |Cons |Rejected?
|*MISMO flattened subset (chosen)* |Industry standard; LOS/servicing systems already export MISMO; examiners recognize field names; HMDA Reporting Dataset well-defined; reduces integration friction |Verbose full schema (mitigated by flattening); requires maintaining MISMO dictionary; version tracking needed |No
|Proprietary canonical schema |Full control; simpler initial implementation; no external dependency |Every customer integration requires custom mapping; no examiner recognition; scaling to new regulations requires reinventing naming for each domain |Yes
|Full MISMO XML schema (unflattened) |Maximum standards compliance; direct XML import/export |Extremely verbose (nested 10+ levels); performance overhead for queries; overkill for audit comparison use case |Yes
|ULAD (Uniform Loan Application Dataset) |Standardized for GSE delivery; well-known in origination |Narrower scope than MISMO (origination-focused); doesn’t cover servicing, HMDA-specific, or QC fields |Yes
|===

*Rationale:* - MISMO is the mortgage industry’s lingua franca — HMDA data, Fannie/Freddie deliveries, MERS, and most LOS systems speak some dialect of MISMO XML or the MISMO Reference Model - The MISMO HMDA Reporting Dataset provides a well-defined ~80-field mapping that directly covers Phase 1’s HMDA compliance audit domain - Adopting MISMO field names (e.g., `ActionTakenType`, `LienStatusType`, `PropertyValuationAmount`) means less translation when customers upload data from MISMO-aware systems - Examiners and regulators recognize MISMO terminology, which increases AuditIQ’s credibility in exam-ready bundles and audit reports - The flattened subset approach avoids the verbosity problem — we store a single-level JSON with MISMO field names rather than deeply nested XML - Non-MISMO fields (custom institution fields, derived metrics) use a `custom:` prefix (e.g., `custom:InternalRiskScore`) for clear separation

* *Risks of This Choice:* 1. *MISMO version drift:* MISMO releases periodic updates; field names or enumerations could change.
* *Mitigation:* Pin to MISMO v3.6 for Phase 1; `mismo_dictionary` table includes version tracking; migration path defined in Section 11.6. 2. *Incomplete MISMO coverage for non-mortgage regulations:* Future regulations (BSA/AML, CRA, fair lending) may have fields not well-represented in MISMO.
* *Mitigation:* `custom:` namespace handles gaps; each pack manifest declares its MISMO coverage percentage; if coverage <60%, evaluate domain-specific standards. 3. *Flattening loses structural context:* MISMO’s nested XML carries semantic relationships (e.g., Borrower → Address → City) that are lost in a flat key-value model.
* *Mitigation:* `mismo_xpath` column in the dictionary preserves the full path for traceability; dot-notation keys (e.g., `Borrower.MailingAddress.City`) retain hierarchy in flat form where needed. 4. *Integration complexity for non-MISMO sources:* Customers uploading CSV/Excel from legacy systems that don’t speak MISMO will require more mapping effort.
* *Mitigation:* Schema mapping engine (Section 10) handles source-to-MISMO mapping; AI-assisted mapping suggestions use MISMO field descriptions for matching.

*Conditions to Revisit/Override:*

* *Adopt a different standard:* If AuditIQ expands to non-mortgage domains (e.g., consumer lending, commercial) where MISMO has <60% coverage, evaluate FpML, XBRL, or domain-specific alternatives
* *Upgrade MISMO version:* When MISMO v4.x releases (if applicable), evaluate migration path and backward compatibility
* *Adopt full MISMO XML:* Only if a major customer or regulator explicitly requires native MISMO XML import/export (unlikely for Phase 1; consider Phase 3 integration layer)
* *Hybrid approach:* If performance testing shows flattened MISMO + `custom:` namespace creates ambiguity, consider a formal schema registry (Apache Avro or JSON Schema with MISMO references)
